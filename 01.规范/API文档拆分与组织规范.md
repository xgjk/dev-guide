## 一、背景与目标

随着业务发展，各产品线的 API 接口数量持续增长，单个 API 文档体积膨胀（部分已超过 80KB），带来两个问题：

1. **AI 消费效率下降** — 单个文档过大，超出模型上下文窗口或挤占有效信息密度
2. **维护成本上升** — 新增接口需在大型文档中定位、插入、重新编号

本规范定义 API 文档的**拆分时机、拆分维度、目录组织方式**，确保文档始终对 AI 友好且便于维护。

---

## 二、核心原则

1. **AI 渐进式加载** — 文档组织方式与 Skill 的 `SKILL.md → references/` 模式一致：先读入口，按需加载明细
2. **单文档可控** — 每个文档控制在合理大小，AI 可在单次上下文中完整消费
3. **自包含优先** — 子文档应尽量自包含，减少跨文档依赖；必须依赖时明确标注引用关系
4. **入口稳定** — 总纲文档变化频率低，子文档独立变更不影响全局结构

---

## 三、文档组织结构

每个业务域下的 API 文档按以下结构组织：

```
02.产品业务AI文档/{业务域}/
├── {业务域}业务说明.md              ← 业务语义（已有，不变）
├── {业务域}API说明.md              ← API 总纲（入口文档）
├── API接口明细/                    ← 拆分后的子文档
│   ├── _通用约定.md               ← 提取的公共部分
│   ├── 01-{能力域名称}.md
│   ├── 02-{能力域名称}.md
│   └── ...
└── Agent合集/                      ← Agent 规范文档（已有）
```

### 3.1 总纲文档（{业务域}API说明.md）

**定位：入口文件 + 路由表。** AI 读此文件后决定需要加载哪些子文档。

**必须包含的章节：**

| 章节 | 内容 | 目的 |
|------|------|------|
| 修订记录 | 统一四列表格 | 变更追溯 |
| 概述 | 系统能力全景列表 | AI 快速理解"能做什么" |
| 通用约定 | 引用 `_通用约定.md`，或在此内联 | 统一鉴权、环境等基础信息 |
| 关键业务流程 | 跨接口的调用链路、状态流转 | AI 理解接口之间的协作关系 |
| 核心数据模型 | 关键实体、枚举、状态机 | AI 理解数据语义 |
| 接口索引 | 能力域 → 子文档的映射表 | **路由枢纽，AI 据此决定加载哪些子文档** |

**接口索引格式：**

```markdown
## 接口索引

| 编号 | 能力域 | 接口数量 | 文档链接 | 简要说明 |
|------|--------|---------|---------|---------|
| 01 | 员工与联系人查询 | 2 | [01-员工与联系人查询.md](API接口明细/01-员工与联系人查询.md) | 模糊搜索员工、获取员工详情 |
| 02 | 分组管理 | 5 | [02-分组管理.md](API接口明细/02-分组管理.md) | 创建/查询/管理个人与公共分组 |
| ... | ... | ... | ... | ... |
```

**总纲不得包含：** 具体接口的请求参数、响应参数、示例。这些内容只存在于子文档中。

### 3.2 子文档（API接口明细/ 目录）

**定位：具体接口的完整说明，按能力域拆分。**

每个子文档是一个独立的接口说明单元，包含：

- 该能力域覆盖的接口列表（概述）
- **通用说明**（简写"见 _通用约定.md"即可，不重复展开）
- 该能力域下所有接口的完整说明（遵循《开放平台API编写规范》中「四、接口详细说明」的格式）
- 该能力域特有的数据结构（遵循「五、公共数据结构」格式）
- 该能力域涉及的错误码（若与全局不同）

**子文档命名规范：**

```
{序号}-{能力域名称}.md
```

- 序号：两位数字，从 01 开始，按业务逻辑排序
- 能力域名称：简洁中文，反映该组接口的核心业务能力
- 示例：`01-员工与联系人查询.md`、`02-分组管理.md`

### 3.3 通用约定（_通用约定.md）

**定位：所有子文档共享的基础信息。**

从原 API 文档的「通用说明」章节中提取，包含：

- 访问地址模板
- 环境信息
- 公共请求头
- 通用响应结构
- 认证与安全约定
- 全局错误码

`_` 前缀表示这是一个辅助文件，AI 在加载子文档时应同时（或先于）加载此文件。

## 四、拆分时机与规则

### 4.1 何时拆分

| 条件 | 动作 |
|------|------|
| 单个 API 文档 **超过 40KB** | 必须拆分 |
| 单个 API 文档 **超过 20 个接口** | 建议拆分 |
| 业务域新增模块且接口数 **预计超过 10 个** | 直接按拆分结构创建，不先写单文件 |

### 4.2 拆分维度

**按「业务对象 / 能力域」拆分**，不是按接口数量均分。

判断标准：
- 一组接口围绕同一个核心业务对象（如"分组"、"汇报"、"草稿"）
- 或服务于同一个业务场景（如"数据采集"、"目标管理"）
- AI 在编写 Skill 时，通常需要某个业务对象的全部接口

**避免：**
- 按接口数量机械均分（如每 5 个接口一个文件）
- 按技术方法拆分（如"所有 GET 放一起"）
- 拆分过细（单个子文档只有 1-2 个接口，不如合并到相关域）

### 4.3 拆分后的文件大小控制

| 指标 | 建议值 |
|------|--------|
| 子文档大小 | **15-35KB** 为宜 |
| 子文档接口数 | **3-12 个**为宜 |
| 通用约定文件 | 尽量精简，控制在 **5KB** 以内 |
| 总纲文档 | 控制在 **10KB** 以内（不含通用约定全文） |

若某能力域接口过多，可进一步按操作类型拆分（如 `03-汇报管理-查询.md` 和 `03-汇报管理-写入.md`），但应优先保持能力域完整。

## 五、迁移步骤

将现有的单一 API 文档迁移到新结构：

1. **创建目录** — 建立 `API接口明细/` 文件夹和 `_通用约定.md`
2. **提取通用约定** — 将原文档的「通用说明」章节移至 `_通用约定.md`
3. **拆分接口** — 按能力域将「接口详细说明」章节拆入各子文档
4. **编写总纲** — 保留概述、业务流程、数据模型，新增接口索引表
5. **验证完整性** — 确认：
   - 接口索引表中的接口总数 = 原文档接口总数
   - 所有子文档中的接口编号无遗漏、无重复
   - 修订记录保留在总纲中，新增一条迁移记录
6. **删除原大文件**（或归档），Git 提交

## 六、新增接口时的操作规范

### 6.1 已拆分的业务域

1. 确定接口属于哪个能力域
2. 在对应子文档中追加接口说明
3. 更新子文档的修订记录
4. 更新总纲的接口索引（若新增了接口条目）
5. 更新总纲的修订记录

### 6.2 尚未拆分的业务域

- 若文档未达到拆分阈值（<40KB，<20个接口），继续在单文件中维护
- 达到阈值后，按「五、迁移步骤」执行拆分
- **新增业务域且预计接口较多时**，直接按拆分结构创建，跳过单文件阶段

## 八、注意事项

1. **总纲不重复子文档内容** — 总纲只做路由和全景描述，不放接口细节
2. **接口编号全局唯一** — 即使拆分到不同子文档，接口编号（如 4.1、4.2）在总纲索引中应保持全局连续
3. **跨文档引用需标注** — 子文档中引用其他子文档的接口时，使用格式：`详见 [02-分组管理.md](02-分组管理.md) 4.3 接口`
4. **修订记录的位置** — 总纲维护全局修订记录，子文档维护各自的修订记录
5. **目录命名一致性** — 所有业务域统一使用 `API接口明细/` 作为子文档目录名，不使用其他命名
