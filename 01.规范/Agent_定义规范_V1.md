## 一、适用范围

本规范用于定义业务 Agent 在当前阶段应当输出的标准内容。

这里的 Agent，指的是运行在具体业务系统中的业务助理。  
当前规范只聚焦 **Agent 的业务定义**，不展开 Skill 设计细则，不展开 API 规范，不展开接入、测试、上线等交付流程。

本规范适用于当前这类业务系统场景，例如：

- BP 系统
- 工作协同系统

---

## 二、规范目标

本规范的目标，是让各业务线在定义 Agent 时，能够使用统一的结构说明清楚以下问题：

- 这个 Agent 是谁
- 这个 Agent 属于哪个业务系统、哪个业务场景
- 这个 Agent 主要服务谁
- 这个 Agent 主要帮助用户完成什么业务目标
- 这个 Agent 围绕哪些核心业务对象开展工作
- 这个 Agent 能做什么、不能做什么
- 这个 Agent 需要哪些 Skill 能力支撑

通过统一这些内容，后续才能继续往下拆解：

- Agent 包含哪些 Skill
- Skill 需要哪些 API 支撑

---

## 三、输出原则

### 1. 贴近现有业务系统
Agent 的定义不能脱离现有业务系统空谈，要直接落到具体系统和具体场景中，例如 BP 系统、工作协同系统。

### 2. 先定义业务，不展开技术
当前阶段先定义 Agent 的业务定位、职责和边界，不展开 Skill 细节、API 细节，也不展开接入、测试、上线流程。

### 3. 名称和内容要业务化、可理解
Agent 的命名和描述要让业务人员能直接看懂，不要写得过于技术化、平台化。

### 4. 职责必须清楚，边界必须明确
每个 Agent 都必须明确说明能做什么、不能做什么，避免职责泛化和能力失控。

### 5. Skill 在 Agent 层只做能力描述
在 Agent 文档中，Skill 不展开成设计规范，只描述“为了完成职责，需要哪些能力”。

## 四、Agent 定义模板 V1

## 1. Agent 名称
**说明：**  
用于标识这个 Agent 是谁。名称应直接体现业务用途，不要写得过于技术化或平台化。

**建议写法：**
- BP 创建助手
- BP 审核助手
- 工作汇报分析助手
- 任务协同助手

---

## 2. 所属业务系统 / 业务场景
**说明：**  
用于说明这个 Agent 落在哪个现有业务系统、哪个具体场景中。  
这里不再写“所属域”这类大而空的概念，直接贴着实际系统写。

**建议写法：**
- 所属系统：BP 系统
- 所属场景：个人 BP 创建
- 所属系统：工作协同系统
- 所属场景：汇报分析 / 任务协同

---

## 3. 服务对象
**说明：**  
用于说明这个 Agent 主要服务谁。  
因为同一个系统里，不同角色的诉求差异很大，所以这个字段必须明确。

**建议写法：**
- 员工
- 部门负责人
- 中心负责人
- 集团管理层

---

## 4. 业务目标
**说明：**  
用于说明这个 Agent 存在的核心目的。  
回答的是：它主要帮助用户完成什么业务目标。

**建议写法：**
- 帮助员工高质量完成个人 BP 创建
- 帮助管理者快速审阅下属 BP
- 帮助管理者从汇报中识别风险、任务和决策点
- 帮助用户将零散沟通整理成可执行任务

---

## 5. 核心业务对象
**说明：**  
用于说明这个 Agent 主要围绕哪些业务对象开展工作。  
这部分很关键，因为系统不是泛聊天，而是围绕明确业务对象运转。

**BP 系统可参考对象：**
- 目标
- 成果
- 举措
- 承接关系
- 证据
- 指标

**工作协同系统可参考对象：**
- 汇报
- 讨论
- 任务
- 决策
- 反馈

---

## 6. 能做什么 / 不能做什么
**说明：**  
这是 Agent 定义里最重要的一部分。  
必须清楚写明职责范围和边界，避免 Agent 失控、泛化。

### 能做什么
写 3–5 条核心职责即可。

### 不能做什么
写 3–5 条边界事项即可。

**示例：BP 审核助手**

**能做什么：**
- 检查 BP 结构是否完整
- 识别目标与成果是否匹配
- 识别承接关系是否缺失
- 给出 BP 优化建议

**不能做什么：**
- 不能替代管理者最终定稿
- 不能越权查看无权限 BP
- 不能直接修改正式 BP 数据
- 不能绕过规则生成不合规内容

---

## 7. 所需 Skill 能力清单
**说明：**  
这里不展开 Skill 设计规范，只写这个 Agent 为完成职责，所需要具备的能力。  
建议写成“能力描述”，而不是过早写成复杂技术名称。

**建议写法：**
- BP 结构校验能力
- BP 承接分析能力
- 汇报摘要能力
- 任务抽取能力
- 风险识别能力
- 规则库查询能力
- 历史数据对比能力

---

## 五、一句话定义模板

每个 Agent 可统一补充一句总定义：

**这是一个面向【服务对象】，运行在【业务系统/业务场景】中的业务助理，围绕【核心业务对象】开展工作，主要帮助用户完成【业务目标】，能够处理【职责范围】，但不处理【边界范围】，其实现依赖【所需 Skill 能力】。**

## 七、为什么当前先保留这 7+2 项

当前保留的核心内容包括：

- Agent 名称
- 所属业务系统 / 业务场景
- 服务对象
- 业务目标
- 核心业务对象
- 能做什么
- 不能做什么
- 所需 Skill 能力清单
- 版本
- 日期

这套内容已经能够说明一个 Agent 当前最关键的定义信息：

- 它是谁
- 它在哪
- 它服务谁
- 它为什么存在
- 它围绕什么业务对象
- 它的职责和边界是什么
- 它需要哪些能力支撑
- 它对应的是哪个版本、哪个时间点的定义

在此基础上，后续再继续往下拆 Skill 和 API，会更清晰、更可控。
