# OpenClaw 钉钉机器人自动绑定指引 v1.2

适用链接：  
[DingTalk-Real-AI/dingtalk-openclaw-connector](https://github.com/DingTalk-Real-AI/dingtalk-openclaw-connector)

本指引用于让 OpenClaw 通过交互式流程，自动完成：
1) 收集用户要绑定机器人的 `APPKey` 与 `APP Secret`  
2) 判断绑定目标 `agentName`（未提供则默认当前会话 Agent 名称）  
3) 自动检测并安装合适版本钉钉插件  
4) 将机器人绑定到目标 Agent  
5) 向用户返回绑定结果

---

## 一、执行原则（必须遵循）

1. 先简单后复杂：优先单机器人 + 文本消息链路跑通。  
2. 收集到完整参数前，不执行安装和绑定。  
3. `agentName` 未提供时，必须使用当前会话 Agent 名称。  
4. 对外统一名称：
   - 插件包名：`@dingtalk-real-ai/dingtalk-connector`
   - channel 名：`dingtalk-connector`
5. 输出中 `clientSecret` 必须打码，不得明文回显。  
6. **多机器人增量绑定**：同一 channel 下使用 `accounts` 存放多个机器人；**新增绑定只能在 `accounts` 中追加新键值对，并追加 `bindings` 条目**，不得用新凭证整体覆盖 `dingtalk-connector`，以免替换已绑定的机器人。  

---

## 二、OpenClaw 对用户的首轮收集话术

请 OpenClaw 向用户发送：

```text
为完成钉钉机器人绑定，请提供以下信息：
1) APPKey（即 clientId）
2) APP Secret（即 clientSecret）
3) 绑定目标 agentName（可选，不填则默认绑定到当前会话 Agent 名称）
4) 本机器人在配置中的 accountId（可选，建议英文短名，如 clawhelper、devhelper；不填则由系统生成唯一 id，且需保证与已有 accountId 不重复）

请按下面格式回复：
APPKey=xxxx
APPSecret=xxxx
agentName=xxxx（可留空）
accountId=xxxx（可留空，留空则自动生成）
```

---

## 三、参数校验规则

OpenClaw 收到用户输入后，按以下规则校验：

1. 若 `APPKey` 缺失：继续追问，不进入下一步。  
2. 若 `APP Secret` 缺失：继续追问，不进入下一步。  
3. 若 `agentName` 缺失或为空：设定 `agentName = 当前会话 Agent 名称`。  
4. 若 `accountId` 缺失或为空：生成新的唯一 `accountId`（或与目标 Agent id 对齐的规则由实现约定），并校验与 `channels.dingtalk-connector.accounts` 已有键名不冲突。  
5. 对敏感字段做最小化展示（例如 `abcd****wxyz`）。  

---

## 四、自动检测与安装插件流程

OpenClaw 需要按顺序执行：

### 1) 预检查
- `openclaw --version`
- `openclaw gateway status`
- `openclaw plugins list`

### 2) 判断是否已安装钉钉插件
- 若已安装 `@dingtalk-real-ai/dingtalk-connector`：记录版本，进入配置绑定步骤。  
- 若未安装：执行安装步骤。  

### 3) 安装步骤（稳定优先）
- 默认安装：
  - `openclaw plugins install @dingtalk-real-ai/dingtalk-connector`
- 若网络导致安装失败，使用镜像重试：
  - `NPM_CONFIG_REGISTRY=https://registry.npmmirror.com openclaw plugins install @dingtalk-real-ai/dingtalk-connector`

### 4) 版本兼容检查（安装后立即做）
根据官方说明：
- `dingtalk-connector v0.8.4+` 需要 `OpenClaw SDK v2026.3.22+`
- `v0.8.3 及以下` 对应较早 `v2026.3.x`

若版本不匹配，OpenClaw 需要先给出“版本对齐建议”（升级 SDK 或调整插件版本），再继续绑定。  

---

## 五、绑定执行流程

官方连接器支持在 `channels.dingtalk-connector.accounts` 下配置**多个机器人**，并通过 `bindings` 将「账号 → Agent」一一对应（参考 [dingtalk-openclaw-connector 多 Agent 配置](https://github.com/DingTalk-Real-AI/dingtalk-openclaw-connector)）。本指引要求：**新增一个 Agent 的钉钉机器人时，采用「追加 accounts + 追加 bindings」，禁止用单套 clientId/clientSecret 覆盖整个 channel 导致已有机器人失效。**

### 1) 首次绑定（尚无 `accounts`）

在 `openclaw.json` 中写入（保留文件内其他节点；若历史上使用过顶层 `clientId`/`clientSecret`，应迁移为 `accounts` 形态后再做多账号扩展）：

```json
{
  "channels": {
    "dingtalk-connector": {
      "enabled": true,
      "accounts": {
        "<accountId>": {
          "enabled": true,
          "clientId": "<APPKey>",
          "clientSecret": "<APPSecret>"
        }
      }
    }
  },
  "bindings": [
    {
      "agentId": "<与 openclaw agents.list 中一致的 agent id>",
      "match": {
        "channel": "dingtalk-connector",
        "accountId": "<accountId>"
      }
    }
  ]
}
```

说明：`agentId` 必须与 `agents.list` 里对应 Agent 的 `id` 一致；`match.accountId` 必须与 `accounts` 下的键名一致。

### 2) 新增绑定（已有其他机器人 —— **必须增量合并**）

当 `channels.dingtalk-connector.accounts` 已存在条目时：

1. **在 `accounts` 中新增一个键**（例如 `clawhelper`、`devhelper`），写入新机器人的 `clientId` / `clientSecret`，**不得删除或覆盖已有键**。  
2. **在 `bindings` 数组末尾追加一条**（或按实现合并去重），指定新 `agentId` 与新的 `accountId`。  
3. 若 `bindings` 中已存在相同 `agentId` + `channel` + `accountId` 的组合，则视为更新该条凭证或绑定关系，而不是整表替换。

示例（两个机器人并存，`accounts` 与 `bindings` 同时体现多路）：

```json
{
  "channels": {
    "dingtalk-connector": {
      "enabled": true,
      "accounts": {
        "clawhelper": {
          "enabled": true,
          "clientId": "<APPKey-1>",
          "clientSecret": "<APPSecret-1>"
        },
        "devhelper": {
          "enabled": true,
          "clientId": "<APPKey-2>",
          "clientSecret": "<APPSecret-2>"
        }
      }
    }
  },
  "bindings": [
    {
      "agentId": "ding-bot1",
      "match": {
        "channel": "dingtalk-connector",
        "accountId": "clawhelper"
      }
    },
    {
      "agentId": "ding-bot2",
      "match": {
        "channel": "dingtalk-connector",
        "accountId": "devhelper"
      }
    }
  ]
}
```

**禁止**：为绑定新机器人，把 `dingtalk-connector` 改回仅含顶层 `clientId`/`clientSecret` 的单机器人写法并丢弃 `accounts` 中已有账号。  
**禁止**：用新凭证直接覆盖整个 `accounts` 对象，导致仅保留一个键。

### 3) 执行 Agent 绑定逻辑

- 若用户提供 `agentName`：解析为对应 Agent（`agents.list` 中的 `id` / `name` 以实际 OpenClaw 配置为准），写入或更新 `bindings`。  
- 若用户未提供 `agentName`：绑定到当前会话 Agent。  
- 若同一 `accountId` 已存在：仅更新该账号下 `clientId`/`clientSecret`（用户主动轮换密钥），**不影响**其他 `accountId`。  
- 若同一 Agent 需换绑到另一钉钉机器人：更新对应 `bindings` 中的 `accountId`，或调整 `accounts` 中该 `accountId` 的凭证，避免误删其他绑定。  

### 4) 重启并验证
- `openclaw gateway restart`
- `openclaw gateway status`
- 短时间查看日志（如 `openclaw logs --follow`）确认无鉴权与启动异常  
- 多机器人场景：分别在钉钉中向各机器人发「你好」，确认路由到预期 Agent

## 六、最终回执模板（给用户）

绑定完成后，OpenClaw 需向用户返回：

```text
钉钉机器人绑定结果：
- 插件状态：已安装（版本：<pluginVersion>）
- 渠道状态：dingtalk-connector 已启用
- accountId：<accountId>
- clientId：<脱敏展示>
- clientSecret：<已脱敏>
- 绑定 Agent：<targetAgentName>（agentId：<agentId>）
- Gateway 状态：运行正常

你可以现在在钉钉给该机器人发送“你好”进行验证；其他已绑定机器人不受影响。
```

如失败，则返回：
1) 失败步骤  
2) 关键错误信息（脱敏后）  
3) 建议修复动作（例如版本对齐、重新安装插件、检查凭证）  

---

## 七、异常处理优先级

当出现“机器人不回复”或“消息触发 Gateway 异常”时，按以下顺序排查：

1. 版本兼容（SDK 与插件版本）  
2. 插件安装完整性（是否半安装）  
3. 凭证正确性（APPKey/Secret 是否错误、空格、过期）  
4. 钉钉侧应用状态（是否发布、是否 Stream 模式）  
5. 高阶配置（如 asyncMode、多媒体消息）  

始终保持策略：先把文本消息链路跑通，再扩展图片/文件/异步能力。
