# API 接口文档编写规范（含模板）

本文档定义 Open API 接口文档的编写规范，说明文档各部分的写作要求，并附完整可复用的接口文档模板。

## 第一部分：编写规范（各章节如何写）

### 一、概述

**目的**：让读者在 30 秒内了解该文档覆盖的能力范围，便于判断是否为目标 API。

**写作要求**：

1. **标题**：使用「{系统/模块名} Open API 接口文档」或等价表述。
2. **首段**：用 1～2 句话说明文档描述的对象（如：BP 目标管理系统对外开放的全部 API）。
3. **能力列表**：用编号列表列出文档覆盖的全部业务能力，每条一行，格式建议为：
  - **能力名称** — 简短说明（可选：典型使用场景或约束）。
4. **条数**：与「四、接口详细说明」中的接口数量对应，便于核对无遗漏。
5. **避免**：冗长背景、实现细节、非本文档范围的接口。

**示例结构**：

```markdown
本文档描述了 XX 系统对外开放的全部 API 接口。通过这些接口，可以实现以下业务能力：
1. **查询XXX** — 获取...，用于...
2. **创建/更新/删除XXX** — ...
...
```

---

### 二、通用说明

**目的**：统一约定访问方式、环境、鉴权与通用响应格式，避免在每个接口中重复说明。

**写作要求**：

#### 2.1 访问地址

- 给出**统一 URL 模板**，用占位符标出可变部分（如 `{域名}`、`{接口地址}`）。
- 格式示例：`https://{域名}/open-api/{接口地址}` 或 `https://{baseUrl}/api/v1/{path}`。
- 若存在路径前缀（如 `/open-api`），在此明确说明。

#### 2.2 环境信息

- 列建议包含：环境名称、域名/Base URL、备注（可选）。

#### 2.3 公共请求头

- 用表格列出所有接口**必须或推荐**携带的 Header。
- 列建议包含：Header 名称、说明、是否必填、示例或取值说明。
- 若全文档统一使用 `appKey` 鉴权，应明确写明：**所有接口均通过请求头携带 `appKey` 调用**；`appKey` 的申请、分发、轮换流程**不在本文档范围内**。

#### 2.4 通用响应结构

- 明确所有接口的**统一包装结构**（如 `Result<T>`）。
- 用表格说明顶层字段：字段名、类型、说明（含成功/失败时取值约定）。
- 若存在「业务 code」与「HTTP 状态码」的对应关系，在此说明或引用「错误码说明」章节。

**示例结构**：


| 字段         | 类型    | 说明                          |
| resultCode   | Integer | 业务状态码，1=成功，其他=失败 |
| resultMsg    | String  | 提示信息，成功时为 null       |
| data         | T       | 业务数据，失败时为 null       |


#### 2.5 认证与安全（文档级）

- 说明全文档统一的**认证方式**。若统一为请求头 `appKey`，则直接写明 Header 名称、是否必填、示例值格式，以及适用范围。
- `appKey` 的获取、开通、发放、轮换流程若由平台统一管理，可明确写「**不在本规范范围内**」。
- 说明**鉴权维度**（应用级 / 用户级 / 租户级；若无额外维度需明确写「仅应用级」）。
- 文档级**审计**要求（是否记审计、统一字段约定）；**行级 / 字段级权限**若在多数接口一致，在此写原则，特例在 **4.x** 补充。
- 与每个接口的差异部分写在 **4.x「权限与安全（本接口）」**，可写「同本节，无额外要求」。

#### 2.6 面向 AI 消费的数据设计

- **最小必要返回**：返回字段应以完成当前业务目标所必需的信息为准。对 AI 明显无用或低价值的字段默认不返回，如头像 URL、装饰图、冗余展示文案、仅供前端渲染的颜色/样式字段。
- **避免重复信息**：同一业务事实应尽量只有一个权威字段来源。若父对象已包含 `parentId` / `parentName`，子对象中仅在**脱离父上下文后仍必须独立使用**时才重复返回；否则应删除重复字段，避免模型误判。
- **扁平优先**：返回结构优先 1 至 2 层；超过 3 层嵌套时，应评估是否拆分为摘要接口 + 详情接口，或将高频字段平铺到上层。避免返回深层树状结构、递归结构、前端组件树等不利于模型稳定解析的数据。
- **摘要优先于全文**：对长文本、富文本、HTML、Markdown、大段日志、会话记录，优先返回摘要、关键片段或结构化结论；仅在确有必要时提供全文，并说明最大长度。
- **富文本统一规则**：富文本字段默认不直接返回原文内容，尤其**严格禁止**在响应中直接返回包含图片 `base64` 的 HTML / 富文本。正文类内容应统一转换为**结构化文本表示**、结构化提炼结果，或返回可访问的外部资源引用；只有极少数确需保留原文的接口，才在 **4.x** 明确写例外。
- **保留格式语义**：当原始内容包含标题、列表、表格等结构时，不应简单降级成一段纯文本。应优先保留其结构语义，例如返回 Markdown 兼容文本、分段后的块结构，或将表格转换为带表头的结构化数组，保证 AI 能识别行列关系与层次信息。
- **列表裁剪**：默认应限制列表返回规模，写清默认分页大小、最大分页大小、排序规则。若接口面向 AI 批量消费，建议支持 `pageSize`、`cursor`、`fields`、`summaryOnly`、`topN` 等裁剪能力中的一种或多种。
- **字段命名与语义单一**：避免同时返回多个含义接近但边界不清的字段，如 `name`、`displayName`、`showName` 并存；若必须并存，应明确主字段与适用场景。
- **显式区分机器字段与展示字段**：标识、状态、时间、枚举等供 AI 决策使用的字段应优先保留；纯展示字段应降级为可选，必要时通过单独字段分组或单独接口返回。
- **空值与缺省可预测**：避免用多种方式表达同一种“无数据”状态，例如同一字段同时出现 `null`、空串、`"-"`、`"暂无"`。全文档应统一约定。
- **token 控制原则**：返回数据量应优先通过**字段裁剪、层级控制、分页限制、文本截断**来控制，而不是依赖调用方事后截断。文档中应明确单次响应的推荐上限，如最大记录条数、单字段最大字符数、是否返回全文。
- **token 预算建议**：面向 AI Agent / Skill 的单次响应，建议默认控制在模型易消费范围内。若难以直接按 token 计量，可使用更稳定的替代约束，如“单次返回不超过 N 条记录”“单个文本字段不超过 N 字”“完整 JSON 不超过 N KB”；超过预算时应拆分接口或增加摘要模式。

### 三、关键业务流程说明

**目的**：通过典型场景说明接口的**调用顺序**和**参数传递关系**，降低集成方的理解成本。

**写作要求**：

1. **按场景组织**：每个场景一个小节，标题格式如「场景一：XXX」「场景二：XXX」。
2. **场景标题**：用一句话概括「要完成什么事」。
3. **需求说明**：用引用块（`> 需求：...`）写清业务目标。
4. **步骤**：用编号步骤写出调用顺序，每步包含：
  - 调用的接口（含章节号与路径，如 **4.1 查询周期列表**（`GET /bp/period/getAllPeriod`））；
  - 传入什么参数（尤其是上一步的返回值如何作为下一步的入参）；
  - 返回什么、如何解析（如按 `status` 筛选启用周期）。
5. **数据流向**：在步骤中显式写出「A 返回的 X 用作 B 的 Y 参数」。
6. **可选**：在场景末尾用「说明」补充易错点（如 ID 类型、员工 ID 与分组 ID 区别）。
7. **场景数量**：覆盖主要集成路径即可，不必穷举所有组合。

---

### 四、接口详细说明

**目的**：对每个接口提供完整、可执行的描述，使调用方能独立完成对接与联调。

每个接口建议包含以下**固定小节**（顺序保持一致）：

#### 4.x 接口名称（与概述中的能力对应）

- **一句话说明**：接口用途、在整体流程中的角色（如「通常作为第一步，用于获取 periodId」）。

#### 基本信息

- **表格**列出：接口地址、请求方式、Content-Type（POST/PUT 等需说明）。
- 接口地址为**相对路径**（与「通用说明」中的 base URL 拼接后为完整 URL）。

#### 请求参数

- **GET**：参数以表格列出，列包括：参数名、类型、必填、说明（含枚举、格式、取值范围）。
- **POST/PUT 等**：区分 Query 参数（若有）与 Body；Body 用「请求体为 JSON，字段如下」+ 表格描述；若 Body 为数组或非标准结构，用简短说明 + 示例补充。
- 必填列使用「是/否」或「必填/可选」。
- 对来自其他接口的入参，注明来源（如「来自 4.1 返回的 PeriodVO.id」）。
- 对 AI Agent / Skill 调用场景，需尽量补全以下可机读约束：**枚举值**、**格式**（日期/时间/邮箱/手机号/URL）、**默认值**、**空值语义**（`null` / 空串 / 空数组）、**是否允许额外字段**、**数组元素结构**。
- 若接口已提供 OpenAPI / JSON Schema / 示例请求体 schema，建议在本节引用或附录给出，避免模型仅凭自然语言猜测字段结构。

#### 请求与行为约定

- 用简短表格或列表说明：**是否支持分页**（参数名与语义）、**是否支持批量**（单次上限、部分失败语义）、**幂等性要求**（是否幂等、幂等键、重复调用结果）。
- 与查询/写入无关的接口可对应写「不适用」或「否」。

#### 请求示例

- 提供 **curl** 示例（或等价 HTTP 请求），包含 URL、Method、必要 Header、Query/Body。
- 若鉴权方式特殊，在通用说明中说明后，此处可写「参见通用说明」。

#### 响应参数

- 先说明 `data` 的类型（如 `List<PeriodVO>`、`PageInfo<TaskReportUnionVO>`）。
- 用表格列出**业务数据结构**的字段：字段名、类型、说明。
- 若为泛型或分页，先描述外层结构（如 total、list、pageNum、pageSize），再描述 list 中元素类型。
- 嵌套对象可单独表格或引用「五、公共数据结构」。
- 枚举值在说明列写清（如 `status：1=启用，0=未启用`）。
- 对 AI Agent / Skill 调用场景，需额外说明：**字段是否可能缺失**、`null` 是否与缺省等价、空数组/空对象含义、是否允许返回未在表中声明的新字段。
- 应说明是否存在**可省略字段**、**展示字段**、**冗余字段**；对 AI 无必要或与上层重复的信息，不建议放入默认响应。
- 若对象存在父子结构、摘要与详情结构，应明确哪些字段只在详情接口返回，哪些字段在列表接口中禁止展开。


#### 响应示例

- 提供**真实可用的 JSON 示例**，与响应参数表一致。
- 成功场景即可；若需体现错误形态，可在「通用说明」或「错误码说明」中统一示例。
- 敏感信息可打码，但结构、类型、字段名需与接口一致。
- 对会触发限流、鉴权失败、参数校验失败的接口，建议各补至少一例，便于 Agent 做自动分支处理。


#### 数据流向（可选但推荐）

- 用简短列表说明：本接口返回的哪些字段会作为**其他接口**的入参（如「返回的 id 用于 4.2 的 periodId」）。
- 便于读者在「业务流程说明」与「接口说明」之间交叉查阅。

#### 数据口径说明（可选）

- 若接口涉及统计、账期、状态机或与业务理解强相关的字段，增加本节：**字段业务含义**、**时间口径**（时区、自然日/账期）、**状态口径**、**主数据来源**、**更新频率**、**空值含义**、**脱敏规则**。
- 与公共定义一致时可写「同 {字段} 全文档约定」并引用「五、公共数据结构」或概述。

#### 权限与安全（本接口）

- **行级权限**（数据范围）、**字段级权限**、**敏感字段**、**审计要求**；若与「二、2.5」完全一致，写「同 2.5，无额外约束」即可。

#### 性能与稳定性（本接口）

- **目标 SLA / 延迟**（若与全局不同）、**超时建议**、**限流**、**重试与幂等**、**熔断 / 降级**、**缓存**；无特殊要求可写「遵循平台默认策略」。
- 若接口主要供 AI 消费，建议额外说明：**默认返回条数**、**最大返回条数**、**单字段最大长度**、**是否支持字段裁剪 / 摘要模式 / 详情模式**，用于控制单次调用的 token 成本。

**写作注意**：

- 接口顺序建议与「概述」中的能力列表一致，编号连续（4.1、4.2、…）。
- 同一术语在全文档保持一致（如「周期 ID」与「periodId」对应）。
- 若某接口已废弃，注明「已废弃」并推荐替代接口。

### 五、公共数据结构

**目的**：集中定义多处复用的 DTO/VO，避免在多个接口中重复展开，便于维护。

**写作要求**：

1. **按结构分小节**（如 5.1 BaseTaskVO、5.2 TaskUserVO）。
2. 每个结构用表格描述：字段名、类型、说明（含枚举值）。
3. 若存在继承关系（如 KeyResultVO 继承 BaseTaskVO），在说明中注明「继承 XXX，下表仅列出扩展字段」。
4. 嵌套类型可单独小节或在本节内子表描述。
5. 与「四、接口详细说明」中的引用一致（如「详见五、公共数据结构 — TaskUserVO」）。

---

### 六、错误码说明

**目的**：统一列出业务与系统错误码，便于前端/调用方做分支与提示。

**写作要求**：

1. 用表格列出：错误码（resultCode 或 code）、说明、可选「HTTP 状态码」或「处理建议」。
2. 先列**成功**（如 1），再列**通用失败**（如 0、500），再按模块或码段列鉴权、限流、业务错误等。
3. 若错误码有层级或模块划分，可在表前用简短文字说明。
4. 与「通用说明」中的「resultCode」约定一致。
5. 对 Agent 调用常见错误，处理建议应尽量明确为以下动作之一：**修正参数后重试**、**稍后重试**、**不可重试**、**联系接口方处理**。

---

### 七、注意事项

**目的**：汇总易错点、兼容性说明、重要约束，减少对接问题。

**写作要求**：

1. 分条列出，每条一个主题（如 ID 精度、员工 ID 与分组 ID 区别、递归结构、排序默认值等）。
2. 每条尽量简短，可包含：现象、原因、正确做法、错误示例（避免）。
3. 与文档中其他章节不重复，可引用「见 X 章」。
4. 涉及安全、限流、幂等、兼容性时单独强调。
5. 明确该接口的数据会直接给AI使用，不会作为其他接口的前置接口，并且数据量比较大的情况下，建议业务层直接返回markdown格式的数据，api文档描述清楚返回的数据内容即可。
6. 若接口主要供 AI Agent / Skill 使用，需特别检查是否存在以下问题：**无关字段过多**、**父子重复字段**、**深层嵌套**、**超长文本未裁剪**、**列表过大未分页**、**展示字段挤占有效信息**。
---

### 八、变更管理

**目的**：约定本 Open API 文档所覆盖接口的**演进规则**，减少对接方因破坏性变更受损。

**写作要求**：用表格列出向后兼容原则、废弃与删除流程、通知对象；与组织级 API 治理规范一致时可引用上级文档，此处只写**本系统特有**约定。对 Agent / Skill 调用场景，建议额外明确：**字段新增是否保证向后兼容**、**枚举值是否允许扩展**、**`null` 与字段缺失是否等价**、**数组顺序是否有语义**、**分页游标 / 页码稳定性**。

**修改记录**：每次提交必须在文档最开头部分的修订记录中，记录本次修改的内容简要（对应具体章节）、修改时间，修改人


## 第二部分：完整接口文档模板

以下为可直接复用的接口文档模板，按「第一部分」规范填写即可。

---

# {系统/模块名} API 说明

## 一、概述

本文档描述了 **{系统/模块名}** 对外开放的全部 API 接口。通过这些接口，可以实现以下业务能力：

1. **{能力1名称}** — {一句话说明，可选：典型用途}
2. **{能力2名称}** — {一句话说明}
3. **{能力3名称}** — {一句话说明}
4. （按实际接口数量继续列举）

---

## 二、通用说明

### 2.1 访问地址

```
https://{域名}/open-api/{接口地址}
```

### 2.2 环境信息


| 环境   | 域名/Base URL               | 备注  |
| ---- | ------------------------- | --- |
| 生产环境 | `https://xxx.example.com` | -   |


### 2.3 公共请求头


| Header         | 说明                         | 是否必填  |
| -------------- | -------------------------- | ----- |
| `appKey`       | 应用密钥，写明获取方法                | 是     |
| `Content-Type` | 请求体类型，如 `application/json` | 按接口要求 |


### 2.4 通用响应结构

所有接口返回统一的 `Result<T>` 结构：

```json
{
  "resultCode": 1,
  "resultMsg": null,
  "data": null
}
```


| 字段           | 类型      | 说明                        |
| ------------ | ------- | ------------------------- |
| `resultCode` | Integer | 业务状态码，`1` 表示成功，其他值表示失败    |
| `resultMsg`  | String  | 提示信息，成功时为 `null`，失败时为错误描述 |
| `data`       | T       | 业务数据，不同接口类型不同，失败时为 `null` |

### 2.5 认证与安全（文档级约定）

### 2.6 面向 AI 消费的数据设计

| 项 | 说明 |
| --- | --- |
| 最小必要返回 | 默认只返回完成当前任务所需字段；头像、样式、装饰文案等对 AI 无决策价值的字段不建议默认返回 |
| 去重原则 | 同一事实尽量只保留一个权威字段；父对象信息不应在每个子对象中重复返回，除非子对象脱离父上下文后仍需独立使用 |
| 层级控制 | 默认建议 1 至 2 层；超过 3 层应拆分摘要接口 / 详情接口，或重新设计结构 |

| 富文本规则 | 默认不返回原始富文本，更不得直接返回包含图片 `base64` 的内容；应统一转换为结构化文本表示、结构化结果或外部资源引用，例外情况在具体接口中单独说明 |

| 裁剪能力 | 建议说明是否支持 `fields`、`summaryOnly`、`topN`、分页或其他裁剪参数 |
| token 控制 | 优先通过字段裁剪、层级控制、分页和文本截断控制单次响应大小；建议同时给出记录条数、字符数或 KB 级上限 |

## 七、注意事项

1. **ID 精度**：所有 ID 为 Long（雪花算法），前端请用字符串接收，避免 Number 精度问题；严禁对 ID 做 `parseInt`、`Number()` 等转换。
2. **AI 无关字段**：头像、颜色、图标、纯展示文案等若不参与模型判断，不应放入默认响应；确需保留时应通过可选字段或详情接口提供。
3. **避免父子重复**：父对象已提供的 ID、名称、状态等信息，不应在每个子对象中重复返回；只有子对象离开父上下文仍需独立消费时才保留。
4. **控制层级与长度**：默认优先扁平结构；超长文本、大列表、深层嵌套应拆为摘要接口和详情接口，避免单次返回过大。
5. **禁止返回富文本原文**：富文本默认统一转为结构化文本表示或结构化结果，严格禁止直接返回带图片 `base64` 的 HTML / 富文本内容；涉及表格、列表、标题层级时应尽量保留结构语义，不应简单压平成纯文本。如确需返回原文，应在具体接口中明确说明原因、长度上限和替代方案。
