## 第一部分：编写规范（各章节如何写）

### 一、概述

**目的**：让读者在 30 秒内了解该文档覆盖的能力范围，便于判断是否为目标 API。

**写作要求**：

1. **标题**：使用「{系统/模块名} Open API 接口文档」或等价表述。
2. **首段**：用 1～2 句话说明文档描述的对象（如：BP 目标管理系统对外开放的全部 API）。
3. **能力列表**：用编号列表列出文档覆盖的全部业务能力，每条一行，格式建议为：
  - **能力名称** — 简短说明（可选：典型使用场景或约束）。
4. **条数**：与「四、接口详细说明」中的接口数量对应，便于核对无遗漏。
5. **避免**：冗长背景、实现细节、非本文档范围的接口。

**示例结构**：

```markdown
本文档描述了 XX 系统对外开放的全部 API 接口。通过这些接口，可以实现以下业务能力：
1. **查询XXX** — 获取...，用于...
2. **创建/更新/删除XXX** — ...
...
```

---

### 二、通用说明

**目的**：统一约定访问方式、环境、鉴权与通用响应格式，避免在每个接口中重复说明。

**写作要求**：

#### 2.1 访问地址

- 给出**统一 URL 模板**，用占位符标出可变部分（如 `{域名}`、`{接口地址}`）。
- 格式示例：`https://{域名}/open-api/{接口地址}` 或 `https://{baseUrl}/api/v1/{path}`。
- 若存在路径前缀（如 `/open-api`），在此明确说明。

#### 2.2 环境信息

- 列建议包含：环境名称、域名/Base URL、备注（可选）。

#### 2.3 公共请求头

- 用表格列出所有接口**必须或推荐**携带的 Header。
- 列建议包含：Header 名称、说明、是否必填、示例或取值说明。

#### 2.4 通用响应结构

- 明确所有接口的**统一包装结构**（如 `Result<T>`）。
- 用表格说明顶层字段：字段名、类型、说明（含成功/失败时取值约定）。
- 若存在「业务 code」与「HTTP 状态码」的对应关系，在此说明或引用「错误码说明」章节。

**示例结构**：

```markdown
| 字段         | 类型    | 说明                          |
| resultCode   | Integer | 业务状态码，1=成功，其他=失败 |
| resultMsg    | String  | 提示信息，成功时为 null       |
| data         | T       | 业务数据，失败时为 null       |
```

### 三、关键业务流程说明

**目的**：通过典型场景说明接口的**调用顺序**和**参数传递关系**，降低集成方的理解成本。

**写作要求**：

1. **按场景组织**：每个场景一个小节，标题格式如「场景一：XXX」「场景二：XXX」。
2. **场景标题**：用一句话概括「要完成什么事」。
3. **需求说明**：用引用块（`> 需求：...`）写清业务目标。
4. **步骤**：用编号步骤写出调用顺序，每步包含：
  - 调用的接口（含章节号与路径，如 **4.1 查询周期列表**（`GET /bp/period/getAllPeriod`））；
  - 传入什么参数（尤其是上一步的返回值如何作为下一步的入参）；
  - 返回什么、如何解析（如按 `status` 筛选启用周期）。
5. **数据流向**：在步骤中显式写出「A 返回的 X 用作 B 的 Y 参数」。
6. **可选**：在场景末尾用「说明」补充易错点（如 ID 类型、员工 ID 与分组 ID 区别）。
7. **场景数量**：覆盖主要集成路径即可，不必穷举所有组合。

### 四、接口详细说明

**目的**：对每个接口提供完整、可执行的描述，使调用方能独立完成对接与联调。

每个接口建议包含以下**固定小节**（顺序保持一致）：

#### 4.x 接口名称（与概述中的能力对应）

- **一句话说明**：接口用途、在整体流程中的角色（如「通常作为第一步，用于获取 periodId」）。

#### 基本信息

- **表格**列出：接口地址、请求方式、Content-Type（POST/PUT 等需说明）。
- 接口地址为**相对路径**（与「通用说明」中的 base URL 拼接后为完整 URL）。

#### 请求参数

- **GET**：参数以表格列出，列包括：参数名、类型、必填、说明（含枚举、格式、取值范围）。
- **POST/PUT 等**：区分 Query 参数（若有）与 Body；Body 用「请求体为 JSON，字段如下」+ 表格描述；若 Body 为数组或非标准结构，用简短说明 + 示例补充。
- 必填列使用「是/否」或「必填/可选」。
- 对来自其他接口的入参，注明来源（如「来自 4.1 返回的 PeriodVO.id」）。

#### 请求示例

- 提供 **curl** 示例（或等价 HTTP 请求），包含 URL、Method、必要 Header、Query/Body。
- 若鉴权方式特殊，在通用说明中说明后，此处可写「参见通用说明」。

#### 响应参数

- 先说明 `data` 的类型（如 `List<PeriodVO>`、`PageInfo<TaskReportUnionVO>`）。
- 用表格列出**业务数据结构**的字段：字段名、类型、说明。
- 若为泛型或分页，先描述外层结构（如 total、list、pageNum、pageSize），再描述 list 中元素类型。
- 嵌套对象可单独表格或引用「五、公共数据结构」。
- 枚举值在说明列写清（如 `status：1=启用，0=未启用`）。

#### 响应示例

- 提供**真实可用的 JSON 示例**，与响应参数表一致。
- 成功场景即可；若需体现错误形态，可在「通用说明」或「错误码说明」中统一示例。
- 敏感信息可打码，但结构、类型、字段名需与接口一致。


#### 数据流向（可选但推荐）

- 用简短列表说明：本接口返回的哪些字段会作为**其他接口**的入参（如「返回的 id 用于 4.2 的 periodId」）。
- 便于读者在「业务流程说明」与「接口说明」之间交叉查阅。

**写作注意**：

- 接口顺序建议与「概述」中的能力列表一致，编号连续（4.1、4.2、…）。
- 同一术语在全文档保持一致（如「周期 ID」与「periodId」对应）。
- 若某接口已废弃，注明「已废弃」并推荐替代接口。

### 五、公共数据结构

**目的**：集中定义多处复用的 DTO/VO，避免在多个接口中重复展开，便于维护。

**写作要求**：

1. **按结构分小节**（如 5.1 BaseTaskVO、5.2 TaskUserVO）。
2. 每个结构用表格描述：字段名、类型、说明（含枚举值）。
3. 若存在继承关系（如 KeyResultVO 继承 BaseTaskVO），在说明中注明「继承 XXX，下表仅列出扩展字段」。
4. 嵌套类型可单独小节或在本节内子表描述。
5. 与「四、接口详细说明」中的引用一致（如「详见五、公共数据结构 — TaskUserVO」）。

---

### 六、错误码说明

**目的**：统一列出业务与系统错误码，便于前端/调用方做分支与提示。

**写作要求**：

1. 用表格列出：错误码（resultCode 或 code）、说明、可选「HTTP 状态码」或「处理建议」。
2. 先列**成功**（如 1），再列**通用失败**（如 0、500），再按模块或码段列鉴权、限流、业务错误等。
3. 若错误码有层级或模块划分，可在表前用简短文字说明。
4. 与「通用说明」中的「resultCode」约定一致。

---

### 七、注意事项

**目的**：汇总易错点、兼容性说明、重要约束，减少对接问题。

**写作要求**：

1. 分条列出，每条一个主题（如 ID 精度、员工 ID 与分组 ID 区别、递归结构、排序默认值等）。
2. 每条尽量简短，可包含：现象、原因、正确做法、错误示例（避免）。
3. 与文档中其他章节不重复，可引用「见 X 章」。
4. 涉及安全、限流、幂等、兼容性时单独强调。

```
https://{域名}/open-api/{接口地址}
```

### 2.2 环境信息


| 环境   | 域名/Base URL               | 备注  |
| ---- | ------------------------- | --- |
| 生产环境 | `https://xxx.example.com` | -   |


### 2.3 公共请求头


| Header         | 说明                         | 是否必填  |
| -------------- | -------------------------- | ----- |
| `appKey`       | 应用密钥，写明获取方法                | 是     |
| `Content-Type` | 请求体类型，如 `application/json` | 按接口要求 |


### 2.4 通用响应结构

所有接口返回统一的 `Result<T>` 结构：

```json
{
  "resultCode": 1,
  "resultMsg": null,
  "data": null
}
```


| 字段           | 类型      | 说明                        |
| ------------ | ------- | ------------------------- |
| `resultCode` | Integer | 业务状态码，`1` 表示成功，其他值表示失败    |
| `resultMsg`  | String  | 提示信息，成功时为 `null`，失败时为错误描述 |
| `data`       | T       | 业务数据，不同接口类型不同，失败时为 `null` |
