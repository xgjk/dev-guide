## 2. 系统概述

### 2.1 系统一句话定义

面向公司全员的 LLM（大语言模型）API 调用费用监控与可视化大屏系统。

### 2.2 系统解决的核心问题

公司多个业务产品（AI问答、AI翻译等）在调用各类 LLM 模型（GPT-4o、Claude、DeepSeek 等）时产生费用，但缺乏统一的费用感知手段。本系统通过读取外部同步的费用数据，以大屏看板形式回答三个核心问题：

1. **花了多少钱**：今日、本周、本月各维度的费用汇总
2. **钱花在哪里**：哪个模型、哪个产品、哪个人花得最多
3. **花得是否正常**：费用是否出现异常波动，是否需要预警

### 2.3 系统在整体业务中的定位

本系统是公司 LLM 使用成本的**监控层**，位于 LLM 调用链路的下游。它不参与实际的模型调用或费用计算，而是从已有的费用数据源中读取、聚合、展示和预警。

定位关系：

- **上游**：各业务产品（通过外部系统将调用数据同步到数据源表）
- **本系统**：读取数据 → 聚合 → 展示 → 预警
- **下游**：CWork 协同平台（接收预警待办通知）

### 2.4 系统不负责解决的问题

* 不负责记录或采集 LLM 调用明细（由外部系统同步）
* 不负责计算单次调用的费用（由上游系统完成）
* 不负责控制或限制 LLM 调用行为
* 不负责用户权限管理或登录认证
* 不负责跨环境的实时数据对比（测试/生产各自独立部署）
* 不负责用户行为分析（仅提供费用视角的用户统计，不分析调用内容或质量）

## 3. 核心业务对象

### 3.1 费用记录

* **定义**：以"产品+模型+人员+日期"为粒度的费用汇总记录，包含调用次数、输入/输出 Token 数量、费用金额等信息。
* **作用**：是系统所有统计分析的数据基础。每条费用记录关联一个产品、一个模型、一个人员、一个日期。
* **与其他对象的关系**：费用记录是原始数据，由外部系统每 5 分钟同步到数据源表。聚合表、排行榜、预警均从费用记录衍生。
* **常见误解**：费用记录不是"单次 LLM 调用"的记录，也不是"一笔订单"或"一次扣费"。它是同一产品、同一模型、同一人员在同一天内所有调用的汇总。

### 3.2 模型

* **定义**：一个具体的 LLM 模型，如 GPT-4o、Claude 3.5 Sonnet、DeepSeek-V3 等。
* **作用**：作为费用统计的核心维度之一，系统按模型统计费用排行和趋势。
* **与其他对象的关系**：每个模型归属于一个厂商。费用记录中的模型通过 aiType 编码标识，系统将其映射为可读的模型名称。
* **常见误解**：模型与厂商是不同层级的概念。GPT-4o 是模型，OpenAI 是厂商。系统同时支持按模型和按厂商两个维度统计。

### 3.3 厂商

* **定义**：提供 LLM 模型的服务商，如 OpenAI、Anthropic、DeepSeek 等。
* **作用**：作为费用统计的维度之一，按厂商聚合可了解不同服务商的费用占比。
* **与其他对象的关系**：一个厂商下有多个模型。系统通过模型编码（aiType）与厂商的映射关系进行聚合。
* **常见误解**：厂商不是独立的数据源表字段，而是通过模型编码间接映射得到的。映射关系存储在 ai_type_company_mapping 配置表中。

### 3.4 产品

* **定义**：公司内部使用 LLM 能力的业务模块，如 AI问答、AI翻译、AI写作等。
* **作用**：作为费用统计的核心维度之一，按产品统计可了解各业务线的 LLM 使用成本。
* **与其他对象的关系**：每条费用记录通过 bizCode 关联到具体产品。产品名称通过 ai_module 映射表获取。
* **常见误解**：产品的 bizCode 是技术标识（如 "myapp"），展示时需要映射为人类可读的模块名称。

### 3.5 人员

* **定义**：调用 LLM 的公司员工，通过人员ID标识。
* **作用**：作为费用统计的核心维度之一，按人员统计可了解个人使用 LLM 的费用情况。
* **与其他对象的关系**：每条费用记录关联一个人员。人员名称通过 personId 映射获取。
* **常见误解**：人员维度的统计关注的是"谁花了多少"，不是"谁被收费"。所有费用都是公司层面的支出，人员维度仅用于了解使用分布。

### 3.6 预警

* **定义**：当某个维度（模型、产品、人员）的今日费用相对于近期日均出现异常增长时产生的告警信号。
* **作用**：帮助相关负责人及时发现费用异常，避免成本失控。
* **与其他对象的关系**：预警对象可以是模型、产品或人员。预警通过 CWork 平台以协同待办的形式发送给配置的接收人。
* **常见误解**：预警不是"费用超预算"告警，而是"费用相比近期突然飙升"的异常波动检测。判断依据是今日费用 vs 近 7 天日均的倍数关系。

### 3.7 聚合数据

* **定义**：将原始费用记录按维度（总计、厂商、模型、产品、人员）和日期预先聚合的汇总数据。
* **作用**：提升历史数据查询性能。原始表数据量大，直接查询开销高，预聚合后可实现毫秒级响应。
* **与其他对象的关系**：聚合数据是从费用记录衍生出来的。历史数据从聚合表查询，当天数据从原始表实时查询。
* **常见误解**：聚合数据不是实时更新的，每天凌晨聚合一次前一天的数据。当天数据始终从原始表实时查询，保证数据新鲜度。

## 4. 核心业务关系

### 4.1 业务主链路

```
外部系统 → 费用记录(原始表) → 预聚合(聚合表) → 展示(大屏) + 预警(CWork)
```

1. **数据同步**：外部系统每 5 分钟将 LLM 调用费用数据同步到原始表（ads_ai_cost_dashboard）
2. **数据聚合**：系统每天凌晨自动将前一天的数据按维度聚合到聚合表（ads_ai_cost_daily_agg）
3. **数据展示**：大屏前端从后端 API 获取聚合数据和历史数据，以可视化图表展示
4. **异常预警**：系统每 5 分钟检测一次费用异常，发现异常后通过 CWork 发送预警通知
5. **产品用户分析**：支持按产品维度查看用户费用分布，统计每日用户数、人均费用、费用中位数（P50）、前 10% 高消费用户费用门槛（P10）等指标
6. **自定义查询**：支持外部系统通过 URL 参数指定时间段、环境、维度进行费用查询，可嵌入 iframe 展示

### 4.2 对象之间的关系

* **厂商 → 模型**：一对多。一个厂商提供多个模型，通过映射表关联。
* **产品 → 费用记录**：一对多。一个产品每天产生多条费用记录（按模型、按人员细分）。
* **模型 → 费用记录**：一对多。一个模型每天在多个产品、多个人员下产生费用记录。
* **人员 → 费用记录**：一对多。一个人员每天使用多个模型、多个产品产生费用记录。
* **费用记录 → 聚合数据**：衍生关系。聚合数据是费用记录按维度和日期的汇总。
* **费用记录 → 预警**：触发关系。当费用异常时产生预警。

### 4.3 关键流转逻辑

**费用统计流转**：

1. 原始表中每条记录包含：产品(bizCode) + 模型(aiType) + 人员(personId) + 日期(date) + 费用(sum_cost)
2. 查询历史数据时，直接读取聚合表（按维度、日期、维度值查询）
3. 查询当天数据时，实时从原始表聚合计算
4. 最终结果 = 历史聚合数据 + 当天实时数据

**预警流转**：

1. 系统获取某个维度（模型/产品/人员）的今日费用
2. 从聚合表获取该维度近 7 天（不含今天）的费用数据
3. 计算 7 天日均费用（去除 0 值后取均值，数据不足时做兜底处理）
4. 判断今日费用是否超过日均的阈值倍数，确定预警等级
5. 检查发送历史（去重：同一天同一条预警只发一次；升级：等级提升时重新发送）
6. 通过 CWork Open API 发送预警待办给配置的接收人

**周期对比流转**：

1. 选择一个主周期（今日/昨日/本周/上周/本月/上月）和对比周期
2. 分别查询两个周期在指定维度（厂商/模型/产品/人员）下的费用数据
3. 计算差异和涨跌幅，按费用排序展示
4. 支持从排行榜项钻取到下级维度（如从某模型钻取到该模型下的产品分布）

## 5. 业务规则

### 5.1 显性规则

1. **费用精度**：所有费用展示统一保留小数点后 2 位，货币单位为美元（USD）。
2. **预警阈值**：采用三级预警机制，以今日费用与近 7 天日均的倍数关系判断：
   - 一级告警（notice）：今日费用 > 日均 × 2 且 今日费用 > $2
   - 二级告警（warning）：今日费用 > 日均 × 4 且 今日费用 > $4
   - 三级告警（critical）：今日费用 > 日均 × 8 且 今日费用 > $8
3. **排行榜排序**：按今日费用降序排列，展示 Top N。
4. **排行榜预警标记**：仅在产品维度的排行榜中展示预警级别标记（模型和人员维度不展示）。
5. **7 天日均计算规则**：取近 7 天（不含今天）的每日费用，去除金额为 0 的天数后计算均值。有效天数不足 3 天时，直接取全部有效天数的均值；有效天数 ≥ 3 天时，排序后去除最低的 2 个值，取剩余值的均值。
6. **聚合频率**：每天凌晨（00:30）自动聚合前一天的数据，按 total、company、model、product、person 五个维度分别聚合。
7. **大屏刷新**：前端每 5 分钟自动刷新数据。
8. **日期范围限制**：自定义查询的日期范围不超过 90 天。
9. **预警发送频率**：后台每 5 分钟执行一次预警检测和发送，不依赖前端是否打开。
10. **产品用户统计**：支持按产品查看用户费用分布，统计指标包括每日用户数、人均费用、P50（中位数）、P10（前 10% 高消费用户费用门槛）、最大费用等。

### 5.2 隐性规则

1. **环境隔离**：测试环境和生产环境各自独立部署，每个实例只连接一个数据库。不存在跨环境数据查询。
2. **数据新鲜度**：原始数据每 5 分钟更新一次，因此系统展示的"实时数据"存在最多 5 分钟延迟。
3. **历史数据不变性**：已过去日期的费用数据不会变化，因此聚合表每天聚合一次即可满足需求。
4. **人员名称兜底**：当 personId 无法映射到人员姓名时，系统使用 "ID:xxx" 格式展示，确保不丢失信息。
5. **首次部署初始化**：系统首次启动时，会自动创建聚合表并初始化近 90 天的历史聚合数据。

### 5.3 权限或范围限制

1. **无需登录**：大屏系统无需登录认证，任何内网用户均可访问。
2. **预警接收人配置**：预警通知的接收人通过 alert_recipient_config 表配置，支持按预警类别（model/product/person/global）分别配置。global 类别作为兜底，当具体类别未配置接收人时使用全局配置。
3. **建议人与抄送人**：预警发送区分"建议人"（收到待办）和"抄送人"（仅查看），通过钉钉 userId 标识。
4. **环境可见性**：每个部署实例只能看到自己连接的数据库对应的环境数据，用户通过不同的访问地址区分测试/生产。

### 5.4 特殊说明或例外

1. **预警去重机制**：同一天对同一个预警对象（如某个模型），只发送一次预警。但如果预警等级升级（如从一级升到二级），会重新发送。
2. **7 天日均边界情况**：当近 7 天完全没有历史费用数据时，不产生预警（无法计算基准值）。当今日费用为 0 时，同样不产生预警。
3. **聚合延迟窗口**：凌晨 00:00 - 00:30 之间，前一天的数据尚未聚合完成，查询前一天数据时可能从原始表实时计算。
4. **模型编码与名称**：系统中使用 aiType 作为模型的唯一编码（如 "47"），展示时需要通过映射表转换为可读名称（如 "gpt-4o"）。这个映射关系可能随上游系统变化。

## 6. 常见歧义与边界

### 6.1 容易混淆的概念

1. **模型 vs 厂商**：模型是具体的 LLM（如 GPT-4o），厂商是提供模型的服务商（如 OpenAI）。系统同时支持按模型和按厂商两个维度统计，但厂商是通过模型编码映射出来的，不是原始数据中的独立字段。

2. **原始表 vs 聚合表**：原始表（ads_ai_cost_dashboard）存储每一条费用明细，数据量大；聚合表（ads_ai_cost_daily_agg）存储按维度和日期预计算的汇总数据。两者不是冗余关系，而是性能优化关系。

3. **bizCode vs 产品名称**：bizCode 是产品的技术标识（如 "myapp"），不是展示名称。产品名称（如 "AI问答"）通过映射表获取。

4. **aiType vs 模型名称**：aiType 是模型的编码（如 "47"），不是模型名称。模型名称（如 "gpt-4o"）通过映射表获取。

### 6.2 容易误解的规则

1. **预警不是预算告警**：预警判断的是"今天是否比最近几天突然花了很多"，不是"是否超出了预算上限"。系统目前没有预算概念。

2. **"今日费用"包含当天所有已同步的数据**：由于原始数据每 5 分钟更新一次，"今日费用"会随着数据同步逐步增加，不是固定值。

3. **"环比"的计算方式**：周环比和月环比使用同期对齐方式。例如"本周"对比"上周"时，不是对比完整的 7 天，而是对比到当前星期几为止的同等天数。月环比同理，对比到本月第几天的同期。

4. **测试/生产不是同一系统**：它们是两个独立部署的实例，各自连接各自的数据库，不共享数据。不存在一个系统同时查看两个环境的能力。

### 6.3 理解边界

1. **系统不产生数据**：本系统的所有数据来源于外部系统同步的费用记录。如果外部系统未同步数据，本系统无法展示任何内容。

2. **预警不控制费用**：预警仅起到通知作用，不会自动限制或暂停 LLM 调用。费用控制需要通过其他系统或流程实现。

3. **聚合表不包含当天数据**：聚合表每天凌晨聚合前一天的数据，当天数据始终从原始表实时查询。因此"今日"相关的所有统计都是实时计算的。

4. **人员维度的局限性**：人员统计依赖原始数据中的 personId 字段。如果上游系统未正确记录 personId，该条费用记录将无法归入具体人员。

## 7. 术语表

| 术语 | 说明 |
| ---- | ---- |
| 费用记录 | 以"产品+模型+人员+日期"为粒度的费用汇总数据，包含调用次数、Token 数、费用金额 |
| 模型 | 具体的大语言模型，如 GPT-4o、Claude 3.5 Sonnet、DeepSeek-V3 |
| 厂商 | 提供大语言模型的服务商，如 OpenAI、Anthropic、DeepSeek |
| 产品 | 公司内部使用 LLM 能力的业务模块，如 AI问答、AI翻译 |
| 人员 | 调用 LLM 的公司员工，以人员ID唯一标识 |
| aiType | 模型的技术编码，是系统中标识模型的唯一标识符，非可读名称 |
| bizCode | 产品的技术编码，是系统中标识产品的唯一标识符，非可读名称 |
| 聚合表（ads_ai_cost_daily_agg） | 按维度和日期预计算的费用汇总数据表，用于加速历史数据查询 |
| 原始表（ads_ai_cost_dashboard） | 存储费用明细数据的数据源表，由外部系统每 5 分钟同步 |
| 预警 | 当某维度今日费用相对于近 7 天日均出现异常增长时产生的告警信号 |
| 预警等级 | 分为一级（notice）、二级（warning）、三级（critical），等级越高表示异常越严重 |
| 预警分发 | 将预警通过 CWork 平台以协同待办形式发送给配置的接收人 |
| 周期对比 | 将两个时间周期（如今日 vs 昨日、本周 vs 上周）的费用数据进行对比分析 |
| 钻取 | 从排行榜中某一项点击进入，查看该项在下级维度的费用分布（如从某模型钻取到其产品分布） |
| 环比 | 两个相邻周期之间的费用变化率，使用同期对齐方式计算 |
| CWork | 公司内部协同办公平台，预警通过该平台以待办形式通知相关人员 |
| Nacos | 配置中心，用于管理数据库连接等运行时配置，支持热更新 |
| 联动查询 | 在排行榜中点击某一项，自动筛选该项在其他维度下的费用分布（如点击某人员，查看其使用的模型分布） |
| 产品用户统计 | 按产品维度分析用户费用分布，统计每日用户数、人均费用、P50（中位数）、P10（前10%高消费用户费用门槛）等指标 |
