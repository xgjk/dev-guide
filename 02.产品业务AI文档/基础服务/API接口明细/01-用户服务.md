## 二、典型业务流程

### 场景：按姓名同步员工组织架构

> 需求：通过员工姓名，拉取其完整的组织树关系进行对齐。

1. 调用 **3.1 按姓名搜索全部员工**（`GET /cwork-user/searchEmpByName`），传入 `searchKey="张三"`，获取 `SearchAddressbookVO`。
2. 从返回的 `inside.empList` 中，获取匹配的 `SearchEmployeeVO.id`（员工 id）。
3. 调用 **3.3 根据员工 id 获取组织架构信息**（`GET /cwork-user/employee/getEmployeeOrgInfo`），传入 `empId`，拿到 `EmployeeOrgInfoVO`，内含二级/三级部门名称及直属领导信息。

## 三、接口详细说明

### 3.1 按姓名搜索全部员工（含外部联系人）

按姓名模糊搜索全部员工，包含内部员工与外部联系人。

**基本信息**

| 项目 | 说明 |
| ------------ | ----------------------------------- |
| 接口地址 | `/cwork-user/searchEmpByName` |
| 请求方式 | `GET` |
| 数据量级别 | 小（通常 ≤ 20 条，约 2KB） |

**请求参数**

| 参数名 | 类型 | 必填 | 说明 |
| ----------- | ------ | ---- | ------------------------------ |
| `searchKey` | String | 是 | 搜索关键词：支持按姓名模糊搜索。**⚠️ 中文必须 URL 编码（UTF-8）** |

**请求示例**

```bash
# 示例：搜索姓名包含"张三"的员工
# 原始参数：searchKey=张三
# URL 编码后（UTF-8）：searchKey=%E5%BC%A0%E4%B8%89
curl -X GET 'https://{域名}/open-api/cwork-user/searchEmpByName?searchKey=%E5%BC%A0%E4%B8%89' \
  -H 'appKey: YOUR_API_KEY'
```

**响应参数**

`data` 类型为 `SearchAddressbookVO`，详见 **[4.1 SearchAddressbookVO](#41-searchaddressbookvo)**。

**响应示例**

```json
{
  "resultCode": 1,
  "resultMsg": "成功",
  "data": {
    "inside": {
      "companyVO": {
        "id": 12435,
        "name": "公司A",
        "dingCorpId": "dingxxxx"
      },
      "empList": [
        {
          "id": 10001,
          "personId": 20001,
          "name": "张三",
          "dingUserId": "ding2323",
          "title": "工程师",
          "mainDept": "技术部"
        }
      ]
    },
    "outside": []
  }
}
```

**数据流向**

- 返回的 `inside.empList[].id` 可用于 **3.3 根据员工 id 获取组织架构信息** 的 `empId` 入参。

### 3.2 根据 personId+corpId 批量获取员工信息

根据企业 ID 和用户 personId 列表，批量查询员工信息。

**基本信息**

| 项目 | 说明 |
| ------------ | --------------------------------------------------- |
| 接口地址 | `/cwork-user/employee/getByPersonIds/{corpId}` |
| 请求方式 | `POST` |
| Content-Type | `application/json` |
| 数据量级别 | 小~中（取决于批量数量） |
| 最大返回条数 | 建议单次传入 ≤ 100 个 personId |

**请求参数**

- **路径参数 (Path)**

| 参数名 | 类型 | 必填 | 说明 |
| -------- | ------ | ---- | ------ |
| `corpId` | Long | 是 | 企业 ID |

- **请求体 (Body)**

请求体为 `Long[]`（personId 列表，雪花算法或大整型）。

**请求示例**

```bash
curl -X POST 'https://{域名}/open-api/cwork-user/employee/getByPersonIds/123456' \
  -H 'appKey: YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '[10001, 10002]'
```

**响应参数**

`data` 类型为 `List<EmployeeVO>`，详见 **[4.5 EmployeeVO](#45-employeevo)**。

**响应示例**

```json
{
  "resultCode": 1,
  "resultMsg": "成功",
  "data": [
    {
      "id": 10001,
      "personId": 10001,
      "name": "张三",
      "dingUserId": "ding_zhangsan",
      "title": "产品经理"
    }
  ]
}
```

### 3.5 批量获取部门信息

根据部门 ID 列表批量获取部门信息，包含部门负责人 empId。

**基本信息**

| 项目 | 说明 |
| ------------ | --------------------------------------------------- |
| 接口地址 | `/cwork-user/department/listByIds` |
| 请求方式 | `POST` |
| Content-Type | `application/json` |
| 数据量级别 | 小~中（取决于批量数量） |

**请求参数**

请求体为 JSON 数组，元素为部门 ID（Long 类型）：

```json
[1512393131319586800, 1512393131319586801]
```

| 参数名 | 类型 | 必填 | 说明 |
| ------ | ------------ | ---- | ------------ |
| (body) | `List<Long>` | 是 | 部门 ID 列表 |

**请求示例**

```bash
curl -X POST 'https://cwork-api.mediportal.com.cn/open-api/cwork-user/department/listByIds' \
  -H 'appKey: YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '[1512393131319586800]'
```

**响应参数**

`data` 类型为 `List<DeptInfoVO>`，详见 **[4.8 DeptInfoVO](#48-deptinfovo)**。

**响应示例**

```json
{
  "resultCode": 1,
  "resultMsg": "成功",
  "data": [
    {
      "id": "1512393131319586800",
      "name": "技术中心",
      "deptLeaderId": "1234567890123456789",
      "parentId": "1512393131319586700"
    }
  ]
}
```

**数据流向**

- `deptId` 来源：BP 系统分组树中 `type=org` 节点的 `deptId` 字段，或其他业务场景中已知的部门 ID。
- 返回的 `deptLeaderId` 为部门负责人的员工 ID，可用于发送汇报等场景。

### 3.6 获取一级部门主管（支持指定用户）

优先获取指定用户的一级部门主管；如果未指定用户，则默认取当前用户的一级部门主管。

取数逻辑如下：

1. 查找目标用户所在部门（可能属于多个部门）。
2. 从每个所属部门向上递归查找，直到找到“上级为根组织”的部门（即一级部门）。
3. 获取该一级部门下标记为 `leader=true` 的部门领导。
4. 若指定用户本人就是一级部门领导，则返回本人。

**基本信息**

| 项目 | 说明 |
| ------------ | --------------------------------------------------- |
| 接口地址 | `/cwork-user/employee/getFirstLevelDeptManagers` |
| 请求方式 | `GET` |
| 数据量级别 | 小（通常 0~1 条，约 0.5KB） |

**请求参数**

| 参数名 | 类型 | 必填 | 说明 |
| ------- | ---- | ---- | ------ |
| `empId` | Long | 否 | 指定用户的员工 id；不传则取当前用户 |

**请求示例**

```bash
# 场景1：指定用户
curl -X GET 'https://{域名}/open-api/cwork-user/employee/getFirstLevelDeptManagers?empId=10001' \
  -H 'appKey: YOUR_API_KEY'

# 场景2：不指定用户（默认当前用户）
curl -X GET 'https://{域名}/open-api/cwork-user/employee/getFirstLevelDeptManagers' \
  -H 'appKey: YOUR_API_KEY'
```

**响应参数**

`data` 类型为 `List<EmployeeVO>`，详见 **[4.5 EmployeeVO](#45-employeevo)**。

**响应说明（边界场景）**

- 大部分情况下返回 1 条数据。
- 可能返回空数组：如用户无部门，或用户直接挂在根组织下（无一级部门）。
- 可能返回多条数据：如用户在多个部门任职，且可回溯到多个一级部门。

**响应示例**

```json
{
  "resultCode": 1,
  "resultMsg": "成功",
  "data": [
    {
      "id": 10002,
      "personId": 10002,
      "name": "李四",
      "dingUserId": "ding_lisi",
      "title": "部门负责人"
    }
  ]
}
```

## 四、公共数据结构

### 4.1 SearchAddressbookVO

企业通讯录搜索结果。

| 字段 | 类型 | AI决策 | 说明 |
| --------- | ------------------------- | ------ | --------------------------------- |
| `inside` | `SearchCompanyRelationVO` | 是 | 内部员工列表，详见 **[4.2](#42-searchcompanyrelationvo)** |
| `outside` | `SearchCompanyRelationVO[]` | 是 | 外部联系人，详见 **[4.2](#42-searchcompanyrelationvo)** |

### 4.2 SearchCompanyRelationVO

关联企业信息。

| 字段 | 类型 | AI决策 | 说明 |
| ----------- | ------------------- | ------ | ------------------ |
| `companyVO` | `CompanyVO` | 是 | 关联的企业信息，详见 **[4.3](#43-companyvo)** |
| `empList` | `SearchEmployeeVO[]` | 是 | 关联企业下的员工信息，详见 **[4.4](#44-searchemployeevo)** |

### 4.3 CompanyVO

企业信息。

| 字段 | 类型 | AI决策 | 说明 |
| ------------ | ------ | ------ | ----------- |
| `id` | Long | 是 | 企业 id |
| `name` | String | 是 | 企业名称 |
| `dingCorpId` | String | 否 | 钉钉 CorpId |

### 4.4 SearchEmployeeVO

搜索结果展示的员工信息。

| 字段 | 类型 | AI决策 | 说明 |
| ------------ | ------ | ------ | --------------------------- |
| `id` | Long | 是 | 员工 id |
| `personId` | Long | 是 | 用户 personId（与企业无关） |
| `name` | String | 是 | 姓名 |
| `dingUserId` | String | 否 | 钉钉 userId |
| `title` | String | 是 | 职位 |
| `mainDept` | String | 是 | 所在部门（只取一条数据） |

### 4.5 EmployeeVO

批量查询返回的员工信息。

| 字段 | 类型 | AI决策 | 说明 |
| ------------ | ------ | ------ | --------------------------- |
| `id` | Long | 是 | 员工 id |
| `personId` | Long | 是 | 用户 personId（与企业无关） |
| `name` | String | 是 | 姓名 |
| `dingUserId` | String | 否 | 钉钉 userId |
| `title` | String | 是 | 职位 |

### 4.6 EmployeeOrgInfoVO

员工组织架构信息。

| 字段                    | 类型 | AI决策 | 说明              |
|-----------------------| ------ | ------ |-----------------|
| `empId`               | Long | 是 | 员工 id           |
| `empName`             | String | 是 | 员工名称            |
| `managerEmpId`        | Long | 是 | 直属领导的员工 id      |
| `managerEmpName`      | String | 是 | 直属领导名称          |
| `secondLevelDeptId`   | Long | 否 | 组织架构第二级部门 id    |
| `secondLevelDeptName` | String | 是 | 组织架构第二级部门名称（中心） |
| `thirdLevelDeptId`    | Long | 否 | 组织架构第三级部门 id    |
| `thirdLevelDeptName`  | String | 是 | 组织架构第三级部门名称（部门） |
| `corpId`              | Long | 是 | 企业id            |

### 4.7 EmpPositionVO

员工岗位信息。

| 字段 | 类型 | AI决策 | 说明 |
| --------- | ------ | ------ | --------------------------------- |
| `id` | Long | 否 | 岗位 id |
| `name` | String | 是 | 岗位名称 |
| `jobDuty` | String | 是 | 岗位职责 |
| `state` | String | 是 | 岗位状态（`NORMAL`：正常，`CLOSED`：停用） |

### 4.8 DeptInfoVO

部门信息（精简版）。

| 字段 | 类型 | AI决策 | 说明 |
| -------------- | ------ | ------ | --------------------------------- |
| `id` | Long | 是 | 部门 ID |
| `name` | String | 是 | 部门名称 |
| `deptLeaderId` | Long | 是 | 部门负责人 empId |
| `parentId` | Long | 否 | 父部门 ID |
