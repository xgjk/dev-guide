## 一、系统概述
**基础服务系统**是支撑全业务线运行的基石，主要由 **身份(Identity)** 与 **资产(Asset)** 两大支柱构成。它通过标准化的人员、文件、标签和 AI 实体管理，为上层应用提供统一的底层支撑。

## 二、系统核心对象
基础服务构建于以下几类高度抽象的核心资源之上：

### 2.1 身份与组织核心对象 (Identity)
| 对象 | 业务说明 | 产生/关联 | 用途定位 |
| ---- | ---- | ------------ | -------- |
| **员工** | (Employee) 具有主键标识的自然人角色 | 组织架构/岗位 | 系统内一切权限、行为和资产归属的最终主体 |
| **组织节点** | (Dept/Company) 业务边界节点 | 树形结构 | 决定数据的天然归属及跨组织物理隔离边界 |
| **虚拟员工** | (Virtual Employee) 真实用户关联的 NPC 实物 | 真实员工 (Owner) | 用于承担 AI 助理、自动化流程等非真实自然人任务 |
| **AI 机器人** | (AI Robot) 接入生产环境的智能通道实体 | agentId / 插件配置 | 负责多端（如 IM、API）的交互承载与自动响应 |

### 2.2 资产与标记核心对象 (Asset & tagging)
| 对象 | 业务说明 | 产生/关联 | 用途定位 |
| ---- | ---- | ------------ | -------- |
| **文件** | (File) 实际的数字业务载体 | resourceId / 转换记录 | 提供预览、下载和流转的业务客体 |
| **个人标签** | (Personal Tag) 用户自定义的分类标识 | 员工 (Creator) | 解决标准组织架构外，个人视角下的高灵活性分类需求 |

## 三、核心业务规则

### 规则一：身份一元化与虚拟实体绑定
系统中所有的 AI 机器人或虚拟员工（NPC），必须显式关联到一个真实的 Employee 账号。资产的最终所有权及行为审计将穿透虚拟层级，回归到背后的真实自然人。

### 规则二：标签的独立性与业务关联性
个人标签仅由基础服务维护其“定义（Definition）”，具体的“引用关系（Reference）”由各业务模块（如汇报、任务）自行维护。删除标签定义后，业务模块应负责清理其冗余引用。

### 规则三：资产流转的 TTL 机制
为保障安全，所有基于 resourceId 产生的物理文件访问链接均设有 **Time To Live (TTL)** 限制（通常为 1 小时）。严禁在业务库中直接持久化存储这些带 Token 的下载链接。

### 规则四：组织权限的“左移”校验
数据隔离不依赖物理路径，而是依赖用户服务提供的组织上下文。任何跨组织的数据交换必须通过授权（Auth）与白名单（WhiteList）机制。

## 四、主要模块职责

1. **组织人事管理 (Identity)**：维护入、离、调、转生命周期，提供 `OrgStructure` 精准数据源。
2. **多模态对象存储 (Storage)**：管理文件的分片上传、断点续传及对象存储（如七牛/内网）的调度。
3. **内容加工工厂 (Fabric)**：提供富文本转 PDF、文档在线协同转换及 AI 音转文识别功能。
4. **分类标记中心 (Tagging)**：提供轻量级的个人与业务标签库服务。
