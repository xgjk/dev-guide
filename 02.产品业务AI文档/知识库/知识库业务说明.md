## 一、系统概述
**知识库系统（Document Database）** 是一套用于组织内部 **沉淀数字资产、管理文档权限、促进文件协作、追踪操作轨迹** 的文件与知识管理平台。

系统以 **文件（含文件夹）** 作为核心信息载体。文件不仅包含具体的知识文档，也包含用于组织这些文档的文件夹结构。用户通过创建或上传文件同步资料，组织可以通过划分 **项目（空间）** 来隔离不同维度的知识，管理者可以通过 **权限（Grant）与标签（Label）** 进行安全管控与分类。当需要在内部或外部进行信息流转时，可以通过 **分享（Share）** 进行内部受控流通，或通过 **外发（OutSend）** 进行外部受控流转。

系统通过 **云端存储 → 权限管控 → 协作共享 → 版本追溯** 的方式，形成组织数字资产的管理与流转闭环。

## 二、系统核心对象
系统主要由以下核心对象构成：

| 对象 | 说明 | 可以产生 | 产生说明 |
| ---- | ---- | -------- | -------- |
| 空间 (Project) | 知识库的顶层容器，通常对应团队或独立的工作区 | 文件/权限规则 | 空间下可建立无限层级的文件结构 |
| 文件夹 (Folder) | 文件组织方式的一种特殊对象或层级容器 | 文件/子文件夹 | 用于结构化管理文档 |
| 文件 (File) | 系统的核心知识对象，也是产生版本、批注、分享、外发等行为的依附主体 | 版本/分享/外发/批注 | 对文件的修改、共享都会被记录和追踪 |
| 资源 (Resource) | 基础服务中的真实文件元信息对象，记录存储方式与存储位置（如 key/path），知识库仅引用该对象 | 资源标识（resourceId） | 每次上传都会生成新的 resourceId；即使同一文件重复上传也会生成不同 resourceId（底层可复用同一物理文件） |
| 权限 (Grant) | 不是知识内容对象，而是作用于空间和文件的访问控制规则 | 权限变更记录 | 决定哪些人（/部门/角色）可以查看或编辑 |
| 分享 (Share) | 将文件在内部受控流通的授权凭证 | 内部访问记录 | 面向内部同事，链路受限且相对信任 |
| 外发 (OutSend) | 将文件向外部环境受控流转的凭证 | 外部访问记录/下载记录 | 面向外部人员，需强力安全管控和追踪 |
| 版本 (FileVersion) | 文件每次更新产生的历史快照 | 恢复操作 | 用户可回溯并恢复所需的文件版本 |
| 批注 (FileAnnotation) | 围绕文件内容展开的讨论和标记 | 回复 | 不同人员可基于文档内容进行沟通 |

其中：
**文件是核心载体，权限是基础保障。** 文件夹作为特殊的容器对象，辅助管理文件层次结构。而版本、批注、分享、外发等流转和历史行为，均紧密依附于具体文件本身。  
当 File 为实体文件时，FileVersion 会引用一个 Resource（resourceId）；知识库不直接存储真实文件，仅维护业务对象与资源引用关系。

## 三、核心业务关系与场景

### 3.1 核心业务关系
系统围绕 **文件与空间** 组织知识管理。

基本从属与流转关系如下：
```
空间 (Project/Workspace)
 └─ 文件 / 文件夹 (File/Folder)
     ├─ 权限管控 (Grant) 👉 验证用户身份与操作边界
     ├─ 协作编辑 👉 产生 版本 (FileVersion) & 批注 (Annotation)
     │                 └─ 版本内容引用 资源 (Resource/resourceId)
     ├─ 内部流通 👉 产生 分享链接 (Share)
     ├─ 对外流转 👉 产生 外发包 (OutSend)
     └─ 风险监控 (AlertRecord) 👉 异常访问与敏感操作拦截
```

当新用户需要访问特定限制文件时：
```
越权访问拦截
 ↓
提交权限申请 (FileGrantApply)
 ↓
管理者审批赋权
 ↓
获取文件访问与操作能力
```

因此系统整体形成如下管理流转：
```
文件上传/创建 → 分类/标签 → 赋权管控 → 协作/流转 → 归档与版本追溯
```

### 3.2 典型业务场景
知识库系统在核心流转框架下，能够支撑以下三种高频典型业务路径：

1. **个人空间沉淀文档**：用户在个人的私密空间或草稿区新建文件，进行资料整理或初稿撰写。此阶段权限严格封闭，利用历史版本记录防止误删，利用文件夹进行个人知识分类。在内容成熟后，再将其流转至团队或公共空间。
2. **团队空间共享协作**：项目组建立独立空间，配置团队成员的基础权限。成员在空间内部基于同份文件（如文档、表格等）高频沟通批注（Annotation）或并发协作。系统依靠空间级的基础授权向下继承，结合文件级精细化管控，形成顺畅的内部共创闭环，并可使用分享（Share）向跨部门同事推广流转。
3. **外部受控外发文件**：当企业需要将定档的资料发送给客户或外部合作伙伴等非内部成员时，业务人员创建外发（OutSend）任务。在外发中配置查阅次数、有效时长、动态水印和防下载阻断等安全策略。文件不再随意于系统内查看流转，而是以高安全水位向外部进行传输，一切外部访问动作全程留痕与受限。

## 四、业务规则
以下规则用于说明系统的重要业务逻辑。

### 规则一：空间是最高层级的隔离单位
在系统中，所有的文件必须从属于某一个具体的 **空间（Project）**。
空间可以设定基准的协作者名单与角色，这些规则会默认向下继承到空间内的所有文件中。

### 规则二：权限支持精细化拦截与向下继承
权限（Grant）不仅控制能否“查看”，还可以控制能否“编辑、下载、分享”等精细化动作。
通常情况下，文件夹上的权限会**自动继承**给其子文件；但在特殊保密场景下，支持对某个子文件进行**单独受控**（权限阻断与独立重置），从而打破继承链。

### 规则三：版本自动记录策略
当用户重新上传同名文件或通过在线编辑保存文档时，系统自动生成新的 **文件版本（FileVersion）**。
历史版本不会被覆盖，而是作为留存节点可供比对或恢复。这是防止知识资产被误删或遭篡改的重要机制。
当触发上传生成新版本时，会同步生成新的 **resourceId** 并指向对应 Resource。即使上传的是同一个源文件，多次上传也会得到不同的 resourceId；底层是否复用同一物理存储文件由基础服务决定。

### 规则四：内部协助与外部流转链路隔离
文件在内部协作与流转主要通过调整 **权限（Grant）与协作批注** 进行，若仅作单次查看，也可借助**分享（Share）**完成企业内部人员的快速流转；
而若需要交于外部人员查看，必须通过独立配置的 **外发（OutSend）** 机制。外发包含专属的对外追踪能力和有效性限制（如阅读次数控制、水印强制启用），在底座上与内部协作链路是物理隔离的两套体系。

### 规则五：操作全链路留痕
对于文件的任何增改删移动、权限调整、分享外发与访问下载行为，系统均会记录 **操作日志（OperationLog）**。
若系统接入了敏感词预警或大批量异地异常下载判断策略，一旦触发相关安全基线，还会额外产生 **预警记录（AlertRecord）** 以保障资产安全。

## 五、常见歧义与边界
针对部分容易混淆的知识库系统核心概念，必须严格划定此系统的相关边界定义：

- **空间 ≠ 文件夹**：空间有独立且为最高层级的角色、权限配置与配额/度量统计能力；文件夹仅仅是在已有的空间下，组织文件的嵌套树形层级容器。
- **分享 ≠ 外发**：分享针对内部用户流动，前提是访问者自身具备系统操作账号，且处于受组织架构制约的安全网络内；而外发重点面向系统外部、不具备组织内部身份的人员，文件流转脱离内网并配以极高强度的管控和追踪策略。
- **删除 ≠ 永久删除**：常规“删除”仅是将文件放入回收站，“永久删除”（或清空回收站）才代表文件的物理抹除、空间释放与深度阻断恢复。
- **版本回退 ≠ 日志消失**：将文件回退到数周前的一个特定版本，仅仅是将当前状态指针还原为过去的定格状态；中间经历的其他修改以及这期间形成的操作日志并不会因此而消失，“产生过被废弃的版本”这一动作记录作为存证依然留底。
- **知识库文件（File）≠ 真实文件内容本体**：知识库存的是业务对象与版本关系；真实文件元信息由基础服务 Resource 承载，知识库通过 resourceId 引用，不直接持有底层文件实体。
- **文件可见 ≠ 必然可编辑或可下载**：用户能看清文件名称结构，乃至预览其文档全文，与他能否下载物理原文件至本人本地终端是基于两套物理隔离管控的不同粒度权限。
- **权限继承 ≠ 绝对继承**：默认情况下文件夹级权限能辐射至它的全部晚辈后代树节点，但在复杂知识防范场景下，可以特别将特定节点单独配置防控制规则以打破继承链，不再绝对受制于甚至推翻父目录权限。

## 六、主要功能
系统当前主要提供以下功能框架：

### 文件管理（涵盖文件及文件夹）
用于结构化存储与维护知识资源。
- 支持新建文档、表格及上传多种格式附件。
- 支持对象重命名、移动变轨、复制及对应回收站生命周期等操作。
- 管理文件多维分类标签（Label）设定与精确检索。
- 支撑文件各版本差异记录、旧版本回退与历史查阅。

### 权限与审批控制
用于建立各空间和各文档内部信息的流转阻断边界。
- 支持对企业部门架构、各类系统角色配置以及面向特定账号个人的多颗粒度权限赋权（含访问、编辑、外发等动作）。
- 权限未覆盖、被拦截防范访问之时可以经由系统向上呈递 **权限申请（Grant Apply）** 寻求管理者特批。
- 完整包含核心要素变化操作时的流转变更日志追踪体系（GrantChangeLog）。

### 共享流转与外部传递分发
负责文件信息安全与有效率的可控传递。
- 支持一键分享快速为内部授权人员开具基于可访问短链接的流通窗口。
- 内置强级别管控能力与安全阻绝模块（不可下载、安全水印等）确保外部敏感流通使用外发模块分发资源。

### 文档协作与互动
围绕数字知识的内容载体的沟通形式。
- 文件多人协作并发在线编写以及独霸式锁定保障能力。
- 在字里行间的区域开展圈定批注（Annotation）及引发多轮快速文字恢复互动并支持归档，形成去口水化的协作生态。

## 七、系统运行模式
知识库系统通过以下底层机制循环运行：

```
沉淀知识 (新建/上传文件至空间体系内并以文件夹管理)
 ↓
确保安全 (空间层级基准赋权确立 / 特殊文件精准受控防护)
 ↓
协同编辑 (内容批注发起研讨 / 多用户多版本无缝协作流转)
 ↓
合规利用 (内部受限制的分享 / 带有追溯和时效管控机制的外部外发分发)
```

该机制使组织的数字知识资产形成 **可积累、可检索、安全受控且易于流通** 的生态闭环。

## 八、术语表
| 术语名称 | 英文标识 | 业务含义 |
| -------- | -------- | --------  |
| 空间 | Project | 知识库最高级隔离单元，类似于独立工作盘或大本营。控制全局根节点配置。 |
| 文件 | File | 数字资产实体，一切知识内容、在线协同和版本、外发流转动作发生的核心载体。 |
| 资源 | Resource | 基础服务中的真实文件元信息对象，记录存储方式（七牛/MinIO/FTP）与存储位置；知识库仅通过 resourceId 与其建立引用关系。 |
| 文件夹 | Folder | 一类特殊的文件容器对象，主要用于分类和组织嵌套树形的目录结构。 |
| 权限 | Grant | 基于各受体作用于空间或对应文件的访问控制规则判定（阅读/修编/下载等动作控制），此概念并不是内容素材本体。 |
| 分享 | Share | 面向组织内部的受控信息流转形式，利于已受保护资料在体系内员工、同事成员中快速转交引荐。 |
| 外发 | OutSend | 面向组织外部的受控独立分出行为，由于缺乏内网等客观防护环境其通常搭载着更加严厉的安全防御机制（如下载阻断限制/强力数字水印/时效过期判定与事后精密审计追踪）。 |
| 版本 | FileVersion | 每逢核心文件内容经线上修改、或本地重复覆盖提交所系统打出的独立快照节点，它用于回溯往事记录以及当面灾害误操作等节点的回滚。 |
| 批注 | FileAnnotation | 对于文档页面内部字句及特殊模块发起的圈点和研讨论述回复，它天然支持紧靠文章内容的业务协作与沟通留证。 |
| 权限申请 | GrantApply | 使用者触碰到被拒绝访问隔离栏栅时系统弹向特定管理者发出请求赋加权限的互动审批诉求单流程。 |
