## 一、概述

本文档描述了 **知识库服务** 对外开放的全部 Open API 接口。在新版文档结构中，我们按**领域驱动 (Domain-Driven)** 原则将繁杂的接口拆分成了五个独立的明细模块。通过这些接口，可以实现：
1. 目录树遍历、空间搜索与节点管理
2. 物理文件的资源挂载与版本演进
3. 纯文本与 Markdown 的高速存取通道
4. 终端 UI 阅读视图与流式加载
5. 专为大语言模型打造的长程记忆沙盒

## 二、核心领域模型与名词解释 (必读)

为了消除调用方（特别是 AI Agent）在参数传递上的混淆，请在调用任何接口前，务必深刻理解以下三个核心概念：

### 2.1 底层存储模型 vs 知识库业务模型
- **资源 (Resource) / `resourceId`**：代表纯粹的**底层物理文件**（存放在 MinIO、七牛云或本地磁盘）。它是不可变的（Immutable），只包含字节流、大小、MD5 等物理元数据。它**不包含**任何业务属性（它不知道自己叫什么名字，也不知道在哪个文件夹下）。
- **节点 (Node/File) / `fileId`**：代表知识库中的**逻辑业务实体**。它可能是一个“文件夹（目录）”，也可能是一个“文件（文档）”。它带有强烈的业务属性：归属哪个空间、父级是谁、谁有权限看、当前处于哪个版本等。

### 2.2 版本控制的本质
知识库的“文件 (`fileId`)”实际上是一个**资源的容器/绑定器**。当文件内容更新时，`fileId` 保持不变，但其背后会关联一个新的版本记录（Version），该版本记录指向一个新的物理 `resourceId`。

### 2.3 路径 (Path)
在系统内部，所有的父子结构都是通过 `parentId` 和 `id` 构建的树。虽然部分接口（如 `saveFileByPath`）允许传入带 `/` 的绝对路径，但在双向同步或精细化管理场景下，强烈建议：**先通过路径查询或创建真实目录，获取到目录的 `fileId`，然后始终以 `parentId` 的形式进行子节点操作**。

### 2.4 空间 (Project) 与“个人知识库”
- **空间 (Project / `projectId`)**：知识库的最顶层容器。所有文件都必须归属于某一个空间。
- **个人知识库的本质**：它并非一套独立的系统，而是**一个特殊的 `projectId`**。它的特殊性在于：每个用户拥有唯一的一个个人空间，由系统自动分配，且没有复杂的权限体系（仅归属人可见）。
- **接口互通性**：
  1. API 中有部分标明为“个人知识库专用”的捷径接口（如 `uploadContent`、`4.10 将文件保存到个人知识库`），这些接口在调用时**不需要传 `projectId`**，后端会根据当前身份的 `appKey` 自动找到该用户的个人空间并写入。
  2. 对于通用接口（如 `4.16 saveFileByParentId` 等带 `projectId` 参数的接口），**同样适用于个人知识库**。你只需要先调用 `4.8 获取个人知识库空间ID` 拿到自己的 `projectId`，再将其作为参数传入通用接口即可。

## 三、通用说明

### 3.1 访问地址与环境
*   **生产环境**: `https://sg-al-cwork-web.mediportal.com.cn`
*   **统一前缀**: `/open-api/`

### 3.2 公共请求头与鉴权
*   所有接口必须在 HTTP Header 中携带 `appKey`。
*   默认按当前 appKey 映射的用户身份过滤数据范围，实行严格越权隔离。

### 3.3 通用响应结构
所有接口响应均返回统一的 `Result<T>` 结构：
```json
{
  "resultCode": 1,
  "resultMsg": null,
  "data": null
}
```
*   `resultCode`: `1` 表示成功，非 `1` 表示失败。
*   空值约定：`null`、空串 `""`、空数组 `[]` 语义相同，调用方无需区分。

### 3.4 面向 AI 消费的数据设计
*   **最小化原则**：列表接口仅返回元数据（名称/类型/大小/后缀），不返回正文内容。
*   **格式提纯**：`getFullFileContent` 专为 AI 设计，返回全局提纯的 Markdown 文本，保留标题/列表/表格等语义。注意：**大文件可能存在异步解析延迟**。
*   **去重原则**：`FileVO` 中的 `ancestorNames` 已包含完整路径信息，不重复返回多级嵌套体。
*   **正文获取的两条入口（v2 起对齐）**：
    - **`getFullFileContent`**（详见 [03-AI与纯文本高速通道.md](API接口明细_v2/03-AI与纯文本高速通道.md)）：永远返回全量正文，AI / 拼接 prompt 首选。
    - **`getFileContent`**（详见 [04-UI终端预览与阅读.md](API接口明细_v2/04-UI终端预览与阅读.md#43-获取ui阅读模式根据文件id和页码分页获取内容)）：UI 流式分页阅读首选；
      对**有分页类型**（pdf / pptx / ppt / docx / doc 等）按 `pageNumber` 返回单页文本；
      对**无分页类型**（md / txt / html / htm / xml / notex / xls / xlsx）忽略 `pageNumber`，**直接返回全量正文，与 `getFullFileContent` 内容等价**。

## 四、核心业务工作流 (Workflow)

我们针对不同的消费主体，梳理了以下两条最佳实践工作流：

### 4.1 分类 A：面向 AI Agent 的自动化工作流
*   **工作流 A1：纯文本快写与双向同步**
    1. AI 生成 Markdown 报告。
    2. 调用【纯文本高速通道】的 `uploadContent`，传入 `folderName` 极速存盘，得到 `fileId`。
    3. 若后续内容有修改，传入 `updateFileId` 追加新版本。
*   **工作流 A2：知识检索与 RAG 喂饭**
    1. 调用【空间与目录树管理】的 `searchFile` 圈定文件范围，拿到一组 `fileId`。
    2. 调用【纯文本高速通道】的 `batchGetContent` 批量拉取这组文件的提纯 Markdown 内容。
    3. 将内容丢给 LLM 处理。
*   **工作流 A3：物理文件更新绑定**
    1. AI 重新生成了一份 PDF 报告，调用底层文件存储接口上传，得到 `resourceId`。
    2. 调用【物理文件入库与版本控制】的 `updateFileVersion`，将老文件的 `fileId` 与新的 `resourceId` 绑定，生成 V2.0 版本。
*   **工作流 A4：知识库冷启动与增量同步（推荐）**
    1. **冷启动全量**：调用【空间与目录树管理】的 `listDescendantFiles(rootFileId, suffix=md, cursor/limit)` 分页拉取子树全部 Markdown 元数据。
    2. **按需拉正文**：仅对需要进入向量库/索引的文件，再调用 `getFullFileContent(fileId)` 获取提纯正文。
    3. **增量轮询**：周期性调用 `listChanges(since=lastSyncTime, cursor/limit)` 拉取变更，直到 `nextCursor` 为空；使用返回的 `serverTime` 更新本地水位。
    4. **对账优化**：对 `event=upsert` 的文件，可先用 `batchGetMeta(fileIds)` 批量检查元数据，必要时再拉全文；对 `event=delete` 则按语义删除本地索引/缓存。
    5. **显式建目录**：需要提前预置目录结构时，调用 `createFolder(projectId,parentId,name,cover/autoRename)` 创建空目录节点。

### 4.2 分类 B：面向前端 / 客户端界面的工作流
*   **工作流 B1：文件管理器导航**
    1. 调用【空间管理】获取用户权限空间。
    2. 调用 `getLevel1Folders` 拉取顶级目录。
    3. 用户点击某文件夹，通过 `getChildFiles` (`parentId`) 动态下钻。
*   **工作流 B2：在线流式阅读**
    1. 拿到 `fileId` 后，调用【终端预览】的 `getDownloadInfo`，将返回的 `previewUrl` 分发给 IFrame 或 H5 套件渲染展示。
    2. 若需要在 UI 内做"段落分页 / 文本流式渲染"，可直接调【终端预览】的 `getFileContent(fileId, pageNumber)`：
       - **有分页类型**（pdf/pptx/docx 等）：按 `pageNumber` 取该页文本；
       - **无分页类型**（md/txt/html 等）：`pageNumber` 被忽略，返回全量正文（与 `getFullFileContent` 内容一致）。

## 五、错误码说明

| resultCode | 说明 | 参考信息 / 异常原因 |
| ---------- | ---------------------- | ------------------------------------------ |
| **1** | **请求成功** | success |
| **0** | **通用失败** | failure |
| **500** | **系统异常/内部错误** | 系统崩溃、微服务超时或内部异常 |
| 610002 | `appKey` 无效 | 应用密钥不匹配或未分配 |
| 610012 | **请求太频繁（触发限流）**| 超过 QPS 限制，请休眠后重试 |

*(其它详细鉴权、企业用户拦截等错误码同原有系统)*
