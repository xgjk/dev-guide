## 2. 系统概述
​
### 2.1 系统一句话定义
​
TBS 是面向医药业务人员的 **AI 拜访陪练与训战平台**：用可配置的 **训练场景** 与 **虚拟医生**，让代表在安全环境中进行多轮 **练习** 或 **训战**，并沉淀可回溯的评分、复盘与统计数据。
​
### 2.2 系统解决的核心问题
​
- **培训与真实拜访情境脱节**：把医院、科室、品种与拜访任务固化进 **训练场景**，让练习对象接近真实沟通，而不是孤立背知识点。
- **高质量带教难以规模化**：由 **虚拟医生** 承担对话对象，由 **教练能力**（访前提示、过程反馈、事后复盘）承担结构化反馈，降低对真人一对一演练的依赖。
- **训练过程难衡量**：把每次 **练习** 或 **训战** 固化为 **训战记录**，支持按人员、**训练场景**、品种、**训战活动** 等维度查看表现与趋势。
​
### 2.3 系统在整体业务中的定位
​
- 上游依赖：企业账号与组织关系、经医学或产品审核的 **产品知识**、按规范撰写的 **训练场景** 与 **医生人设** 等内容供给。
- 下游产出：**训战记录** 与聚合统计，供培训管理、一线辅导与管理复盘使用。
- 服务形态：**训战管理后台** 承担配置、运营、分析与权限；**学员端** 承担选择 **训练场景**、完成对话训练、查看复盘；学员端可嵌入企业自有 App。
​
### 2.4 系统不负责解决的问题
​
- 不替代真实拜访中的医患关系与合规终审责任。
- 不承担完整学习管理系统的全部能力（例如课程体系管理、线下排课、薪酬绩效）；与「学习视频」等能力并存时，以当前配置范围为准。
- 不在本说明中约定技术实现方式与对外系统对接规则。

## 3. 核心业务对象
​
### 3.1 训练场景
​
- **定义**：一次「拜访在什么情境下发生」的业务描述单元，包含拜访背景与任务、代表可见的说明，以及仅用于驱动 **虚拟医生** 与 **教练能力**、不对代表直接展示的内部说明。
- **作用**：作为学员进入对话训练的入口载体；决定「练的是哪一段拜访」。
- **与其他对象的关系**：关联品种、科室、业务领域等维度；挂接 **医生人设** 与 **产品知识**；可被纳入 **训战活动**；每次训练会产生一条 **训战记录**。
- **常见误解**：**训练场景** 不是某一个具体界面，而是可发布、可复制、可对学员停止开放的业务内容单元。
​
### 3.2 医生人设（模拟对象）
​
- **定义**：**虚拟医生** 的沟通风格、关注点与难度等特征的组合，决定对话对象如何回应代表。
- **作用**：在同一套 **训练场景** 正文下，用不同 **医生人设** 改变对话张力与训练侧重点。
- **与其他对象的关系**：在 **训练场景** 中被选用；**训战记录** 会体现当时所用人设或难度。
- **常见误解**：**医生人设** 不是独立的一门课程，而是依附于 **训练场景** 的对话对手设定。
​
### 3.3 产品知识
​
- **定义**：与品种相关的、结构化沉淀的专业信息条目（如适应症、用法用量、关键说法等），作为训练时专业表述的依据。
- **作用**：约束与支撑对话与点评中的专业表述，减少与业务事实不一致的内容。
- **与其他对象的关系**：按品种维护；通过配置与 **训练场景** 关联；维护职责常由 **知识库专员** 承担。
- **常见误解**：**产品知识** 不是 **训练场景** 正文本身，而是可被多个 **训练场景** 复用的底层素材。
​
### 3.4 训战活动
​
- **定义**：在明确时间范围内，将多个 **训练场景** 打包成一个运营单元的容器，用于集中训战或项目化运营。
- **作用**：统一活动周期与可见范围，便于学员按活动参与、便于按活动统计。
- **与其他对象的关系**：包含多个 **训练场景**；学员发起训练时可携带活动来源信息；**训战记录** 可与 **训战活动** 关联。
- **常见误解**：**训战活动** 不是「单次对话」；单次对话属于某次训练过程，**训战活动** 是更高一层的编排名。
​
### 3.5 训战记录
​
- **定义**：某人在某次 **练习** 或 **训战** 中，从进入到产生评分与复盘为止的完整过程与结果快照。
- **作用**：支撑个人复盘、管理者抽查、导出与训练数据分析。
- **与其他对象的关系**：依附于人员与 **训练场景**（及当时选用的 **医生人设**、品种等）；可与 **训战活动** 关联；区分 **练习** 与 **训战** 两种类型。
- **常见误解**：**训战记录** 不是 **训练场景** 本身；**训练场景** 被删除或停止开放后，可能影响历史展示的完整性，但记录作为「当时一次训练」的事实是否保留，以企业数据策略为准。
​
### 3.6 企业与人员权限
​
- **定义**：多企业隔离下的成员身份，以及 **企业管理员、品种管理员、知识库专员** 等角色及其可管理的品种或数据范围。
- **作用**：决定谁能配置 **训练场景** 与 **训战活动**、谁能维护 **产品知识**、谁仅能以学员身份训练。
- **与其他对象的关系**：权限作用于基础数据、**训练场景**、**训战活动**、**产品知识**、**训战记录** 可见范围等全部核心对象。
- **常见误解**：「能登录」不等于「能改内容」；改内容与查看全量 **训战记录** 通常需要明确角色与品种范围。

## 4. 核心业务关系
​
### 4.1 业务主链路
​
1. 企业在 **训战管理后台** 维护基础维度（领域、科室、品种等）、**产品知识**、**医生人设**。
2. 内容负责方配置 **训练场景**，为 **训练场景** 选用 **医生人设** 并关联所需 **产品知识**。
3. 运营方按需编排 **训战活动**，将多个 **训练场景** 纳入同一活动与时间窗。
4. 学员在 **学员端** 选择 **训练场景** 与 **练习 / 训战** 模式，完成多轮对话；具备相应权限的人员也可在 **训战管理后台** 发起同类对话训练（用于内容验证或辅导）。
5. 系统生成 **训战记录**（含评分、维度、对话与教练反馈等），供学员复盘与管理人员查询、导出及看板分析。
​
### 4.2 对象之间的关系
​
- **训练场景** 依附于企业的品种、科室、领域等分类，并 **选用** **医生人设**、**关联** **产品知识**。
- **训战活动** **聚合** 多个 **训练场景**，不替代 **训练场景** 自身的生命周期（未对学员开放、已对学员开放、已停止对学员展示等）。
- **训战记录** 由学员在某 **训练场景** 下发起的一次训练 **产生**，并带有 **练习** 或 **训战** 类型；可与 **训战活动**、品种等来源信息绑定。
- **企业与人员权限** 决定谁能 **创建 / 修改** **训练场景**、**训战活动**、**产品知识**，以及谁能 **查看** 哪些 **训战记录** 与汇总数据。
​
### 4.3 关键流转逻辑
​
- **内容侧**：**训练场景** 从「未对学员开放」到「已对学员开放」，学员方可按权限进入；「已停止对学员展示」后学员不可再进入。**训战活动** 具有筹备中、未开始、进行中、已结束、已下架等状态；已下架后学员不可再进入该活动。
- **训练侧**：同一 **训练场景** 可多次训练，每次产生一条新的 **训战记录**。
- **分析侧**：训练数据分析中的通过率、均分等口径由业务规则定义（例如按 **训练场景**、按人员取历次最高分是否达标），与单条 **训战记录** 的得分并存、不同口径不可混读。

## 5. 业务规则
​
### 5.1 显性规则
​
- **训练场景** 必须处于 **已对学员开放** 的状态，学员方可进入该 **训练场景** 训练；未对学员开放或已作废等状态不对学员开放。
- **训战活动** 须有明确 **起止时间**；**训战活动** **下架** 后不再对学员展示。
- **产品知识** 条目在必填项齐全时才允许保存（品种、分类、标题、正文等，以当前制度为准）。
- 授权时，**非企业管理员** 类角色须指定 **品种管理范围**。
​
### 5.2 隐性规则
​
- **练习** 与 **训战** 在交互过程上相似，业务用途不同：**练习** 偏日常反复试错；**训战** 偏活动或考核语境下的正式完成；统计与筛选时通常按类型区分。
- **虚拟医生** 与 **教练能力** 在环节上分工：访前、对话中、复盘使用不同侧重的信息，避免过程提示与事后评价口径混为一谈（以系统当时策略为准）。
- 学员端是否展示某品种、某 **训练场景**，以及是否开放 **练习** 入口与 **训战** 入口，由 **配置与训战活动** 决定；未开放的 **训练场景** 不可发起训练。
​
### 5.3 权限或范围限制
​
- **平台超级管理员 / 企业管理员**：本企业内 **训练场景**、**训战活动**、基础数据、授权等管理权限（具体以企业隔离策略为准）。
- **品种管理员**：在授权品种范围内管理 **训练场景**、**训战活动** 及相关配置。
- **知识库专员**：以维护 **产品知识** 为主，一般不承担全部 **训练场景** 与 **训战活动** 管理。
- **普通学员**：在可见范围内发起训练与查看个人复盘；全量 **训战记录**、导出、训练数据分析通常面向培训与管理人员。
​
### 5.4 特殊说明或例外
​
- **调试记录** 与 **按人员维度的训战记录**：前者按 **训练场景** 聚合，便于内容与运营侧观察试跑；后者按人员管理，便于导出与考核视角；服务对象不同，不应混为一种报表口径。
- 学员端可能存在 **运营活动**（积分、排行、游戏、奖品等）与 **训战活动** 并存：口语中都可称「活动」时，以 **是否绑定多个训练场景并承担培训组织目标** 区分 **训战活动** 与 **营销或激励向运营活动**。
- **训战管理后台** 中的对话训练能力另含 **内容侧的调参与试运行**，与 **学员端** 标准训练路径的业务范围不同，不互相等同。

## 6. 常见歧义与边界
​
### 6.1 容易混淆的概念
​
- **训练场景** 与 **训战活动**：**训练场景** 是单次拜访情境的内容单元；**训战活动** 是多个 **训练场景** 在一段时间内的打包与运营容器。
- **练习** 与 **训战**：同属对话训练，业务标签不同；报表与列表筛选中须严格区分。
- **医生人设** 与 **训练场景**：**医生人设** 是对话对手设定；**训练场景** 是拜访情境与任务；同一 **训练场景** 可换不同 **医生人设** 以增加变式。
- **产品知识** 与 **训练场景** 中代表可见的叙事与任务描述：前者是可复用的专业条目；后者是这一次拜访的故事与任务说明。
- 两种「活动」：**训战管理后台** 编排的 **训战活动**，与学员端 **积分、游戏、奖品** 等所依附的 **运营活动**，名称相近但目的不同；讨论需求时须先对齐指哪一种。
​
### 6.2 容易误解的规则
​
- **得分高不等于合规无风险**：**训战记录** 中可有 **合规是否通过** 等维度，须与总分一并理解，不能只看分数。
- **看板通过率不等于单次得分**：通过率常按「同一人员在同一 **训练场景** 上历次最高分是否达标」等规则计算，与单条 **训战记录** 的即时分数不是同一口径。
- **系统可参考话术不等于可直接用于真实拜访**：学员可使用系统推荐的参考话术完成本轮训练，真实拜访仍须真人判断与合规约束。
​
### 6.3 理解边界
​
- 本说明描述 **稳定业务语义**；具体阈值（如合格分 60、高分 80 等）以运营配置为准，变更时以发布说明为准。
- **学员端** 与 **训战管理后台** 可能分阶段演进；部分能力（如榜单、部分 **运营活动** 玩法）在特定阶段可能未完全开放，以当前环境可见范围为准。
- **数据保留期限**、**导出文件包含哪些内容**，由企业与平台的 **运维与安全制度** 另行约定，本说明不作承诺。
- **训练数据分析看板** 的访问范围与账号要求，由 **运维与安全制度** 另行约定，本说明不作承诺。
- **数据留存与导出字段** 若在制度中有正式定义，应在后续版本 **业务说明文档** 中补充，并与对外口径对齐。
- **练习 / 训战** 与绩效或活动奖惩的挂钩方式，若有正式制度，应在后续版本中补充。
- 学员端 **运营活动** 与 **训战活动** 在数据上的对应关系，若需对外统一说明，应在后续版本中补充。
- 首页 **有效训战次数**、**超越同事百分比** 等指标的业务定义，应与数据产品口径对齐后在后续版本中补充。

​

| 训练场景 | 本系统核心内容对象之一：一次拜访情境的业务单元，可发布、复制、可对学员停止开放；全文不另用简称「场景」指代其他含义。 |

| 医生人设 | 与 **模拟对象** 同指：驱动 **虚拟医生** 行为与难度的设定。 |
| 虚拟医生 | 训练中与代表对话的模拟医生角色，由 **医生人设** 驱动。 |
| 教练能力 | 访前提示、对话中反馈、事后复盘等陪练侧能力的总称；**不作为独立业务台账对象**，与 **虚拟医生** 分工不同。 |

| 学员端 | 面向学员训练与复盘的使用形态，可嵌入企业 App。 |

| 合规通过 | **训战记录** 中与合规要求相关的通过标识，须与总分区分理解。 |
| 运营活动 | 学员端积分、排行、游戏、奖品等所依附的活动形态；与 **训战活动** 区分使用。 |
