## 第1章｜BP的定义与说明（含G-R-A三层结构）

### 1.1 BP的基本定义（统一口径）

BP（Business Plan）在本体系中不是“商业计划书”，而是一套**责任导向的管理载体**，用于把“想做成的事”结构化为：

- **目标（Goal）**：期望达成的结果状态（What）
- **关键成果（Result）**：判断目标是否达成的依据（So what / How to verify）
- **关键举措（Action）**：实现成果的关键行动路径（How）

BP必须同时回答四类问题，缺一不可：

- **要做成什么**（目标）
- **做成什么样算达成**（成果 + 衡量标准）
- **靠哪些关键行动做成**（举措）
- **谁承担什么责任**（责任模型与角色字段；后续章节展开）

### 1.2 为什么必须采用三层结构（不是形式选择）

三层结构的存在，是为了避免三类长期、结构性偏差：

1. **只有目标，没有落地路径** → 目标空转、无法承接
2. **只有行动，没有结果判断** → 忙而无效、无法评价
3. **行动与目标直接绑定、混写** → 责任边界混乱、协作成本失控

三层结构的核心作用是强制分离三件事：

> **想达成什么**、**如何判断达成**、**通过哪些关键行动达成**。

### 1.3 三层结构的严格边界（硬规则）

以下边界属于“制度硬规则”，不允许跨层书写；AI在生成/审查/纠偏BP时必须以此为准。

#### 1.3.1 目标（Goal）边界

- **目标只写“结果状态”，且严格禁动词**：描述达成后世界是什么样，不写“推动/建立/提升/加强/推进/完成/落地/实施”等动作性表述。
- **目标推荐使用“静态事实定格句式”**：仅允许表达为「**[业务对象]已处于[理想状态]**」的静态描述（用于强制区分“手段/任务”与“目标/终态”）。
- **目标不承担衡量细节**：衡量细节放到成果的衡量标准里，而不是塞进目标句子里堆数字。

常见违规（示例）：
- “推动XX落地”“建立XX机制”“加强XX管理”“持续优化XX”
- “完成XX建设”“实施XX方案”“推进XX上线”

合规写法（示例方向）：
- “XX机制在全范围内按标准运行并可被审计”
- “XX交付能力达到约定SLA并稳定运行”

#### 1.3.2 关键成果（Result）边界

- **成果只用于判断目标是否达成**：成果是“可验收的达成状态/阶段交付状态”，不是动作清单。
- **成果必须符合SMART**：至少满足“具体、可衡量、有时限”，并能说明与目标的支撑关系。
- **成果之间尽量满足MECE**：相互独立、共同穷尽，覆盖目标的关键维度，避免“重复计功/口径打架”。
- **每条成果必须可被衡量**：必须配置衡量标准（至少包含：指标/阈值/口径/数据源/频率）。
- **成果之间应尽量不重叠**：避免同一指标在多个成果重复出现，导致“重复计功”或“口径打架”。

常见违规（示例）：
- “客户满意度明显提升”“业务持续增长”“系统体验大幅优化”（不可验收）

合规写法（示例方向）：
- “NPS达到X（问卷口径，季度统计）”
- “P0/P1问题7日内闭环率≥Y%（工单系统，月度统计）”

#### 1.3.3 关键举措（Action）边界

- **举措只写关键行动**：回答“要做哪些关键动作”，不写结果判断。
- **举措必须可指派、可承接、可追踪**：能落到明确承接人/部门与时间节奏；空泛动作必须拆解。
- **举措必须挂在对应成果下，不允许写在目标层**：目标层只放“终态”，举措必须与某条成果形成清晰支撑关系。

常见违规（示例）：
- “达成收入5亿”“实现市场份额15%”（把结果写成举措）
- “积极配合”“持续跟进”“加强沟通”（不可追踪）

合规写法（示例方向）：
- “建立月度经营分析会机制并形成复盘纪要（固定模板、固定节奏）”
- “完成XX方案评审→试点→推广三阶段交付并留痕归档”

### 1.4 三层数量关系与书写复杂度控制（建议口径）

为保证可管理、可复盘，建议采用以下结构约束（可按组织实际微调，但不建议突破太多）：

- **数量关系**：1个目标 → N个成果 → 每个成果M个举措
- **建议上限**：
  - 每个目标下关键成果：建议 2–5 条（优先2–3条；超过3条需证明必要性）
  - 每条成果下关键举措：建议 2–6 条（优先2–3条；超过3条需证明必要性）

若超过建议上限，AI应触发“合并/拆分/聚焦”提示：不是不允许多，而是必须证明“多的必要性”，否则会显著降低承接与管理质量。

### 1.5 衡量标准（Metric）最小完备要求（成果的必填项）

衡量标准用于把“是否达成”从感觉判断变为可核验事实。对每条成果，衡量标准至少应包含：

- **指标（Metric）**：衡量什么
- **阈值（Target）**：达到什么数值/等级算达成
- **口径（Definition）**：计算方式/范围/排除项（避免数据口径争议）
- **数据源（Source）**：从哪个系统/流程取数
- **频率（Frequency）**：统计与更新频率（月/季/里程碑等）

AI检查规则（用于审查/纠偏）：
- 若成果出现“明显/大幅/持续/优化/提升”等模糊词，必须追问并补齐上述要素。
- 若无数据源与口径，视为“不可验收成果”，不得进入可评价状态。

### 1.6 为什么要做BP（背景与理念）

在长期管理实践中，组织普遍会遇到一些反复出现的问题，这些问题往往不是“员工不努力”，而是组织缺少稳定、可传递、可追溯的管理载体：

- **战略目标存在，但无法持续稳定向下传递**：不同层级对同一目标的理解不断漂移，落到执行层就变成“各做各的”。
- **目标管理、绩效管理、项目管理彼此混用**：同一件事在不同场景被用不同语言描述，导致角色与责任边界模糊。
- **管理者两难**：既担心“目标没人扛”，又担心“责任被无限放大”，最终用更强的过程控制来替代责任机制。
- **员工不清楚自己承担的责任类型**：到底是“执行责任”还是“结果责任”，常常说不清，进而影响承诺与协作质量。

这些问题的本质在于组织缺少一套能被统一回答的问题集：

> **我们想做成什么（目标）？如何判断是否做成（成果与衡量标准）？通过哪些关键行动做成（举措）？谁分别承担何种责任（责任模型与角色）？**

BP（Business Plan）体系的引入，不是一次流程升级，而是为了解决一个基础问题：

> **如何把“想做成的事”，变成“有人兜底、有人推进、有人执行、可被评价”的事。**

### 1.7 BP与KPI/OKR/项目管理的区别与定位

BP并非要取代KPI或OKR，而是作为“责任导向”的统一载体，把目标、结果判断、关键行动与责任边界绑定到同一套结构里。

- **与KPI的区别**  
  - KPI强调“指标是否达标”，容易只剩数字追踪。  
  - BP强调“责任与行动承接”，必须回答“谁来兜底、谁来落实、如何验证”。

- **与OKR的区别**  
  - OKR强调“目标与关键结果对齐”，但不强制把“谁来兜底、如何承接、如何避免责任混用”制度化。  
  - BP在目标与结果对齐基础上，进一步强化**责任分层**与**承接机制**（尤其是组织BP与个人BP的责任性质区分）。

- **与项目管理的区别**  
  - 项目管理关注范围、进度、资源与交付物，适合做“怎么把一件事做完”。  
  - BP用于管理“为什么做、要达成什么状态、如何判断达成、关键动作路径、责任链条”，并可把项目作为关键举措或举措集合纳入承接。

## 第2章｜康哲实施规则与行为约束（创建、承接、责任与质量红线）

> 本章用于回答“康哲怎么用BP、怎么确保不被用歪”。AI在创建/审查/纠偏BP时，必须以本章为默认规则集。

### 2.1 组织BP与个人BP的使用边界（先把“能不能拆、能不能指派”说清楚）

基于现阶段共识：

- **组织BP**：可拆解、可协作，用于战略实施的向下拆解与横向协同拆解。  
  - **目标与成果**：由组织负责人对结果负责（组织维度的责任承诺）。  
  - **举措**：允许指派给下级部门或个人承接，用于形成承接链条。

- **个人BP**：不可向外拆解或指派，完全由个人负责。  
  - 个人BP的目标/成果/举措均应体现“本人承接的责任与贡献”，不得写“由他人负责完成”。  
  - **个人BP必须承接组织BP**：不承接上级组织BP（或上级指派举措）的个人BP属于“无源之水”，不纳入集团/部门战略落地体系。

### 2.2 BP层级结构与人员配置规则（责任人与承接人的含义）

BP采用三层结构：**目标 → 关键成果（含衡量标准） → 关键举措**。

- **目标层**：战略方向与期望达成的结果状态；配置**责任人**（通常为该组织负责人）。
- **关键成果层**：用于判断目标是否达成；配置**责任人**（可为组织负责人或其指定的事务负责人/助手）；每条成果必须配置衡量标准。
- **关键举措层**：实现关键成果的行动计划；配置**承接人/承接部门**（承接方需要再创建自己的目标来对齐承接上级关键举措）。

行为约束（必须执行）：

- **衡量标准无编码、不是独立层级**：衡量标准是关键成果的配置属性，用于量化评估；不得把衡量标准当成“带编号的小节点”去录入，否则系统会误拆为举措，造成结构混乱。
- **关键举措必须具备可承接性**：举措颗粒度需要能让下级“转成目标”，否则承接链条断裂。

### 2.3 各层级创建深度与拆解要求（康哲推荐口径）

康哲当前按组织层级对拆解深度做了明确建议，用于统一训练与运营检查：

- **集团层级**：拆解到 **2层**（目标 → 关键成果），不拆关键举措  
  - 目标：集团战略目标  
  - 关键成果：对目标的可衡量拆解，并为每条关键成果配置衡量标准

- **中心层级**：拆解到 **3层**（目标 → 关键成果 → 关键举措）  
  - 中心目标应能对应承接集团关键成果  
  - 中心关键举措必须可供一级部门承接

- **一级部门层级**：拆解到 **3层**（目标 → 关键成果 → 关键举措）  
  - 一级部门目标应能对应承接中心关键举措  
  - 一级部门关键举措必须可供核心员工承接

- **核心员工层级**：拆解到 **3层**（目标 → 关键成果 → 关键举措）  
  - 个人目标应能对应承接一级部门关键举措  
  - 个人关键举措必须具体到可执行动作（可按周/日节奏落地）

### 2.4 承接规则（红线 + 推荐路径）

本章采用“强约束口径”，用于统一承接逻辑并防止链路断裂。

#### 2.4.1 承接规则总句（用于AI审查）

> **承接规则：下级用“目标”承接上级的“关键成果/关键举措”。**

系统内承接以“对齐/承接关系”体现。承接关系分两类（对应创建指导汇报的蓝色虚线）：

- **成果 → 目标**：下级用“目标”对齐承接上级“关键成果”
- **举措 → 目标（责任人=承接人）**：下级用“目标”对齐承接上级“关键举措”，且下级目标的责任人通常应为该举措的承接人（保证责权一致）

#### 2.4.2 组织层级承接链路（推荐路径）

- **集团层仅拆解到关键成果（2层）**：集团BP结构为「目标 → 关键成果」，不设关键举措
- **中心/一级部门/核心员工拆解到关键举措（3层）**：均采用「目标 → 关键成果 → 关键举措」，确保能够向下传递“可承接”的行动
- **唯一特例（中心承接集团）**：由于集团无关键举措，中心层的“目标”需直接承接集团的“关键成果”（即“成果 → 目标”）

典型链路示例：

- 集团关键成果 → 中心目标（成果→目标）
- 中心关键举措 → 一级部门目标（举措→目标）
- 一级部门关键举措 → 核心员工目标（举措→目标）

#### 2.4.3 人员对应（责任人/承接人一致性）

为保证“责权一致、承接可落地”，下级“目标”的责任人应匹配上级被承接对象的人员配置：

- 承接上级**关键举措**时：**下级目标责任人 = 上级关键举措承接人**
- 承接上级**关键成果**时：下级目标责任人应与该成果在下级的实际负责方一致，并与该成果的责任体系保持一致

### 2.5 组织BP举措指派与多承接规则（协作约束）

依据当前共识：

- **举措可以指派给多个承接部门**，允许跨组织选择部门。
- 也可以直接指派给个人承接。
- **多部门承接时**：所有被选部门均为承接方，系统层面不刻意区分“主责/协同”。因此在写作与运营上必须额外约束：
  - 同一举措被多人/多部门承接时，必须在举措描述或成果验收口径中明确“各承接方交付物边界”，避免重复与扯皮。
- **虚拟组织**：系统暂不支持（如管委会、CVT工作组），相关职责需由具体部门或个人承接并在BP中体现。

### 2.6 创建质量检查清单（康哲必检项）

用于AI自检、运营抽检与培训对齐的“必检项”：

- **结构完整性**
  - [ ] 是否按“目标→关键成果→关键举措”拆解，且拆解深度符合所在层级要求（集团2层，其余3层）
  - [ ] 是否把举措写在目标层（禁止）
  - [ ] 目标下是否有关键成果支撑；关键成果下是否有关键举措落地（集团层除外）
- **衡量标准**
  - [ ] 所有关键成果是否都配置了衡量标准
  - [ ] 衡量标准是否包含“指标+阈值+口径/数据源+统计频率”
  - [ ] 是否把衡量标准错误编码/拆成了多个举措（禁止）
- **承接关系**
  - [ ] 承接关系是否正确：下级目标能清晰对应承接上级关键成果/关键举措
  - [ ] 关键举措是否具备可承接性（下级能转成目标）
- **举措颗粒度**
  - [ ] 举措是否具体到“可执行动作”，能明确责任与跟踪节奏
  - [ ] 举措是否足以让下级直接转写为“静态事实定格”的目标终态（可承接性）
- **汇报周期与时间边界（系统运行约束）**
  - [ ] 汇报周期是否满足“自下而上汇报”的基本约束：**举措汇报周期 ≤ 关键成果汇报周期 ≤ 目标汇报周期**
  - [ ] 子任务时间是否在父任务时间范围内（举措/成果起止时间不应超出其所属成果/目标的起止范围）
- **必填项（最小可运行集）**
  - [ ] 责任人/承接人是否填写（与层级角色一致）
  - [ ] 起止时间是否填写
  - [ ] 汇报周期是否设置（集团层如不启用举措，也应保证目标/成果可汇报）

## 第3章｜BP系统设计、功能模块与使用流程（系统侧）

> 本章以“**从0到1用好BP系统的一整条链路**”为主线，说明：系统管理员如何配置周期与组织树、业务用户如何借助 AI 完成 BP 创建与检查、以及后续围绕举措与汇报的日常使用。制度与行为约束仍以第1-2章为准。

### 3.1 从“空系统”到“可用BP”的整体流程

在康哲的实践中，一套 BP 从“什么都没有”到“可以跑起来”，系统侧大致经历以下步骤：

1. **系统管理员配置周期**  
   - 在后台的“周期管理”中创建年度/季度等周期，并设置“当前周期”。  
   - 所有 BP 数据都挂在某个周期下；不同周期的数据相互隔离，便于年度/季度切换与历史回溯。

2. **在周期下配置组织架构分组**  
   - 在“组织架构树”中创建分组：可继续创建**组织节点**或**个人节点**。  
   - 每个分组节点都会自动生成一段**编码**（见 3.3），用于层级识别与战略溯源。

3. **为分组设置管理员与成员权限**  
   - 在分组的“权限设置”中指定**目录管理员**，负责该分组结构与 BP 的日常管理。  
   - 为成员授予“只读/新增”等权限，控制谁可以在该分组下**新增/编辑 BP**，谁只能查看。

4. **在分组下通过“BP智能创建”建立BP**  
   - 进入某个分组节点的 BP 管理页，点击 **BP 创建 / AI 智能对话**。  
   - 用户既可以**手动填写 BP 表单**，也可以**通过对话让 AI 生成 BP 草稿，并自动写入表单**，用户确认后保存。  
   - 这一能力背后对应两类提示词：  
     - 组织 BP 创建提示词：`组织BP创建完整提示词.md`  
     - 个人 BP 创建提示词：`个人BP创建完整提示词.md`

5. **用“BP智能检查”对内容做结构与承接质检**  
   - BP 草稿完成后，用户可在节点下调用“AI 智能检查”，对当前组织或个人 BP 进行质量审查：  
     - 检查 G-R-A 结构是否完整  
     - 检查衡量标准是否符合 SMART  
     - 检查与上级 BP 的承接是否正确、是否有漏承接  
   - 背后对应两类检查提示词：`组织检查.md`、`个人检查.md`。

6. **启动并运行 BP：围绕举措管理工作与汇报**  
   - 在 BP 内容齐全、通过质检后，可进入“运行阶段”：  
     - 在工作协同系统中围绕举措管理具体任务，或  
     - 在 BP 系统中，为每条举措/成果**关联已有汇报**、或发起新汇报计划。  
   - 日常通过“汇报详情”“个人看板”“座舱监控树”等视图跟进执行。

### 3.2 功能模块总览（自上而下的模块视图）

结合前端代码与后端设计，BP 系统的主要功能模块可以归为八类：

1. **周期与组织架构管理模块**：周期管理、分组树管理、权限授权、编码管理  
2. **编码与溯源模块**：分组编码、任务编码，支撑“战略溯源”和“精确引用”  
3. **BP智能创建模块**：BP 创建、AI 智能对话（组织 BP / 个人 BP 提示词）  
4. **BP编辑与结构管理模块**：G-R-A 结构编辑、视图切换、版本与导入  
5. **BP智能检查模块**：组织 BP / 个人 BP 的结构与承接检查（检查提示词）  
6. **承接与对齐管理模块**：对齐关系设置
7. **工作协同与汇报联动模块**：举措与工作汇报系统的关联、汇报详情集中查看  
8. **视图与分析模块**：BP 预览、个人看板、座舱监控树

下面按“模块 → 关键功能”的顺序展开说明。

### 3.3 周期与组织架构管理模块

#### 3.3.1 周期管理（后台）

- **目的**：为 BP 提供时间框架（年度/季度等），并实现数据隔离。  
- **能力**：
  - 创建/编辑/删除周期
  - 设为当前周期（只有一个当前周期）
  - 批量删除（不能删除当前周期，系统会提示）
- **使用建议**：  
  - 每年初由系统管理员统一创建新年度周期，并复制/导入上年结构。  
  - 在创建或切换周期前，需明确“旧周期是否需要继续维护/仅供浏览”。

#### 3.3.2 组织分组树与权限

- **分组创建**：
  - 类型：组织 / 个人  
  - 组织分组：手动输入名称  
  - 个人分组：通过“选人”单选，名称自动带入
- **权限设置**：
  - 目录管理员：负责该分组的 BP 与子节点管理  
  - 成员权限：只读 / 新增（控制是否可在该分组下新增/编辑 BP）
- **行为约束**：
  - 非超级管理员无法直接新增根节点（无上级会校验失败）  
  - 无权限的节点不允许选作上级，也不允许修改 BP

### 3.4 编码与溯源模块（分组编码 + 任务编码）

模块能力来自 `编码功能.md` 和系统实现。

#### 3.4.1 分组编码

- **作用**：唯一标识组织节点，体现层级关系，用于跨部门沟通与战略溯源。  
- **基本规则**：
  - 根节点（公司/集团）：`G`
  - 第二层：`A1/A2/...`
  - 第三层：`A1B1/A1B2/...`
  - 依此类推，字母代表层级，数字记录创建时的排序位置。
- **个人分组编码**：`父分组编码 + P + 工号`（如：`A1B2P10086`）。

#### 3.4.2 目标/成果/举措编码

- **作用**：在一个分组下唯一标识 G-R-A 结构中每个节点的位置序号。  
- **基本规则**：
  - `1`：第1个目标  
  - `1.1`：第1个目标下的第1个关键成果  
  - `1.1.2`：第1个目标 > 第1个关键成果 > 第2个关键举措  
  - 编码记录的是“创建顺序”，后续调序不会改变已有编码。

> 实务建议：在跨层级沟通时，用“分组编码 + 任务编码”精确指代某条 BP，例如：`A1B2 - 1.2.1`。

### 3.5 BP智能创建模块（组织 BP / 个人 BP）

#### 3.5.1 入口与模式

- 入口：在某个分组节点的 BP 管理列表顶部，点击 **BP 创建** 或 **AI 智能对话**。  
- 模式：
  - 纯表单模式：用户手动填写目标/成果/举措等字段  
  - AI 对话模式：用户描述业务背景，AI 按提示词结构生成 BP 初稿，并自动写入表单，用户确认后保存

#### 3.5.2 组织 BP 智能创建（对应组织创建提示词）

- AI 会引导组织负责人/管理者：  
  - 先确认“本组织在当前周期的职责与上级 BP”的承接关系  
  - 再按 G-R-A 结构拆解：目标 → 关键成果（含衡量标准）→ 关键举措  
  - 提醒用户：集团层只拆到成果，中心/一级部门需拆到举措且可供下级承接

#### 3.5.3 个人 BP 智能创建（对应个人创建提示词）

- AI 会先判断用户是**管理岗**还是**员工岗**，再分别引导：  
  - 管理岗：体现“管理动作 + 关键执行”，目标必须承接组织 BP 的关键举措/关键突破事项  
  - 员工岗：聚焦可执行任务与交付物，目标必须承接上级关键举措或岗位职责  
- 在表单层面仍然是 G-R-A 结构，但 AI 会自动避免“个人 BP 再向外指派”的违规行为。

### 3.6 BP编辑与结构管理模块（视图 + 版本 + 导入）

- **视图切换**：列表 / 思维导图 / 表格  
- **结构操作**：创建子目标、编辑目标/成果/举措、设置时间与汇报周期、添加关联汇报  
- **版本与导入**：
  - 版本保存与查看历史  
  - 通过 Excel 模板导入 BP（全部追加 / 智能合并 / 删除重建）

> 删除重建属于高风险操作，系统会以“此操作不可逆，请谨慎操作”提示。

### 3.7 BP智能检查模块（组织检查 / 个人检查）

- **入口**：在某个分组/个人节点下，调用“AI 智能检查”或类似入口。  
- **组织检查**：基于 `组织检查.md` 的规则，系统会让 AI 从以下角度质检：  
  - 是否按层级拆解到正确深度（集团2层，其余3层）  
  - 所有关键成果是否有合格的衡量标准（SMART）  
  - 承接关系是否正确：中心承接集团关键成果，其他层承接上级关键举措  
  - 下级是否已合理承接当前层的举措
- **个人检查**：基于 `个人检查.md` 的规则，AI 会聚焦：  
  - 个人目标是否承接上级举措，而不是“自创无源目标”  
  - 个人目标/成果是否为“状态/结果”而非动作  
  - 举措是否细化到本人可执行动作，是否便于考察贡献。

#### 3.8 承接与对齐管理模块
 
- **前端入口（实际产品行为）**：  
  - 仅在“**目标**”节点上提供“设置对齐关系”入口。  
  - 在弹窗中，目标可以选择要对齐的对象（目标/成果/举措）。
- **康哲使用约束（必须遵守）**：  
  - 只使用「**下级目标对齐上级关键举措**」这一种承接方式，即：**上级关键举措 → 下级目标**。  
  - 不使用“目标对目标”“成果对成果”来表达承接，更不允许用成果对成果代替举措承接。
- **任务关系管理页**：用于集中查看并调整“哪些目标对齐了哪些上级举措/成果”，即上下游承接关系的技术呈现。  
- **后端约束**（简要）：禁止重复对齐、禁止对齐到自身父链或子孙、限制循环依赖深度（避免 A↔B 互相对齐）。
对齐关系是实现“标准承接链路”的技术基础，对应第2章 2.4 的承接规则，产品使用上需同时满足“系统能力 + 康哲约束”两层。


### 3.9 工作协同与汇报联动模块
- **工作协同系统 → 关联 BP**：  
  - 在工作协同/汇报系统中，填写或编辑汇报时，可选择要关联的 BP 任务（通常是某条举措或成果）。  
  - 关联成功后，该汇报会在 BP 系统中对应任务下以“关联汇报”的形式呈现。
- **BP 系统 → 选择/查看汇报**：  
  - 在 BP 列表或详情中，可以看到“已关联到本任务的汇报”（来自工作协同），并可点击查看详情。  
  - 也可以在 BP 系统中，从某条举措/成果出发，在可见汇报中**选择要关联的汇报**，建立反向关联。
- **联动效果**：  
  - 工作协同侧：通过“汇报关联 BP”，知道这次汇报支撑的是哪条 BP 举措/成果。  
  - BP 侧：通过“关联汇报”，可以在“汇报详情”等界面集中看到与某条举措/成果相关的所有汇报记录，用于运行诊断与复盘。

### 3.10 视图与分析模块

- **BP 预览**：一键生成结构化“责任书视图”，用于对齐确认与汇报呈现。  
- **个人看板**：从个人视角汇总“待承接 BP”“我负责的 BP”。   
- **汇报详情**：集中按“组织 + 任务树”维度浏览所有汇报。

### 3.11 后台与工具模块

- **用户/管理员管理**：添加管理员、启用/禁用、查询。  
- **组织隐藏/显示管理**：按周期批量显示/隐藏分组。
