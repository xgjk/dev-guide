## 第2章｜系统概述（BP是什么、为什么要这么做）

### 2.1 BP的定义（业务语义）

BP（Business Plan）在本体系中不是“商业计划书”，而是一套**责任导向的管理载体**，用于把“想做成的事”结构化为：

- **目标（Goal）**：期望达成的结果状态（What）
- **关键成果（Result）**：判断目标是否达成的依据（So what / How to verify）
- **关键举措（Action）**：实现成果的关键行动路径（How）

BP必须同时回答四类问题，缺一不可：

- **要做成什么**（目标）
- **做成什么样算达成**（成果 + 衡量标准）
- **靠哪些关键行动做成**（举措）
- **谁承担什么责任**（责任模型与角色字段；由承接关系体现）

### 2.2 为什么要采用三层结构（G-R-A）

三层结构的存在，是为了避免三类长期、结构性偏差：

1. **只有目标，没有落地路径** → 目标空转、无法承接  
2. **只有行动，没有结果判断** → 忙而无效、无法评价  
3. **行动与目标直接绑定、混写** → 责任边界混乱、协作成本失控  

三层结构强制分离三件事：

> 想达成什么、如何判断达成、通过哪些关键行动达成。

### 2.3 与KPI/OKR/项目管理的区别与定位

BP并非要取代KPI或OKR，而是作为“责任导向”的统一载体，把目标、结果判断、关键行动与责任边界绑定到同一套结构里。

- 与KPI的区别  
  - KPI强调“指标是否达标”，容易只剩数字追踪。  
  - BP强调“责任与行动承接”，必须回答“谁来兜底、谁来落实、如何验证”。  

- 与OKR的区别  
  - OKR强调“目标与关键结果对齐”，但不强制把“谁来兜底、如何承接、如何避免责任混用”制度化。  
  - BP在目标与结果对齐基础上，进一步强化**责任分层**与**承接机制**（尤其是组织BP与个人BP的责任性质区分）。  

- 与项目管理的区别  
  - 项目管理关注范围、进度、资源与交付物，适合做“怎么把一件事做完”。  
  - BP用于管理“为什么做、要达成什么状态、如何判断达成、关键动作路径、责任链条”，并可把项目作为关键举措的补充载体。

## 第3章｜核心业务对象（业务模型实体）

### 3.1 目标（Goal）

目标是期望达成的结果状态承诺。目标用于描述“达成后世界是什么样”，并作为承接链条中的下游对齐对象。

### 3.2 关键成果（Result）

关键成果用于回答“目标是否达成、达成到什么程度”。成果必须可被验收，且每条成果都必须配置衡量标准。

### 3.3 关键举措（Action）

关键举措用于回答“通过哪些关键行动实现成果”。举措必须可指派、可承接、可追踪，并且与某条成果建立清晰支撑关系。

### 3.4 衡量标准（Metric / 衡量口径）

衡量标准用于把“是否达成”从感觉判断变为可核验事实。它是成果的配置属性，不构成额外层级。

衡量标准至少应包含：

- 指标（Metric）：衡量什么
- 阈值（Target）：达到什么数值/等级算达成
- 口径（Definition）：计算方式/范围/排除项等
- 数据源（Source）：取数系统/流程
- 频率（Frequency）：统计与更新频率

### 3.5 承接关系（Alignment / 承接链路）

承接关系用于把“上级的成果/举措”转化为“下级的目标”，形成可传递的责任链条。承接不是简单复制，而是下级把上级要求转译成自身可承接的“目标终态”。

### 3.6 组织BP（Organization BP）

组织BP用于战略实施的向下拆解与横向协同拆解；其特点是允许结构化拆解与承接指派，从而形成组织层级间的承接链条。

### 3.7 个人BP（Individual BP）

个人BP由个人对自身责任承诺；其特点是不可向外拆解或指派，且必须承接上级组织BP体系中的责任源头，否则属于“无源之水”。

## 第4章｜核心业务关系（对象之间怎么连起来）

### 4.1 G-R-A 关系（目标-成果-举措）

G-R-A关系要求：

- 目标用于描述终态结果（不承载动作与衡量口径）
- 关键成果用于判断目标是否达成（并必须绑定衡量标准）
- 关键举措用于实现关键成果（并必须能被下游承接转写为目标终态）

### 4.2 上下级承接关系（承接方向与两类承接方式）

承接规则总句：

> 下级用“目标”承接上级的“关键成果/关键举措”。

承接关系分两类（在业务口径层面对应两种承接来源）：

- **成果 → 目标**：下级用目标对齐承接上级的关键成果
- **举措 → 目标**：下级用目标对齐承接上级的关键举措，并使责权一致（下级目标责任人通常应与举措承接方保持一致）

典型链路示意（以组织层级常见拆解深度为前提）：

- 集团关键成果 → 中心目标（成果→目标）
- 中心关键举措 → 一级部门目标（举措→目标）
- 一级部门关键举措 → 核心员工目标（举措→目标）

### 4.3 责权一致性（责任人与承接人的对齐）

为保证“责权一致、承接可落地”，下级目标的责任人应与被承接对象的责任体系保持一致：

- 承接上级**关键举措**时：下级目标责任人应匹配该举措的承接人
- 承接上级**关键成果**时：下级目标责任人应与该成果在下级的实际负责方一致，并保持责任体系连贯

## 第5章｜业务规则（通用规则 + 康哲实施规则）

### 5.1 通用 BP 规则（制度层面的写法边界）

#### 5.1.1 目标边界（Goal）

- 目标只写“结果状态”，且严格禁动词：描述达成后世界是什么样，不写“推动/建立/提升/加强/推进/完成/落地/实施”等动作性表述。
- 目标推荐使用“静态事实定格句式”：仅允许表达为「[业务对象]已处于[理想状态]」的静态描述。
- 目标不承担衡量细节；衡量细节放到成果的衡量标准里，而不是塞进目标句子里堆数字。

#### 5.1.2 关键成果边界（Result）

- 成果只用于判断目标是否达成：成果不是动作清单。
- 成果必须符合SMART：至少满足“具体、可衡量、有时限”，并能说明与目标的支撑关系。
- 成果之间尽量满足MECE：相互独立、共同穷尽，覆盖目标关键维度，避免重复计功与口径打架。
- 每条成果必须可被衡量：必须配置衡量标准（至少包含：指标/阈值/口径/数据源/频率）。
- 成果之间应尽量不重叠：避免同一指标在多个成果重复出现。

#### 5.1.3 关键举措边界（Action）

- 举措只写关键行动：回答“要做哪些关键动作”，不写结果判断。
- 举措必须可指派、可承接、可追踪：能落到明确承接人/部门与时间节奏；空泛动作必须拆解。
- 举措必须挂在对应成果下：不允许把举措写在目标层。
- 举措应能让下级直接转写为“目标终态”：保证承接可承接性。

#### 5.1.4 三层数量关系与复杂度控制（建议口径）

为保证可管理、可复盘，建议采用：

- 数量关系：1个目标 → N个成果 → 每个成果M个举措
- 建议上限（建议值用于控制质量风险）：
  - 每个目标下关键成果：建议 2–5 条（优先2–3条；超过3条需证明必要性）
  - 每条成果下关键举措：建议 2–6 条（优先2–3条；超过3条需证明必要性）

若超过建议上限，AI应触发“合并/拆分/聚焦”提示：不是不允许多，而是必须证明多的必要性，否则会显著降低承接与管理质量。

### 5.2 康哲实施规则（组织层级与责任/承接约束）

#### 5.2.1 组织BP与个人BP的使用边界

基于现阶段共识：

- **组织BP**：可拆解、可协作，用于战略实施的向下拆解与横向协同拆解。  
  - 目标与成果：由组织负责人对结果负责  
  - 举措：允许指派给下级部门或个人承接，形成承接链条  

- **个人BP**：不可向外拆解或指派，完全由个人负责。  
  - 个人BP的目标/成果/举措均应体现“本人承接的责任与贡献”，不得写“由他人负责完成”。  
  - 个人BP必须承接组织BP；不承接上级组织BP（或上级指派举措）的个人BP属于“无源之水”，不纳入集团/部门战略落地体系。  

#### 5.2.2 BP层级结构与人员配置规则

BP采用三层结构：目标 → 关键成果（含衡量标准） → 关键举措。

- 目标层：战略方向与期望达成的结果状态；配置责任人（通常为该组织负责人）
- 关键成果层：用于判断目标是否达成；配置责任人；每条成果必须配置衡量标准
- 关键举措层：实现关键成果的行动计划；配置承接人/承接部门（承接方需要再创建自己的目标来对齐承接上级关键举措）

行为约束（必须执行）：

- 衡量标准无编码、不是独立层级；不得把衡量标准当成“带编号的小节点”去录入，否则会造成结构混乱。
- 关键举措必须具备可承接性：颗粒度需要能让下级“转成目标”，否则承接链条断裂。

#### 5.2.3 各层级创建深度与拆解要求（康哲推荐口径）

- 集团层级：拆解到 2层（目标 → 关键成果），不拆关键举措
- 中心层级：拆解到 3层（目标 → 关键成果 → 关键举措）
- 一级部门层级：拆解到 3层（目标 → 关键成果 → 关键举措）
- 核心员工层级：拆解到 3层（目标 → 关键成果 → 关键举措）

#### 5.2.4 承接规则（红线 + 推荐路径）

承接规则总句：

> 下级用“目标”承接上级的“关键成果/关键举措”。  

承接链路建议（配合“拆解深度”口径）：

- 集团层仅拆解到关键成果（2层），中心/一级部门/核心员工拆解到关键举措（3层）
- 唯一特例：由于集团无关键举措，中心层“目标”需直接承接集团的关键成果（成果→目标）

人员对应（责任人/承接人一致性）：

- 承接上级关键举措时：下级目标责任人 = 上级关键举措承接人
- 承接上级关键成果时：下级目标责任人应与该成果责任体系保持一致

#### 5.2.5 举措指派与多承接规则

- 举措可以指派给多个承接部门，也可以直接指派给个人承接。
- 多部门承接时：系统侧不刻意区分主责/协同；因此写作与运营上必须额外约束各承接方交付物边界，避免重复与扯皮。

#### 5.2.6 虚拟组织处理口径

如存在管委会、CVT工作组等“跨组织工作主体”，则相关职责需落到具体部门或个人承接方，并在BP中体现承接链条。

### 5.3 质量红线与业务质检要点（用于审查/纠偏）

以下要点用于AI自检、运营抽检与培训对齐（本质是业务质检标准，而非系统操作入口）：

- 结构完整性
  - 是否按“目标→关键成果→关键举措”拆解，且拆解深度符合所在层级要求
  - 是否把举措写在目标层（禁止）
  - 目标下是否有关键成果支撑；关键成果下是否有关键举措落地（集团层除外）

- 衡量标准
  - 所有关键成果是否都配置了衡量标准
  - 衡量标准是否包含“指标+阈值+口径/数据源+统计频率”
  - 是否把衡量标准错误拆成多个节点/举措（禁止）

- 承接关系
  - 下级目标能否清晰对应承接上级关键成果/关键举措
  - 承接的举措颗粒度是否具备可承接性（下级能转成目标终态）

- 举措颗粒度
  - 举措是否具体到可执行动作，能明确责任与跟踪节奏
  - 举措是否足以让下级转写为“静态事实定格”的目标终态

- 时间与汇报边界（业务约束）
  - 汇报周期是否满足“自下而上汇报”的基本约束：举措 ≤ 成果 ≤ 目标
  - 子任务时间是否在父任务时间范围内：举措/成果起止不应超出其所属成果/目标起止范围

- 必填项（最小可运行集）
  - 责任人/承接人是否填写（与层级角色一致）
  - 起止时间是否填写
  - 汇报周期是否设置（集团层如不启用举措，也应保证目标/成果可汇报）

## 第6章｜常见歧义与边界（必须澄清）

### 6.1 BP ≠ 商业计划书

BP在本体系中是责任导向的管理载体，不是“预算/融资/投资计划”的别称，也不是单纯的叙述型文档。

### 6.2 目标 ≠ 动作

目标描述的是结果状态，不能写“推动/建立/提升/加强/推进/完成/落地/实施”等动作性表述；动作必须写在关键举措中。

### 6.3 关键成果 ≠ 关键举措

关键成果用于判断是否达成目标（并配衡量标准）；关键举措用于实现成果（并必须能被承接转写为下游目标终态）。

### 6.4 衡量标准 ≠ 独立层级

衡量标准是成果的配置属性，不是单独的层级节点；不得按“带编号的小节点”方式录入，否则会破坏G-R-A结构。

### 6.5 承接 ≠ 简单复制上级内容

承接的本质是转译：下级把上级的成果/举措要求转化为自己的目标终态，并形成责权一致的承接链路。

### 6.6 个人BP ≠ 可继续向外拆解的组织任务

个人BP不可向外拆解或指派；个人目标/成果/举措应体现个人承接与贡献，不得依赖“他人完成”来替代自身责任。

| 关键成果 | 用于判断目标是否达成的可验收达成状态；必须配置衡量标准，符合SMART |
| 关键举措 | 实现成果的关键行动路径；可指派、可承接、可追踪，并挂在成果下 |
| 衡量标准 | 成果的核验属性；至少包含指标、阈值、口径/数据源、统计频率 |
| 承接 | 上下级之间的责任链传递机制：下级目标对齐承接上级关键成果/关键举措（而非复制） |
| 对齐 | 承接链在业务模型层面的表达方式，体现“下级目标对应的承接来源对象” |
| 组织BP | 可拆解、可协作，用于战略实施的组织层级承接链条（允许指派举措） |
| 个人BP | 由个人对自身责任承诺，不可外拆不可外指；必须承接组织BP责任源头 |
