## Skill 1：BP.ORCH.CreateDraftFromScratch（新建草稿编排：四段内容生成）

### 1. 基本信息
- **Skill 名称**：新建草稿编排（四段内容生成）
- **Skill 编码**：BP.ORCH.CreateDraftFromScratch
- **版本号**：1.0.0
- **所属业务域**：BP编制修订
- **负责人**：待你指定
- **适用 Agent 列表**：BP编制修订助手

### 2. Skill 定位
- **业务目的**：从用户提供的自由文本材料出发，结合时间/周期信息与系统默认模板规则，生成 BP 的四段正文（目标/成果/关键举措/衡量标准），用于写入 BP 系统草稿字段。
- **解决的动作**：选模板 → 抽取材料要点 → 生成四段正文 → 按模板要求组织表达 → 输出用于系统填充
- **类型**：编排类（Orchestration）

### 3. 输入输出契约（自然语言）
**输入（调用方需要提供）**
- **业务上下文（用于默认选模板）**：业务域、部门、角色  
  - 模板规则：系统按“业务域/部门/角色”默认选模板；个人走“个人BP编写skill”，部门走“部门BP编写skill”。
- **时间与周期信息**：起止时间、汇报周期、通报/通知周期日（例如每月第几日）、通报/通知类型（通报或通知）
- **用户材料**：自由文本（用户输入，要生成内容的依据）；可选附件文本（如果有）
- **输出绑定信息**：明确本次输出用于“BP系统草稿内容”的四段字段填充
- **requestId**：用于追踪的一次性请求标识（如有）

**必填/选填**
- **必填**：业务域/部门/角色、起止时间、汇报周期、通报/通知周期日、通报/通知类型、用户自由文本、写入目标（BP系统草稿）、requestId  
- **选填**：附件文本、输入元信息（如语言/输入来源类型）

**校验规则**
- 起始日期不得晚于结束日期
- 通报/通知周期日必须在 1–31
- 汇报周期必须是系统支持的枚举值（如：每天/每周/每月/每季度/每年/自定义）
- 用户自由文本去空后不能为空

---

**输出（Skill 返回给系统，用于落库填充）**
- 四段**纯正文文本**（不附带口径说明/来源字段/证据）：
  - 目标（targetText）
  - 成果（resultText）
  - 关键举措（initiativeText）
  - 衡量标准（metricText）
- 本次命中的模板信息（如 templateId、templateVersion，若系统有）
- requestId 原样回传（如有）

**失败返回**
- 按“系统统一失败返回格式”返回（你当前未提供固定字段名，因此此处不写死 key；仅约定失败语义与错误类型，见第 9 点）。

**错误码结构（占位建议，落地时以系统真实 code 为准）**
- 建议前缀：BP_ERR_*
- 建议语义：
  - BP_ERR_BAD_REQUEST（参数错误）
  - BP_ERR_TEMPLATE_NOT_FOUND（模板未命中）
  - BP_ERR_INPUT_EMPTY（输入为空或无法抽取）
  - BP_ERR_GENERATION_FAILED（生成失败）
  - BP_ERR_PERMISSION_DENIED（权限不足）
  - BP_ERR_DEPENDENCY_TIMEOUT（依赖超时）
  - BP_ERR_UNSUPPORTED_CAPABILITY（越界能力不支持）

### 4. 前置条件
- 调用方/当前用户具备“创建/保存草稿”的写入权限
- 业务域/部门/角色信息可用于默认选模板
- 用户输入材料至少包含可用于生成四段正文的部分信息（允许不完整，但不能完全空）

### 5. 执行逻辑说明
1. **模板解析**：根据业务域/部门/角色，走系统默认模板选择（个人/部门分支）
2. **输入抽取**：从自由文本（+可选附件文本）抽取与四段正文相关的要点  
   - 约束：不虚构业务事实/执行结果
3. **生成四段纯文本**：分别生成目标/成果/关键举措/衡量标准的正文
4. **按模板整理**：按模板的段落组织与文体要求整理四段表达  
   - 约束：不做数字对齐、不做口径算法定义
5. **返回结果**：输出四段纯文本，供 BP 系统写入草稿字段

- 多 API：是（至少模板获取/解析、抽取与组装）
- 数据处理：包含清洗、抽取、聚合与段落组织
- 规则判断：用于“缺失关键信息判断”“按模板/角色约束组织”
- 缓存：允许模板缓存（可选）

### 6. 所依赖的业务 API 清单（抽象名，落地替换为真实接口）
1. **BP.Template.resolveByDomainDeptRole**
   - 用途：按业务域/部门/角色选择模板
   - 顺序：第 1 步
   - 超时要求：如 3s（待系统标准）
   - 降级策略：模板找不到则失败（不返回半成品）
2. **BP.UserInput.extractForBP**
   - 用途：从自由文本抽取四段要点（内部可用规则/LLM）
   - 顺序：第 2 步
   - 降级策略：抽取为空则失败
3. **BP.DraftText.applyTemplateAndAssemble**
   - 用途：按模板规则组装四段纯文本
   - 顺序：第 3-4 步
4. （可选）**BP.DraftWriter.validateWritePermission**
   - 用途：写权限校验
   - 顺序：写入前置（如系统由编排层负责）

### 7. 结果可信度与解释
- 不附证据
- 不返回来源字段
- 不返回口径说明
- 不提供追溯原始数据能力（建议由系统侧存档原始输入）

### 8. 安全与权限
- **读/写属性**：
  - 读：模板（必要）
  - 写：由 BP 系统负责落库；本 Skill 只输出用于填充的文本
- 脱敏：不额外做脱敏（按系统输入合规处理）
- 审计日志：建议记录 operatorId/requestId/traceId（字段名以系统为准）
- 二次确认/审批令牌：按系统统一权限体系（此处不强加）

### 9. 异常处理
- 参数错误：按系统正常失败返回（语义 BAD_REQUEST）
- 输入为空/不可抽取：INPUT_EMPTY
- 模板未命中：TEMPLATE_NOT_FOUND
- 依赖超时：DEPENDENCY_TIMEOUT
- 权限不足：PERMISSION_DENIED
- 越界能力请求（审计结论/数字对齐/进展分析/评价结论/虚构事实）：**直接不提供能力并失败返回**（UNSUPPORTED_CAPABILITY）
- 部分成功：不支持（四段作为整体交付；缺关键生成条件则整体失败）

### 10. 测试与验收
- 单元测试
  - startDate > endDate：失败
  - noticeDayOfMonth=0/32：失败
  - 用户文本为空：失败
- 业务场景测试
  - 个人角色：能命中个人模板分支并生成四段
  - 部门角色：能命中部门模板分支并生成四段
- 异常场景
  - 模板不存在：失败
  - 依赖超时：失败
- 回归要求
  - 四段始终输出纯文本
  - 不生成口径说明/审计结论等越界内容

## Skill 2：BP.ORCH.SupplementExistingBP（基于旧 BP 补充完善编排：四段增强）

> 注：原文中该 Skill 的部分输入字段与策略仍为“默认假设版”，需要你后续确认 oldBpId/版本/补充策略是否固定。本规范**忠于原文保留“待确认/默认假设”**。

### 1. 基本信息
- **Skill 名称**：基于旧 BP 补充完善编排（四段增强）
- **Skill 编码**：BP.ORCH.SupplementExistingBP
- **版本号**：1.0.0
- **所属业务域**：BP编制修订
- **负责人**：待你指定
- **适用 Agent 列表**：BP编制修订助手

### 2. Skill 定位
- **业务目的**：在已有 BP 四段正文的基础上，结合用户补充材料，识别缺口并补齐/增强四段内容（目标/成果/关键举措/衡量标准），输出用于 BP 系统字段填充。
- **类型**：编排类（Orchestration）

### 3. 输入输出契约（自然语言，默认假设版）
**输入（调用方需要提供）**
- **业务上下文**：业务域、部门、角色（用于默认选模板）
- **旧 BP 识别信息（待确认字段）**：
  - 默认假设：oldBpId（旧 BP 的唯一标识）
  - 可选：oldBpVersion（如系统存在版本号）
- **时间与周期信息**：起止时间、汇报周期、通报/通知周期日、通报/通知类型
- **用户补充材料**：自由文本（可选附件文本）
- **补充策略（默认假设）**：
  - mode 可选：补齐缺口 / 增强已有 / 强制重生成_四段 / 部分补充
  - 若选择“部分补充”：需要指定补充哪些段（目标/成果/关键举措/衡量标准）
- **输出绑定信息**：四段字段填充；写入目标为 BP 系统草稿或修订稿内容
- **requestId**

**必填/选填（默认假设）**
- **必填**：业务域/部门/角色、oldBpId、起止时间/周期信息、用户自由文本、写入目标、requestId、补充策略 mode  
- **选填**：oldBpVersion、附件文本、部分补充的目标段落列表

**校验规则**
- oldBpId 不能为空
- 时间校验同新建草稿
- 若 mode=部分补充：必须明确要补充的段落列表

---

**输出（Skill 返回给系统，用于落库填充）**
- 四段**补充/增强后的纯正文文本**：
  - 目标（targetText）
  - 成果（resultText）
  - 关键举措（initiativeText）
  - 衡量标准（metricText）
- 差异信息（默认假设）：哪些段发生了变化（changedSections）
- 模板信息（如 templateId、templateVersion）
- requestId

**失败返回**
- 仍按系统统一失败返回格式（字段名待补全）。

### 4. 前置条件
- 当前用户具备“编辑草稿/修订稿”的写入权限
- 具备读取旧 BP（oldBpId 对应内容）的读权限
- 用户补充材料能支持补齐缺口（否则可能失败，或按策略决定）

### 5. 执行逻辑说明
1. **模板解析**：按业务域/部门/角色默认选模板
2. **拉取旧 BP 四段正文**：用于复用与对比缺口
3. **缺口识别**：结合旧正文与用户补充材料，判断哪些段需要补齐/增强
4. **生成/改写（受边界约束）**：
   - 仅生成/增强四段正文
   - 不生成审计结论、不做数字对齐、不做进展分析、不做评价结论、不虚构事实
5. **输出结果**：返回四段纯文本供系统写入

### 6. 所依赖的业务 API 清单（抽象名）
1. **BP.Template.resolveByDomainDeptRole**：默认选模板
2. **BP.BPContent.fetchById**
   - 用途：按 oldBpId 获取旧 BP 四段纯文本
3. **BP.BPSupplement.diffAndGapAnalyze**
   - 用途：差异比较与缺口识别
4. **BP.DraftText.applyTemplateAndAssemble**
   - 用途：按模板组织输出四段正文

### 7. 结果可信度与解释
- 不附证据
- 不返回来源字段
- 不返回口径说明
- 不提供追溯能力（由系统侧自行留存原始输入/旧版本）

### 8. 安全与权限
- 读：读取旧 BP 内容（必须）
- 写：由系统落库；本 Skill 输出用于填充
- 审计日志：建议记录 requestId/oldBpId（字段名以系统为准）
- 脱敏：不额外处理（按系统合规）

### 9. 异常处理
- 参数错误：失败（系统正常失败返回）
- oldBpId 不存在或无权限：失败
- 旧 BP 内容缺失：
  - 原文默认：若 mode=补齐缺口，尝试补齐并输出完整四段；若仍无法补齐某段则整体失败（是否允许降级待你确认）
- 越界能力请求：不提供能力并失败返回（建议语义 UNSUPPORTED_CAPABILITY）
- 依赖超时/依赖失败：失败
- 部分成功：原文倾向不支持（除非 mode 明确允许“部分补充”，但仍输出四段结果；具体落地策略待确认）

### 10. 测试与验收
- 单元测试
  - oldBpId 为空：失败
  - mode=部分补充但未指定段落：失败
- 业务场景测试
  - 旧 BP 缺“衡量标准”，用户补充指标描述：补齐并输出完整四段
  - 旧 BP 内容较完整，用户补充少量信息：按策略增强而不是完全重写（取决于 mode）
- 回归要求
  - 始终输出四段纯正文
  - 不产生口径说明/审计结论等越界内容
