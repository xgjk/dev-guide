## 基本信息
- Skill名称：BP完整性与规范检查能力
- Skill编码：BP_AUDIT_01
- 版本号：V1.1
- 所属业务域：BP管理 / 审计
- 负责人：BP审计助手团队
- 适用Agent：BP审计助手
- 创建日期：2026-03-23

## Skill定位
- 业务目的：判断当前 BP 是否具备“能被验收与承接”的基础结构与规范性（结构完整性、衡量标准、周期链路、语义边界与颗粒度）。
- 类型：分析类（只输出问题清单，不做拍板）

## 输入
> 注：该 Skill 面向 Agent 可调用设计，输入建议是“已解析的任务树”，而不是纯文本长段落。

### 1. 必填入参
- `taskTree`：BP 三层结构树（目标 → 关键成果 → 关键举措）
  - `goalNodes`：目标列表（每个目标需含名称、描述、汇报周期、计划时间范围、子关键成果）
  - `keyResultNodes`：关键成果列表（每个成果需含名称、衡量标准、汇报周期、计划时间范围、子关键举措）
  - `actionNodes`：关键举措列表（每个举措需含名称、汇报周期、计划时间范围）
- `currentLevelType`：当前检查层级类型
  - 枚举建议：`employee`（员工层，需要 3 层：目标→成果→举措）
  - `org_group` / `org_dept`（组织层，通常只要求 2 层：目标→成果；举措可能为空）
  - 若调用方无法提供，必须在 `taskTree` 中明确当前层级的“举措是否应存在”的事实（例如：任务树是否包含关键举措节点），否则容易出现误判。

### 2. 可选入参
- `orgResponsibilityText`：组织职责/业务背景描述（用于“相关性 R”的辅助判断）
- `parentTaskTimeRange`：父任务/承接范围（用于校验子任务时间是否在父范围内）
- `semanticConstraints`：额外的表达约束（如：禁止用“提升/优化”等动词）

### 3. 参数校验规则
- `taskTree` 必须可遍历出目标层（至少有 0 个目标节点是允许的，但后续会生成高严重结构问题）。
- 若 `currentLevelType = employee`，则关键举措节点缺失将生成高严重结构问题。
- 若输入节点存在关键举措，但 `currentLevelType` 为组织层且配置规范要求不需要举措，则按“是否误配”生成中严重问题（需要用户确认该层级类型是否正确）。

## 输出
> 输出必须能定位到“具体 BP 节点”，且不展示内部 ID（只展示任务名称/路径）。

### 1. JSON 输出契约（建议）
```json
{
  "resultCode": 1,
  "resultMsg": null,
  "data": {
    "conclusion": "可审计",
    "summary": {
      "highCount": 0,
      "mediumCount": 0,
      "lowCount": 0
    },
    "issues": [
      {
        "severity": "high|medium|low",
        "issueType": "structure|measureStandard|cycle|semantic|granularity|requiredField",
        "locationPath": "目标『...』/关键成果『...』/关键举措『...』",
        "locationSummary": {
          "goalName": "...",
          "keyResultName": "...",
          "actionName": "..."
        },
        "description": "问题描述（为什么不合规）",
        "evidence": "定位到的原始写法/字段值片段",
        "suggestion": "可执行的修复建议（怎么改）",
        "checkRule": "对本问题命中的检查规则编号"
      }
    ]
  }
}
```

### 2. Severity 分级口径
- `high`：必须修改，属于“验收无法成立/承接无法落地”的结构硬缺陷
- `medium`：需要关注，存在明显风险（可能引发验收争议、周期不闭环、语义不清）
- `low`：建议优化，不影响基础可审计性，但可能降低质量

## 前置条件
- BP已创建，`taskTree` 可被遍历。
- 输入中的节点包含至少以下字段之一：`name`、`reportCycle`、`planDateRange`、`measureStandard`、以及父子关系（结构层级）。
- 若要做“语义与相关性”更严谨判断，建议提供 `orgResponsibilityText`。

## 执行逻辑
1. 数据清洗与节点映射：构建“目标-成果-举措”的父子索引，生成 `locationPath`。
2. 结构完整性检查：
   - 检查目标是否存在
   - 检查关键成果是否存在
   - 检查关键举措是否存在（按 `currentLevelType` 取规则）
3. 衡量标准检查（SMART 及可量化性）：
   - 检查每个关键成果是否配置衡量标准
   - 校验衡量标准是否可量化、可验证、时间边界明确
4. 汇报周期链路检查：
   - 检查举措 ≤ 成果 ≤ 目标（周期粒度不倒置）
   - 检查时间范围是否闭环（子任务在父范围内）
5. 必填项检查：
   - 关键字段缺失（负责人/承接人、周期、起止时间等）是否会导致无法承接与验收
6. 语义边界与违规表达检查：
   - 目标/关键成果是否结果导向（“要达到什么结果/状态”）
   - 关键举措是否行动导向（“为了达成结果采取什么具体行动”）
7. 关键举措颗粒度检查（员工层）：
   - 举措是否足够具体可执行（避免过于笼统）
8. 汇总输出：问题去重（同一节点、同一规则仅保留一条或聚合证据），输出 `issues` 与 `summary`。

## 检查规则（核心）
> 下述规则编号用于 `checkRule` 字段，便于追溯与复用。

### 1. 结构完整性
1.1 `缺目标`（high）
- 条件：`goalNodes` 为空
- 描述：当前 BP 不具备“目标→成果→举措”的起点，无法进行验收与承接。
- 建议：补充至少一个目标，并确保其下挂关键成果。

1.2 `缺关键成果`（high）
- 条件：任一目标下的 `keyResultNodes` 为空
- 描述：关键成果是量化验收依据，缺失将导致验收无据可依。
- 建议：为该目标补充至少一个关键成果，并配置对应衡量标准。

1.3 `缺关键举措`（high / medium 取决于层级）
- 条件：任一关键成果下的 `actionNodes` 为空
- 判定：
  - `currentLevelType = employee`：高问题（举措是可承接执行的载体）
  - 组织层（`currentLevelType != employee`）：如果层级规范要求举措为可选，则不强制；若输入显示该层级应存在举措（例如调用方声明 `actions必填`），则报中问题
- 建议：补充关键举措，并让举措具体承接关键成果的达成。

### 2. 衡量标准（measureStandard）
2.1 `衡量标准缺失`（high）
- 条件：任一关键成果的 `measureStandard` 为空/仅空白/仅口号
- 描述：衡量标准是验收的核心依据之一。
- 建议：补充可量化、可验证的衡量口径。

2.2 `衡量标准不可量化或歧义`（high / medium）
- 条件：
  - 缺少可验证要素（如数值、阈值、单位、范围、达成条件）
  - 或出现明显歧义词（仅“提升/优化/加强”等）
- 建议：改写为“指标 + 当前/目标 + 比例/阈值 + 口径/范围 + 时间范围”（尽可能接近 SMART）。

2.3 `SMART 不通过`（medium）
- 规则建议（启发式，不要求严格逐字匹配）：
  - S：描述是否清晰明确（避免纯抽象）
  - M：是否可衡量（数值/单位/阈值）
  - A：是否在合理范围（可由 `orgResponsibilityText` 辅助）
  - R：是否与组织职责相关（可由 `orgResponsibilityText` 辅助）
  - T：是否存在时间边界（可结合 `planDateRange` / 周期）
- 触发：仅当明显偏离时生成中问题；若信息不足，输出低问题并提示补充背景或口径。

### 3. 汇报周期链路与时间闭环（cycle）
> 周期粒度含义：`day/week/doubleWeek` 视为“较短”；`month/doubleMonth/threeMonth` 视为“较长”。

3.1 `周期链路倒置`（high）
- 条件：存在节点的周期粒度关系违反 `举措 ≤ 成果 ≤ 目标`。
- 说明：系统逻辑要求下级先汇报，上级才能汇总；周期越短代表汇报频率越高。
- 建议：调整周期，使子任务周期不长于父任务周期。

3.2 `子任务时间超出父范围`（medium）
- 条件：`planDateRange` 可解析时，子节点范围不在父节点范围内。
- 建议：调整起止时间或让子节点匹配父节点周期与范围。

3.3 `周期字段不可解析`（low）
- 条件：周期字段缺失或无法从 `reportCycle` 判别其粒度顺序。
- 建议：补充可解析的周期字段值（例如 `week+1`、`month+1` 等）。

### 4. 必填字段缺失（requiredField）
4.1 `关键字段缺失`（medium / high）
- 条件：缺失将导致无法承接与验收（例如：节点名称为空、`reportCycle` 缺失、`planDateRange` 缺失、关键字段仅空白等）。
- 分级建议：
  - 若缺失导致无法定位/无法承接：high
  - 若缺失只影响精度或需要二次确认：medium
- 建议：补全字段并确保同一节点信息一致可读。

### 5. 语义边界与违规表达（semantic）
> 该部分不依赖 ID，必须输出“当前写法片段/证据”以便用户理解问题。

5.1 `目标/关键成果带行动性`（medium）
- 条件：目标或关键成果命名/描述明显偏“做什么事/开展什么活动”，缺少“结果/状态/达成口径”。
- 典型风险词（启发式）：开展/实施/推进/优化/建设/调研/培训/组织/落地/执行等
- 建议：将表达改为结果导向（例如“达到X”“完成Y”“覆盖Z%”“缩短N%”）。

5.2 `关键举措带结果性`（medium）
- 条件：关键举措描述偏“结果/指标”，而非“行动/路径/交付物”，导致责任分配与承接不清。
- 典型风险词（启发式）：达到/完成/提升/降低/覆盖等后接具体指标
- 建议：举措改写为行动导向（例如“制定/开展/实施计划，并交付可验收产物或动作”）。

5.3 `结果/动作严重混淆`（high）
- 条件：同时触发 5.1 与 5.2，或语义完全不可分辨（例如既无结果也无可执行动作）。
- 建议：重写该节点表达，至少保证“成果是结果、举措是行动”。

### 6. 关键举措颗粒度（granularity）
6.1 `举措过于笼统`（medium）
- 触发条件：
  - 举措名称无法明确“谁做、做什么、交付什么产物/动作”
  - 或仅有“提升/优化/加强/做好/推进”等形容词与抽象承诺
- 建议：改为具体可执行动作（例如“完成XX的调研并输出报告”“建立XX流程SOP并培训N人”）。

6.2 `举措颗粒度合适`（no-issue）
- 定性判定：能在不额外追问的情况下理解举措交付物/动作边界，并能承接到责任人执行。

## 结果可信度与解释
- 可信度要求：
  - 所有问题必须可定位到具体 BP 节点（`locationPath` 中必须包含名称与层级路径）
  - 每条问题需要至少一条证据字段：`evidence`
  - 每条问题必须给出可执行建议：`suggestion`
- 口径声明：
  - 该 Skill 主要做结构与规范性审计，不做跨 BP 的冲突结论（跨 BP 属于 `BP_AUDIT_03` 处理）。

## 安全与权限
- 只读：不修改 BP 内容。
- 不输出/不依赖内部 ID：对外输出仅使用名称与路径定位。

## 所依赖的业务 API 清单（可选）
> 如果调用方不直接提供 `taskTree`，可通过 Open API 拉取目标树数据后再审计。

- `4.1 查询周期列表`：获取 `periodId`
- `4.2 获取分组树`：获取组织/个人分组结构
- `4.3 批量查询员工个人分组 ID`：员工列表转 `groupId`
- `4.4 查询任务树`：快速获取目标→成果→举措简要信息
- `4.5 获取一组目标数据（含关键成果与举措）`：用于补全 `measureStandard` 等明细字段（推荐）
- `4.6 获取一组关键成果` / `4.7 获取关键举措详情`：用于补全缺失字段

## 异常处理
- 参数错误：
  - `taskTree` 不可遍历：返回 `resultCode=0` 并在 `resultMsg` 提示数据格式错误
- 数据缺失：
  - 缺少目标/成果/举措：生成 `high` 问题并继续输出其它可判定项
- 周期无法解析：
  - 生成低问题“周期字段不可比较”，提示补充 `reportCycle` 的有效值
- 语义输入不足：
  - 若无法判断相关性：降级为低问题并建议补充 `orgResponsibilityText`

## 测试与验收
### 单元测试样例（必覆盖）
1. `缺成果`：某目标下 `keyResultNodes` 为空 → 生成 `high`，issueType=structure
2. `缺衡量标准`：某关键成果 `measureStandard` 空 → 生成 `high`，issueType=measureStandard
3. `缺举措`（employee层）：某关键成果无举措 → `high`，issueType=structure
4. `周期倒置`：举措 `week`，成果 `quarter`（举措周期长于成果）→ `high`，issueType=cycle
5. `语义偏行动`：目标命名为“推进XX建设” → `medium`，issueType=semantic
6. `语义偏结果`：关键举措命名为“完成客户拜访量提升到50家” → `medium`，issueType=semantic
7. `举措过于笼统`：举措“优化流程” → `medium`，issueType=granularity

### 边界/异常场景
- `taskTree` 只有目标，没有成果：按缺成果生成 high
- 节点名称为空：按必填字段缺失生成 medium/high
- 子任务时间不可解析：跳过时间闭环校验并生成低问题提示字段格式

### 回归测试要求
- 任一规则更新后，必须验证 `locationPath` 始终可定位到对应节点。
