## 二、系统概述

### 2.1 系统一句话定义

文档服务将调用方**在一次请求中上传的业务文件**，转换为可供机器处理的 **Markdown 正文**及摘要级元信息。

### 2.2 系统解决的核心问题

1. **格式异构**：Word、PDF、PPT、表格、图片等格式不一，下游检索与 AI 场景需要统一、可切分的文本形态。
2. **能力复用**：避免各业务系统重复集成转换引擎，解析规则、安全与稽核口径难以统一。
3. **耗时与排队**：大文件或含识别增强的路径耗时长，需要提供**同步等待**与**异步任务 + 查询/回调**两种形态。
4. **扫描件与图片**：电子版 PDF 可抽文本；扫描页或纯图片在策略开启时需补充识别后才能形成可用正文。

### 2.3 系统在整体业务中的定位

文档服务定位于**内容解码与标准化层**：

- **向上**：承接持有合法 **应用密钥（appKey）** 的业务系统或后端服务调用。
- **向下**：输出 **Markdown 产出物**，供知识库、RAG、工作流、内容分析等消费。
- **不替代**：不负责原件长期存储、权限审批、版本管理；原件由调用方在上传前自行管控。

### 2.4 系统不负责解决的问题

- **原件托管**：不充当企业网盘；异步场景仅短暂暂存以完成一次转换，完成后按策略清理。
- **版式还原**：不保证与原件版式一致，目标是可阅读、可检索的文本结构。
- **办公协同**：不承载流程、评论、协同编辑等业务。
- **全局削峰编排**：异步队列默认可理解为单服务实例内的排队；跨实例、跨机房的一致队列语义需架构层另行设计。

## 三、核心业务对象

### 3.1 直传文件

**定义**：调用方在一次请求中直接提交的**文件内容**（对应上传表单中的文件字段），作为当次转换的唯一输入。

**作用**：

- 适用于业务侧已持有字节、或无需先进入其他存储系统再指回资源号的场景。
- 同步路径：上传后立即转换并返回结果。
- 异步路径：上传后写入暂存，由后台任务消费，避免长时间占用调用方连接。

**与其他对象的关系**：

- 经**转换管线**生成 **Markdown 产出物**。
- 异步场景下会关联一条**转换任务**。

**常见误解**：

- **误解**：直传等于免鉴权。纠正：须携带有效 **appKey**，服务端据此解析调用方身份并落库审计。
- **误解**：服务会长期保存上传原件。纠正：暂存仅服务于当次任务生命周期，不做长期档案库。

### 3.2 转换任务

**定义**：在**异步直传**模式下，代表「从已上传字节到 Markdown 产出」的一次可追踪执行单元；可被查询进度与结果，并可选择向调用方配置的地址发送完成通知。

**作用**：

- 将排队与长耗时转换与调用方解耦。
- 保留提交时的**调用方身份**信息，供审计与后台执行阶段合规调用外部能力。

**与其他对象的关系**：

- 由一次**直传文件**的异步提交创建。
- 终态成功时产出 **Markdown 产出物**（与同步成功语义一致）。

**常见误解**：

- **误解**：任务失败会自动无限重试。纠正：默认可查询失败原因；是否由调用方重新提交，属于上层编排责任。
- **误解**：多实例部署下任务必然全局唯一可见。纠正：任务持久化与队列消费与部署拓扑相关，业务上「能建单」不等于「任意节点立即可见同一进度」。

### 3.3 Markdown 产出物

**定义**：一次成功转换的主要交付：**Markdown 正文**，以及帮助理解转换路径的摘要级元信息（如是否经过图像识别、PDF 是否走混合页策略等）。

**作用**：

- 作为下游推荐的**机器可读正文**输入。
- 元信息便于判断本次是否走了增强识别路径，支持运营与排障。

**与其他对象的关系**：

- 由**直传文件**经转换管线生成。
- 与一次成功的**转换任务**（或一次同步直传）为一对一终结关联。

**常见误解**：

- **误解**：Markdown 等于原始 Word 排版。纠正：侧重结构与可读性，不是印刷级还原。
- **误解**：元信息包含全文敏感明细。纠正：仅传递摘要级标识；全文敏感内容不应扩散到非必要日志。

### 3.4 调用方身份（appKey）

**定义**：业务系统向文档服务证明「本次调用来自哪个已注册应用」的凭证；服务端据此解析出人员与组织维度信息，用于审计及（若开启）外部识别能力的合规调用。

**作用**：

- 将「哪个应用、在什么身份上下文中触发了转换」与业务流水对齐。
- 不满足校验的调用不应进入转换管线。

**与其他对象的关系**：

- 附属每一次同步直传、异步建单与任务查询。
- 不表示调用方对任意第三方文件的天然下载权；直传模式下输入即本次上传内容本身。

**常见误解**：

- **误解**：appKey 可暴露在前端页面。纠正：应由后端服务持有并转发，避免泄露与滥用。
- **误解**：身份解析失败仍可匿名转换。纠正：鉴权失败即拒绝，无「降级匿名」业务路径。

### 3.5 转换结果缓存

**定义**：对**相同文件内容**（按内容摘要比对）在近期成功转换过的结果，可在策略开启时直接复用，而不再重复完整计算。

**作用**：

- 降低重复上传相同文件时的成本与等待时间。
- 调用方可通过产出物上的「是否命中缓存」类信息判断本次是否为复用结果。

**与其他对象的关系**：

- 作用于**直传文件**进入转换管线之前或之中的短路逻辑。
- 不改变 **Markdown 产出物** 的业务定义；命中缓存时产出语义与实算成功一致。

**常见误解**：

- **误解**：缓存按文件名判断。纠正：以**文件内容**为准；同名不同内容不会误复用。
- **误解**：缓存永久有效。纠正：保留时长由部署策略决定，过期后需重新实算。

### 3.6 可选文字识别增强（OCR / 扫描 PDF）

**定义**：在运维开启策略时，对栅格图片、PDF 中疑似扫描页等调用外部识别能力，将图像中的文字并入 Markdown。

**作用**：

- 补齐常规文档解析难以从图像直接得到的文本。
- PDF 混合策略下按页决定是否识别，平衡质量与成本。

**与其他对象的关系**：

- 是**转换管线**内的可选分支，不改变直传与任务的对象定义。
- 通常依赖与 appKey 配套的识别服务鉴权策略。

**常见误解**：

- **误解**：所有 PDF 都会整本 OCR。纠正：电子版优先抽文本；仅低正文字数的页在策略允许时可能触发识别。
- **误解**：识别结果可作为唯一法律依据。纠正：适用于机器处理与人工复核前的草稿层。

## 四、核心业务关系

### 4.1 业务主链路（同步直传）

```
携带 appKey → 上传直传文件 → 转换管线（含可选识别分支、可选结果缓存）→ Markdown 产出物
```

### 4.2 业务主链路（异步直传）

```
携带 appKey → 上传直传文件并建单 → 立即获得任务标识
        ↓
后台：暂存字节 → 转换管线 → Markdown 产出物
        ↓
调用方：轮询任务状态 / 接收 Webhook 通知
```

### 4.3 对象关系说明

| 关系 | 说明 |
|------|------|
| 直传文件 → 字节 | 上传内容仅用于当次计算；同步结束后不承诺长期保留原件。 |
| 字节 → Markdown 产出物 | 一次转换的主交付，一对一。 |
| 异步提交 → 转换任务 | 提交创建任务；任务驱动暂存读取与完整转换。 |
| 转换任务 → 回调通知 | 若配置了可达的回调地址且策略允许，终态时向调用方投递通知。 |
| 相同内容 → 转换结果缓存 | 可选；命中时缩短路径，产出语义与实算成功一致。 |

### 4.4 关键流转逻辑（转换任务生命周期）

```
待排队 → 执行中 → 成功（可交付 Markdown） / 失败（原因可查询）
```

- **成功**：调用方通过轮询或回调获得与同步成功等价的产出语义。
- **失败**：典型包括格式不支持、体积超限、识别服务不可用、服务繁忙排队超时等；具体归类以运行反馈为准。

## 五、业务规则

### 5.1 显性规则

1. **鉴权规则**：业务转换类能力须提供有效 **appKey**；校验失败不得进入转换。
2. **输入规则**：每次请求仅处理**一个**上传文件；不支持一次请求批量多文件。
3. **同步与异步分工**：同步路径在连接保持期间完成转换并返回；异步路径提交后立即返回任务标识，结果通过查询或回调获取。
4. **结果保留规则**：异步任务结果在约定保留期内可查询，过期后不再提供正文，业务侧应及早消费或自行持久化。
5. **体积与复杂度边界**：同步路径对单文件体积设有上限；PDF 页数等亦有保护上限，超限视为本次不支持而非无限排队。
6. **强制重算规则**：若调用方声明跳过缓存（业务上的「强制重新转换」），则不得因历史命中缓存而返回旧结果。

### 5.2 隐性规则

1. **只读输入**：转换过程读取上传字节，不回写调用方业务系统中的任何原件。
2. **幂等与回调**：同一任务终态可能因重试机制被多次通知，调用方宜按任务标识幂等处理。
3. **识别能力开关**：图片 OCR、PDF 混合策略由环境级策略控制，不是单次请求随意组合的能力菜单。
4. **审计留痕**：同步成功或失败均可形成任务维度的记录，便于与异步任务统一查询与追溯（具体可见性以对接约定为准）。

### 5.3 权限或范围限制

1. **应用范围**：仅已分配 appKey 的应用可调用；密钥不得转借未授权系统。
2. **直传开关**：运维可关闭直传同步或异步入口；关闭后该路径不可用，调用方需调整方案。
3. **回调安全**：配置的回调地址须满足 HTTPS 等安全策略，禁止指向不可信内网或元数据地址。

### 5.4 特殊说明或例外

1. **探测能力**：存活与就绪探测用于运维编排，不参与「上传转换」业务语义。
2. **服务繁忙**：高峰时同步可能因转换槽满而快速失败或超时失败，大文件宜走异步。
3. **多 Worker / 多副本**：提升吞吐时，任务是否跨实例共享存储与队列，取决于部署架构，不属于本说明承诺的全局语义。

## 六、常见歧义与边界

### 6.1 同步与异步

| 歧义点 | 澄清说明 |
|--------|----------|
| 异步是否支持更大文件 | 异步主要解决排队与长耗时，体积上限类别与同步相近或按部署略有差异，并非「异步无上限」。 |
| 同步失败是否应改异步重试 | 因格式、超限导致的失败，改异步通常仍失败；因繁忙导致的失败，可改异步或退避重试。 |

### 6.2 缓存与强制重算

| 歧义点 | 澄清说明 |
|--------|----------|
| 命中缓存是否算「没转换」 | 业务上仍视为成功产出；正文与实算成功一致，仅未重复消耗计算资源。 |
| 内容微调后仍命中 | 以内容比对为准；字节级变化即视为新内容。 |

### 6.3 文档服务与调用方

| 歧义点 | 澄清说明 |
|--------|----------|
| 谁保管原件 | 调用方在上传前自行保管；文档服务不做长期原件库。 |
| 谁决定能否调用 | 文档服务校验 appKey；能否拿到文件字节由调用方上游业务决定。 |

### 6.4 产出物性质

| 歧义点 | 澄清说明 |
|--------|----------|
| Markdown 用途 | 面向检索、问答、二次编辑的中间态，不是法务存档格式。 |
| 复杂版式与嵌入对象 | 可能简化或丢失，以正文可抽取为先。 |

### 6.5 Webhook 与轮询

| 歧义点 | 澄清说明 |
|--------|----------|
| 是否必须配回调 | 否；可仅轮询任务状态。 |
| 回调失败是否等于转换失败 | 否；转换可能已成功，仅通知投递失败，可通过任务查询与投递状态字段区分。 |

## 七、术语表

| 术语 | 说明 |
|------|------|
| 文档服务 | 将上传文件转为 Markdown 的公司内部标准化后端能力。 |
| 直传文件 | 调用方在单次请求中直接提交的文件内容。 |
| 转换任务 | 异步场景下可查询进度与结果的任务单元，可选回调。 |
| Markdown 产出物 | 转换成功后的正文与摘要级元信息整体。 |
| appKey / 应用密钥 | 标识调用应用的凭证，用于身份解析与鉴权。 |
| 调用方身份 | 由 appKey 解析得到的人员与组织维度，用于审计。 |
| 转换管线 | 从文件字节到 Markdown 的内部处理链路总称（业务抽象）。 |
| 转换结果缓存 | 按文件内容复用近期成功结果的可选机制。 |
| 文字识别增强（OCR） | 对图像或扫描页补充文字的可选能力。 |
| PDF 混合策略 | 按页区分文本抽取与扫描页识别的组合策略。 |
| Webhook / 回调 | 任务终态后向调用方指定地址投递通知的机制。 |
| 强制重新转换 | 调用方要求跳过结果缓存、重新实算的业务语义。 |
