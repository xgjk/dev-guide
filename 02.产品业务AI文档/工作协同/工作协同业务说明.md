## 一、系统概述

**工作协同系统**是一套用于组织内部 **记录工作信息、进行沟通讨论、形成管理决策、跟踪工作执行过程** 的协同平台。

系统以 **汇报** 作为核心信息载体。员工通过发送汇报同步工作情况，相关人员可以在汇报上进行 **回复讨论、提出反馈、给出建议或做出决策** 。当需要持续执行某项工作时，可以创建 **任务** 进行跟踪。

系统通过 **汇报 → 讨论 → 决策 / 任务 → 汇报** 的方式，形成组织工作的记录与协同闭环。

# 二、系统核心对象

系统主要由以下六类核心对象构成。**对象定义采用统一结构：定义、系统定位、与相邻概念的边界、典型产生或关联。**

**总述：汇报是系统的核心对象；回复是最基础的交互行为。反馈不是与回复并列的独立对象，而是回复的一种业务处理类型。事项不是汇报本身，而是发起汇报时可选的业务归类或模板来源。任务不承担决策职能，是执行与跟踪对象。待处理是系统归集的待办载体，由汇报流程节点、需反馈/需决策指派与任务链路触发产生。**

---

### 1. 汇报

| 维度 | 说明 |
| ---- | ---- |
| **定义** | 工作信息在系统中的主要记录与呈现单元，承载进展、问题、建议、执行结果等内容。 |
| **系统定位** | 核心对象；信息进入系统的主入口；建议与决策流程的载体。 |
| **边界** | 汇报本身不是任务；决策记录在汇报流程中，而非在任务上完成。 |
| **典型产生或关联** | 可由人员直接新建，或基于「事项」模板发起；可关联附件、其他汇报/BP/邮件等；执行人通过发送汇报反馈任务进展与结果。 |

**主要能力（交互入口）**

- 新建并发送汇报：包含标题、正文、流程（可设置多个节点的接收人、建议人、决策人）、可上传附件，以及关联其他内容（汇报/BP/邮件等）。
- 可选择系统设置的事项模板发起汇报：事项可配置特殊表单字段（文字、时间、选项、上传文件等）及对应流程人员。
- 支持收件箱/发件箱，用于查找发送与接收的汇报。

---

### 2. 回复

| 维度 | 说明 |
| ---- | ---- |
| **定义** | 用户对某条汇报发表的讨论性或处理性内容，是系统中最基础的交互行为。 |
| **系统定位** | 围绕汇报展开；可承载讨论、补充信息、提出建议，以及 **指定他人处理（即触发反馈类处理）** 。 |
| **边界** | 回复是行为与载体；「反馈」是回复在业务上的一种处理类型（见下），二者不是同一层级的并列对象。 |
| **典型产生或关联** | 产生自查看汇报后的互动；可含文字、语音、附件；可升级为需指定人员处理的反馈场景。 |

**主要能力**

- 支持文字/语音/附件回复。
- 回复内容可设置“需反馈”：设置反馈对象后，对应人员会收到待办；反馈也可进一步升级为任务推进执行，执行人通过回流汇报反馈结果。
- 支持“我的回复”查询：用于查找用户发出的所有回复。

---

### 3. 反馈

| 维度 | 说明 |
| ---- | ---- |
| **定义** | **回复的一种业务处理类型**：当回复需要指定具体人员进行跟进、说明或落实时，通过设置反馈对象形成待处理事项。 |
| **系统定位** | 从属于「回复」的业务语义，不单独作为与回复平行的第一类核心实体；被反馈人须对该反馈做出处理。 |
| **边界** | 反馈不是任务；处理反馈的方式可以是 **再回复说明结果** ，或在必要时 **创建任务** 推进执行，但「反馈」本身不等于「任务」。 |
| **典型产生或关联** | 由带反馈指向的回复产生；被反馈人回复处理结果，或创建任务后通过汇报反馈执行进展。 |

**主要能力**

- 以“带反馈对象”的回复形态触发待办分配：反馈人需进行处理。
- 反馈处理完成的方式为再回复处理结果；必要时可升级为任务，由任务链路承接持续执行。

---

### 4. 任务

| 维度 | 说明 |
| ---- | ---- |
| **定义** | 用于对需要持续跟踪的工作进行执行安排与进展管理的对象。 |
| **系统定位** | **执行对象**，用于落实工作、约定执行人与汇报频次等；**不承担决策职能**。 |
| **边界** | 决策只发生在汇报（及汇报上的建议/决策流程）中；任务状态与结果以 **汇报记录** 为准，而非在任务上「做决策」。 |
| **典型产生或关联** | **主要由人员直接创建**（约 90%）；少部分由反馈场景升级产生；执行人按频次通过 **发送汇报** 提交进展或结果。 |

**主要能力**

- 创建任务：支持设置任务要求，指定汇报人、接收人、汇报频次来跟踪任务进展，并可配置截止时间等。
- 系统按频次向执行人触发待办/提醒；执行人通过发送汇报提交进展或结果（回流到汇报链路）。
- 任务的相关列表可用于查看用户所有相关任务，并可发起任务关闭/结项申请（形成待办）。

---

### 5. 事项

| 维度 | 说明 |
| ---- | ---- |
| **定义** | 系统预置或配置的一类工作归类与 **汇报模板来源**（可含固定表单字段、默认流程人员等）。 |
| **系统定位** | 发起汇报时的 **可选项** ，用于归类与规范输入，**不是每条汇报的必备前提**。 |
| **边界** | 事项 ≠ 汇报：选用事项只是用模板/归类发起一条 **新的汇报** ，事项本身不是汇报实体。 |
| **典型产生或关联** | 用户发送汇报时可「直接发送」或「按事项模板发送」；实际使用中 **大多数汇报为直接发送、不关联事项** 。 |

**主要能力**

- 作为模板/归类来源带出特殊表单字段与流程人员：用户选择事项发起汇报时，事项会携带对应字段与流程配置。

### 6. 待处理

| 维度 | 说明 |
| ---- | ---- |
| **定义** | 系统归集的待办/处理项载体，用于承载“需要用户做处理”的要求。 |
| **系统定位** | 待办归属与提醒的载体；可追溯到触发对象（汇报流程节点、需反馈/需决策指派、任务链路等），用于驱动用户需完成输入。 |
| **边界** | 待处理不是汇报、回复、反馈或任务本身；其完成动作由用户提交的对应对象内容（回复/汇报/结果）来体现。 |
| **典型产生或关联** | 汇报流程节点需要用户给出建议/决策时产生；当收到反馈指派时产生；任务要求执行人提交进展汇报或发起关闭/结项申请时产生。 |

**主要能力**

- 待办展示与提醒：在待处理列表中呈现归属对象（触发汇报/节点/回复/任务等）、发起人、发起时间。
- 待办完成与闭环：用户完成对应输入提交后，可将待处理标记为完成；完成动作会触发后续步骤（进入下一个流程节点、更新任务链路、或关闭相关待办）。

# 三、核心业务关系

以下三段为系统内主线的硬性归纳，与后文业务规则一致。

**（一）汇报是信息进入系统的主要入口。** 绝大部分工作信息以汇报形式进入系统；汇报可承载进展、数据、问题、建议、任务执行结果等。发送时可不关联事项，也可选用事项作为归类或模板。

**（二）回复、反馈、建议、决策均围绕汇报展开。** 用户在汇报上回复讨论；建议与决策流程附着在汇报上；反馈是回复的一种业务处理类型，用于指定人员跟进。管理上的「结论/决策」记录在汇报流程中。

**（三）任务用于承接执行，执行结果通过汇报回流。** 任务负责执行与跟踪；进展与结果由执行人通过 **发送汇报** 记录，形成 **汇报 → 讨论 / 决策 / 任务 → 再汇报** 的闭环。

关系示意（与上文互补）：

```
汇报（主入口）
 ↓
回复（讨论 / 建议 / 含「反馈」类处理）
 ↓
产生结果
 ├─ 决策（仅发生在汇报流程中）
 └─ 任务（执行；决策不在此发生）
```

```
任务 → 执行 → 汇报（进展/结果）
```

整体循环：

```
汇报 → 讨论 → 决策 / 任务 → 汇报
```

# 四、业务规则

以下条款为系统应遵循的业务约束表述，与「对象定义」「核心关系」一致。

---

**规则 1 — 汇报为信息入口**  
系统中大部分工作信息通过 **汇报** 进入。汇报可含：工作进展、数据说明、问题反馈、决策建议、任务执行结果等。发送方式包括：**直接发送汇报**，或 **选用已配置的事项（模板/归类）发送** 。**大多数汇报可不关联事项。**

---

**规则 2 — 回复为基础交互**  
用户查看汇报时可 **回复** 。回复用于：讨论问题、提出建议、补充信息、以及 **指定人员处理（触发反馈类处理）** 。

---

**规则 3 — 反馈是回复的处理类型**  
当回复需指定某人处理时，设置 **反馈对象** 。被反馈人须处理；处理方式为：**直接回复说明处理结果**，或 **创建任务** 推进落实。**反馈从属于回复的业务语义，不与回复并列作为独立对象层级。**

---

**规则 4 — 任务来源与创建方式**  
**任务主要由人员直接创建（约 90%）** ；**少部分由反馈场景升级产生** 。创建时可设执行人、汇报频次、截止时间等；系统可按频次 **提醒执行人** 。**执行人须通过发送汇报提交进展或结果。**

---

**规则 5 — 建议与决策仅在汇报流程**  
发送汇报时可启用 **建议流程** ，收集多位管理者意见并形成决策。决策须明确（如同意/不同意等），并作为 **决策记录生成在汇报流程的决策节点上**（关联到该汇报流程）。

---

**规则 6 — 决策只发生在汇报上**  
**决策行为只发生在汇报（及其建议/决策流程）中，不发生在任务上。** 任务仅用于执行；若决策需落地，再通过创建任务等方式执行。

---

**规则 7 — 任务不承担决策职能**  
**任务为执行对象，不承担决策职能。** 是否批准、是否采纳等管理判断均在汇报侧完成。

---

**规则 8 — 任务进展与结果仅通过汇报记录**  
任务执行过程中的进展与结果，**须通过执行人发送汇报进行记录** ；业务上 **执行结果归集在汇报链路** ，不将任务本身作为决策或结论的载体。

---

**规则 9 — 事项为可选归类与模板**  
汇报可关联事项，也可不关联。**事项不是所有汇报的必备前提** ；**事项 ≠ 汇报本身** ，仅为发起时的模板/归类来源。

---

**规则 10 — 任务与汇报的来源关系**  
任务 **主要** 由人员直接创建；**亦可** 由回复/反馈路径升级产生。**任务不必定来源于某条汇报** ，但执行结果 **应通过汇报反馈** 。

---

# 五、常见歧义与边界

本章用于统一最易混淆的说法，与对接口、流程、权限设计时的理解一致。

| 歧义点 | 边界说明 |
| ------ | -------- |
| **汇报 ≠ 任务** | 汇报是信息记录与决策载体；任务是执行与跟踪载体。执行工作安排用任务，同步信息与定调用汇报。 |
| **回复 ≠ 反馈** | 回复是行为与内容单元；反馈是「回复需指定他人处理」时的 **业务处理类型** ，从属于回复语义。 |
| **反馈 ≠ 任务** | 反馈的处理方式 **可以** 包含创建任务，但反馈本身是待跟进的处理指派，不是任务实体。 |
| **决策发生在汇报上，不发生在任务上** | 建议、意见、决策记录均附着在汇报流程；任务只执行已形成的决策结论，不在任务上完成决策。 |
| **事项 ≠ 所有汇报的必备前提** | 大多数汇报可直接发送；事项仅为可选模板与归类，不选事项不影响该条作为「汇报」成立。 |

# 六、术语表

| 术语 | 简要定义 |
| ---- | -------- |
| **汇报** | 系统中工作信息的主要记录单元，信息主入口，建议与决策流程的载体。 |
| **回复** | 用户对汇报发表的讨论或处理内容，最基础交互；可触发反馈类处理。 |
| **反馈** | 回复的一种业务处理类型：指定具体人员跟进、说明或落实，不等于任务本身。 |
| **任务** | 用于持续执行与跟踪的对象，不承担决策职能；结果通过汇报反馈。 |
| **事项** | 发起汇报时可选的业务归类与模板来源，非汇报本身，非必备前提。 |
| **建议** | 在汇报（及建议流程）上形成的意见性输入，可参与形成决策。 |
| **决策** | 在汇报（及建议流程）上形成的确定性结论（如同意/不同意）；不在任务上发生。 |
| **待处理** | 系统归集的需用户处理项，如：须填建议、须填决策、须反馈、任务相关待写汇报、任务关闭申请等。 |

# 七、系统运行模式

工作协同系统通过以下机制运行：

```
汇报记录工作
 ↓
回复进行讨论
 ↓
形成反馈或决策
 ↓
创建任务执行
 ↓
通过汇报反馈执行结果
```

该机制使组织工作形成 **完整记录与执行闭环** 。

# 八、典型场景库

本章给出可直接用于接口/业务推理的样例。每个场景都围绕同一套主线：**汇报**作为输入入口，**回复/反馈**作为讨论或指派，**任务**负责执行，执行结果再通过**汇报回流**沉淀。

### 场景模板

| 项 | 内容 |
| ---- | ---- |
| 输入触发 | 谁在什么对象上发起、是否关联事项、是否启用建议/决策流程 |
| 关键动作 | 依次发生的交互（回复类型、是否产生反馈、是否升级任务等） |
| 生成/更新对象 | 汇报、回复、反馈、待处理、任务、回流汇报（如适用） |
| 判定点 | 哪些字段/条件使得系统判定“形成决策/形成反馈/任务推进/闭环完成” |

---

### 场景 1：仅发起汇报（知会/同步信息）

| 项 | 内容 |
| ---- | ---- |
| 输入触发 | 员工创建并发送 `汇报 R1`（可不关联事项）。目的仅是让接收人了解当前工作内容，不强制启用建议/决策流程。 |
| 关键动作 | 接收人查看 `汇报 R1`，如需要可进行一般讨论回复（不形成反馈指派、不产生决策节点结论）。 |
| 生成/更新对象 | `汇报 R1`，以及可能产生的讨论型回复（不形成反馈、决策记录或任务）。 |
| 判定点 | `汇报 R1`成功发送并可被接收人查看，即视为该场景完成；是否产生后续回复不影响“同步信息”的目标成立。 |

---

### 场景 2：发起汇报 + 建议/决策流程（递进；可多轮决策）

| 项 | 内容 |
| ---- | ---- |
| 输入触发 | 员工发送 `汇报 R1`，启用该汇报上的建议/决策流程，并配置建议节点与决策节点（可按节点顺序执行）。 |
| 关键动作 | 建议节点依次产生建议回复；随后进入决策节点，由决策人给出“同意/不同意”等结论，系统生成对应的决策记录。典型多轮示例：节点1建议（张三）、节点2建议（李四、王五）、节点3决策（张六给出同意/不同意）、节点4决策（李七再次给出同意/不同意，并生成新的决策记录，按节点顺序演进）。 |
| 生成/更新对象 | `汇报 R1`（关联建议回复链 + 一轮或多轮决策记录）；以及汇报流程侧的“待填写建议/待填写决策/待给出反馈”（如产生）。 |
| 判定点 | 每个决策节点产生一条或多条决策记录即视为该节点完成；同一汇报可出现多个决策节点并产生多次决策记录，严格按节点顺序完成。 |

---

### 场景 3：建议/决策完成后 -> 决策任务回流（闭环的执行阶段）

| 项 | 内容 |
| ---- | ---- |
| 输入触发 | 在 `汇报 R1` 的建议/决策流程完成后，需要将决策落地执行。 |
| 关键动作 | 系统创建 `任务 T1`（或由决策相关流程触发任务创建）。执行人按汇报频次发送进展/结果汇报 `回流汇报 R2`，必要时发起任务关闭/结项申请，形成任务闭环。 |
| 生成/更新对象 | `任务 T1`、任务链路待办/提醒、执行中产生的回流汇报 `R2`，以及任务关闭相关记录。 |
| 判定点 | 任务闭环以执行人回流汇报与任务关闭/结项待处理完成为准；决策记录来自 `汇报 R1` 的决策节点（不在任务上完成）。 |

---

### 场景 4：反馈指派后直接处理（不升级任务）

| 项 | 内容 |
| ---- | ---- |
| 输入触发 | 员工发送 `汇报 R1`，接收方发起“反馈类型”回复并设置反馈对象（指派某人处理）。 |
| 关键动作 | 被反馈人直接在该反馈语义下提交处理结果（无需创建任务）。系统将该反馈对应的待处理标记为完成。 |
| 生成/更新对象 | `汇报 R1`、一条“反馈类型”回复、被反馈人的处理结果回流（通过回复呈现）以及待处理完成记录。 |
| 判定点 | “反馈完成”以被反馈人对反馈对应处理结果的回复为准；不以任务状态作为反馈完成条件。 |

### 场景 5：反馈升级任务（反馈 -> 任务 -> 回流汇报）

| 项 | 内容 |
| ---- | ---- |
| 输入触发 | `汇报 R1` 上的回复设置反馈对象，但该处理需要持续执行，于是将该反馈推进为执行任务。 |
| 关键动作 | 生成 `任务 T1`（任务来源为反馈升级）。执行人按频次提交回流汇报（进展/结果），并按需要完成任务关闭/结项相关动作。 |
| 生成/更新对象 | `汇报 R1`、反馈类型回复、`任务 T1`、执行过程的回流汇报 `R2/R3...`、任务关闭相关待处理（如适用）。 |
| 判定点 | 任务来源可标记 `from_feedback`；决策仍只在 `汇报 R1` 的汇报流程侧形成。 |

### 场景 6：按事项模板发起 + 建议/决策流程（入口变体）

| 项 | 内容 |
| ---- | ---- |
| 输入触发 | 使用 `事项 I1` 的模板发起 `汇报 R1`，并启用该汇报上的建议/决策流程。 |
| 关键动作 | 建议人回复形成意见；决策人在汇报流程的决策节点上给出“同意/不同意”等结论，生成决策记录。 |
| 生成/更新对象 | `事项 I1`（作为发起来源）、`汇报 R1`、建议/决策回复链、（可能多轮的）决策记录；决策落地后再进入任务链路。 |
| 判定点 | 事项作为模板来源，不改变“汇报成立 + 决策只发生在汇报流程”的边界；多轮决策遵循决策节点顺序规则。 |
