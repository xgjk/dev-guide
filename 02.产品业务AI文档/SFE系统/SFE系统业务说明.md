## 二、系统概述

### 系统定义

SFE系统是一个面向医药行业的销售效能管理平台，核心支撑"目标规划-结果填报-审批发布"的业务闭环，帮助企业管理销售目标、跟踪执行结果、收集市场情报。

### 解决的核心问题

1. **规划填报割裂**：销售目标的规划与实际完成数据的填报在线下通过邮件、Excel等方式进行，效率低、格式不统一、难以追溯
2. **进度难以追踪**：无法实时了解各销售区域的目标规划提交进度和审批状态
3. **数据采集分散**：市场情报、医院开发状态、客户跟进结果等数据缺乏统一的收集渠道和格式标准
4. **审批流程不透明**：规划数据的审阅、调整、锁定、发布等环节缺乏系统支撑，权责不清

### 在整体业务中的定位

SFE系统定位于**销售运营的数据采集与管理中枢**：

- **向上**：为BI分析平台、管理报表提供结构化的规划数据和实际完成数据
- **向下**：对接组织架构系统获取销售区域和人员信息，对接消息系统发送任务通知
- **横向**：与CRM系统协同，作为客户聚焦计划等数据的数据源或目的地

### 不负责解决的问题

- **销售执行过程管理**：不管理日常拜访路线规划、活动执行细节（由CRM系统负责）
- **薪酬绩效计算**：不涉及销售提成、绩效考核的自动计算（由HR系统负责）
- **库存订单管理**：不处理库存调拨、订单流转等供应链业务（由ERP系统负责）

## 三、核心业务对象

### 3.1 销售区划（Zone）

**定义**：销售区划（简称：区划）是销售组织的层级单元，按层级从上到下形成树状结构，如总部-大区-省区-地区-辖区。辖区是最底层的区划，由代表直接负责。

**作用**：

- 作为数据采集项目的执行范围，决定哪些区域的人员需要参与填报
- 作为数据项的归属单元，明确每条数据由哪个区域的负责人填报和管理

**与其他对象的关系**：

- 区划与人员绑定，一个区划可有一个或多个负责人
- 数据采集项目指定目标区划范围，生成任务下发给区划负责人
- 数据项归属特定区划，由区划负责人填报和维护

**常见误解**：

- 误解：区划就是地理区域。纠正：区划是销售组织单元，与地理区域有对应关系但不完全等同，同一个地理区域可能划分为多个区划
- 误解：区划层级固定。纠正：区划层级因企业组织架构而异，不同企业可能有总部-大区-省区-地区四级，也可能有三级或更多

### 3.2 覆盖分管（Coverage）

**定义**：覆盖分管描述人员的职责范围，包括"直接负责的区划"（直属）和"协助管理他人的区划"（分管）。

**作用**：

- 明确人员在数据采集项目中的角色：直属区划用于填报，分管区划用于审批和管理
- 支持一人多岗场景，如地区经理同时负责自己的区划和分管下属区划

**与其他对象的关系**：

- 覆盖分管基于人员与区划的关系建立
- 数据采集项目根据覆盖分管关系自动分配填报任务和审批权限
- 在表格提交界面，左侧导航区分展示"直属区划"和"分管区划"节点

**常见误解**：

- 误解：覆盖分管就是汇报关系。纠正：覆盖分管是业务职责范围，汇报关系是行政管理关系，两者不完全一致
- 误解：分管区划的人可以直接修改数据。纠正：分管区划默认只读，可执行审批操作，但数据填报由直属负责人完成

### 3.3 画像（Profile）

**定义**：画像是客户在特定产品下的属性标签集合，由客户与产品共同决定。同一客户在不同产品下可能拥有不同的画像标签。

**作用**：

- 描述客户在特定产品领域的特征，如客户等级、合作潜力、市场地位等
- 为覆盖分管提供筛选维度，支持针对特定类型客户发起采集任务
- 辅助销售分析和客户分层管理

**与其他对象的关系**：

- 画像由客户 + 产品共同定义，一个客户在不同产品下有不同画像
- 覆盖分管是区划 + 画像的组合，确定谁负责哪个区划内哪类客户的采集工作
- 数据采集项目可通过画像筛选目标客户范围

**常见误解**：

- 误解：画像是区划属性。纠正：画像是客户属性，与产品相关，不同产品下同一客户可能有不同画像
- 误解：画像只与客户相关。纠正：画像由客户和产品共同决定，客户在不同产品线可能有不同标签

### 3.4 数据采集项目（Project）

**定义**：数据采集项目是一次或周期性的数据收集任务，定义了采集什么数据、由谁填报、在什么时间范围内完成。

**作用**：

- 统筹数据采集活动，将任务下发给目标区划的负责人
- 管理采集周期（一次性/周期性）和项目状态（规划中/进行中/暂停/已完成）

**与其他对象的关系**：

- 项目指定目标区划范围，生成任务下发给区划负责人
- 项目关联模板，定义需要采集的字段结构
- 项目可关联其他项目（如结果填报项目关联目标规划项目）
- 项目周期触发数据项的生成

**常见误解**：

- 误解：项目创建后数据就自动产生。纠正：项目只是任务框架，数据项需要填报人创建或在周期开始时自动生成
- 误解：周期性项目每个周期都是新项目。纠正：周期性项目是同一个项目，按周期触发不同批次的数据项

### 3.5 周期（Period）

**定义**：周期是数据采集的时间维度，支持年、季、月、周、日等粒度。周期性项目按周期触发任务，一次性项目只有一个周期。

**作用**：

- 定义数据的时间归属，便于对比分析（如月度实际 vs 月度目标）
- 支持周期切换和跨周期数据查看

**与其他对象的关系**：

- 周期属于数据采集项目的配置属性
- 数据项关联特定周期，代表该数据项属于哪个时间范围

**常见误解**：

- 误解：周期就是日历周期。纠正：系统支持自然周和ISO-8601周两种周定义，企业可根据业务需求选择

### 3.6 模板（Schema）

**定义**：模板定义了数据采集的字段结构，包括需要填写哪些字段、字段的类型（文本、数字、下拉选择等）、是否必填等。

**作用**：

- 标准化数据格式，确保同一类型的数据采集项目收集的数据结构一致
- 支持表单动态渲染，根据模板定义生成填报界面

**与其他对象的关系**：

- 模板被数据采集项目引用
- 数据项的字段值按模板定义存储

**常见误解**：

- 误解：修改模板会影响已创建项目的数据。纠正：项目创建时会对模板进行快照，修改模板只影响后续新建的项目，不影响已创建的项目

### 3.7 数据项（Data Item）

**定义**：数据项是数据采集的基本单位，代表一条待填报、审批、发布的记录。一个数据项对应一个销售区划在一个周期内的采集数据。

**作用**：

- 作为填报、审批、发布操作的对象
- 通过状态流转管理数据生命周期

**与其他对象的关系**：

- 数据项归属特定项目和周期
- 数据项归属特定区划，由区划负责人填报
- 数据项字段值按模板定义存储

**常见误解**：

- 误解：数据项状态就是项目状态。纠正：项目状态（进行中/已完成）描述项目整体生命周期，数据项状态（待编辑/已审核等）描述单条数据的审批发布流程，两者独立管理

## 四、核心业务关系

### 业务主链路

```
创建项目 → 下发任务 → 填报数据 → 审批数据 → 锁定发布
```

1. **创建项目**：销售运营人员创建数据采集项目，指定目标区划、选择模板、配置周期和权限
2. **下发任务**：项目发布后，系统向目标区划的负责人下发填报任务通知
3. **填报数据**：区划负责人通过表格或AI对话方式填写数据，提交待审批
4. **审批数据**：审批人对提交的数据进行审阅，批准或驳回
5. **锁定发布**：总部对已审核数据进行锁定，最终发布生效

### 关键流转逻辑

#### 数据项状态流转

```
待编辑 → 已编辑 → 待审核 → 已审核 → 已锁定 → 已发布
   ↑         │       │        │         │
   └─────────┴───────┴────────┴─────────┘
              驳回/否决/下架 回退
```

| 操作 | 执行者 | 说明                                       |
| ---- | ------ | ------------------------------------------ |
| 提交 | 填报人 | 已编辑 → 待审核                            |
| 撤回 | 填报人 | 待审核 → 已编辑（审批前可撤回修改）        |
| 批准 | 审批人 | 待审核 → 已审核                            |
| 驳回 | 审批人 | 待审核 → 已编辑（必须填写驳回原因）        |
| 锁定 | 总部   | 已审核 → 已锁定（锁定后不可驳回）          |
| 解锁 | 总部   | 已锁定 → 已审核                            |
| 否决 | 总部   | 已审核/已锁定 → 已编辑（必须填写否决原因） |
| 发布 | 总部   | 已锁定 → 已发布                            |
| 下架 | 总部   | 已发布 → 已编辑（必须填写下架原因）        |

## 五、业务规则

### 5.1 显性规则

1. **区划本位规则**：所有数据采集都以区划为核心单元，数据项归属特定区划，由区划负责人填报和维护

2. **周期唯一规则**：同一项目下，同一区划在同一周期内只能有一条数据项

3. **状态流转规则**：数据项状态必须按照"待编辑→已编辑→待审核→已审核→已锁定→已发布"的顺序流转，不能跳过中间状态

4. **审批强制规则**：规划类数据必须经过审批才能发布生效，不能直接发布

5. **关联项目规则**：结果填报项目可关联目标规划项目，关联关系在项目创建时通过手动输入项目编码建立

### 5.2 隐性规则

1. **模板快照规则**：项目创建时对模板进行快照，修改模板不影响已创建项目的数据结构

2. **权限继承规则**：上级区划负责人对下级区划有查看和管理权限，但不能直接修改下级数据

3. **周期触发规则**：周期性项目在周期开始时自动生成数据项，填报人无需手动创建

4. **数据归属规则**：数据项归属填报时的区划，区划负责人变更不影响历史数据归属

### 5.3 权限或范围限制

1. **区划范围限制**：用户只能创建和管理自己权限范围内区划的数据采集项目

2. **角色权限矩阵**：不同角色（销售代表、地区经理、大区经理等）对数据项的操作权限通过权限矩阵配置，可能包括查看、编辑、审批、发布等

3. **总部权限**：仅"区域运营经理"角色拥有对数据项执行锁定、发布、否决、下架等高级操作的权限

4. **分管区划只读**：用户在分管区划视图下，表格内容全部只读，只能执行审批操作

### 5.4 特殊说明或例外

1. **一次性项目例外**：一次性项目只有一个周期，周期结束即项目完成，不需要周期触发

2. **通用采集简化**：通用数据采集项目可能不需要审批流程，填报后直接生效

3. **下架重新发布**：已发布数据被下架后，状态回退到已编辑，需要重新走审批发布流程

4. **非工作日排除**：日周期项目可配置排除非工作日，任务不会在非工作日触发

## 六、常见歧义与边界

### 6.1 区划相关

| 歧义点               | 澄清说明                                                                                                   |
| -------------------- | ---------------------------------------------------------------------------------------------------------- |
| 区划 vs 地理区域     | 区划是销售组织单元，与地理区域有对应关系但不完全等同。同一个地理区域可能划分为多个销售区划，由不同代表负责 |
| 直属区划 vs 分管区划 | 直属区划是用户直接负责的区划，用于填报数据；分管区划是用户协助管理他人区划，用于审批和管理，数据只读       |

### 6.2 项目相关

| 歧义点                   | 澄清说明                                                                                                                         |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
| 项目状态 vs 数据项状态   | 项目状态（规划中/进行中/暂停/已完成）描述项目整体生命周期；数据项状态（待编辑/已审核等）描述单条数据的审批发布流程，两者独立管理 |
| 一次性项目 vs 周期性项目 | 一次性项目只有一个周期，周期结束即项目完成；周期性项目按周期持续触发任务，项目长期存在                                           |
| 目标规划 vs 结果填报     | 目标规划是收集预期目标数据（如季度销售目标），结果填报是收集实际完成数据。两者可关联，便于对比分析                               |

### 6.3 数据相关

| 歧义点         | 澄清说明                                                                                                       |
| -------------- | -------------------------------------------------------------------------------------------------------------- |
| 数据项创建方式 | 支持两种方式：手动勾选创建（填报人从主数据列表勾选生成数据项）和自动生成（系统根据模板配置自动生成待填报记录） |
| 修改模板的影响 | 修改模板只影响后续新建的项目，不影响已创建项目的数据结构（项目创建时对模板进行快照）                           |

### 6.4 操作相关

| 歧义点                 | 澄清说明                                                                                             |
| ---------------------- | ---------------------------------------------------------------------------------------------------- |
| 驳回 vs 否决           | 驳回是审批人操作，将待审核数据回退到已编辑；否决是总部操作，可将已审核或已锁定数据回退到已编辑       |
| 锁定 vs 发布           | 锁定是发布前的准备操作，锁定后数据不可驳回；发布是最终生效操作，发布后数据可用于下游分析             |
| 表格提交 vs AI对话提交 | 两种方式生成的数据在结构和流程上完全一致，只是交互方式不同。表格适合复杂填报，AI对话适合简单快捷填报 |
